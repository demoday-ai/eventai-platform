# Developer Manifest

Этот документ описывает роли и правила взаимодействия для подключения дополнительных агентов к работе над репозиторием.

## Роли

### Разработчик
- Следует правилам из `AGENTS.md`, `CLAUDE.md` и пр.
- Пишет код и тесты, ведет BDD/TDD процесс.
- Делает осмысленные коммиты и обновляет документацию.
- Никакого спагетти кода

### Ревьювер кода
- Не пишет код и не вносит правки в файлы.
- Проверяет качество кода и риски регрессий.
- Создает ишьюсы с замечаниями и рекомендациями.

### Ревьювер тестов
- Не правит код и тесты.
- Проверяет, что тесты действительно проверяют поведение и покрывают критерии, а не всё время "зелёные".
- Допускает "не зеленые" тесты, если это отражает реальное состояние задачи.
- Создает ишьюсы с замечаниями и рекомендациями.

## Взаимодействие
- Разработчик отвечает за реализацию и исправления по ишьюсам.
- Ревьюверы дают обратную связь через ишьюсы, без прямых изменений в коде.

## Требования к артефактам (Issue/PR)
- Issue: содержит "Why/What/How/Acceptance Criteria", примеры ввода/вывода и ограничения.
- PR: краткое описание, список сценариев BDD или TDD, команда для запуска тестов, статус (Red/Green).
- Изменения UI/вывода: приложить примеры до/после в тексте PR или issue.

## Политики работы
- Если требования неясны — остановиться и открыть issue с вопросами.
- Не расширять scope задачи без новой issue.
- Все замечания ревьюеров оформляются как отдельные issues.
- Если обнаружен баг вне задачи — зафиксировать в issue и не чинить в текущем PR.

## Метрики качества
- Для каждой задачи: минимум один BDD‑сценарий / TDD / User-story и соответствующие step definitions.
- Тесты должны подтверждать поведение, а не реализацию.
- Готовность задачи: тесты в Green, документация обновлена, коммиты осмысленные.
