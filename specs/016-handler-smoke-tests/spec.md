# Feature Specification: Smoke-тесты на критичные хендлеры

**Feature Branch**: `016-handler-smoke-tests`
**Created**: 2026-02-03
**Status**: Draft

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Онбординг нового пользователя (Priority: P1)

Разработчик запускает smoke-тесты для проверки критичного flow онбординга. Тесты эмулируют взаимодействие пользователя с ботом: команда /start, выбор роли, регистрация. При обнаружении бага тесты падают с понятным сообщением об ошибке.

**Why this priority**: Онбординг — точка входа для всех пользователей. Если он сломан, никто не сможет использовать бота на Demo Day.

**Independent Test**: Можно запустить только тесты онбординга и убедиться, что новые пользователи могут зарегистрироваться и выбрать роль.

**Acceptance Scenarios**:

1. **Given** бот запущен и БД пуста, **When** запускаются тесты онбординга, **Then** тесты проверяют команду /start и выбор каждой из 5 ролей (организатор, студент, эксперт, гость, бизнес)
2. **Given** пользователь уже зарегистрирован, **When** он снова вызывает /start, **Then** тест проверяет корректную обработку повторной регистрации
3. **Given** некорректный callback (неизвестная роль), **When** бот получает такой callback, **Then** тест проверяет graceful error handling

---

### User Story 2 - Профилирование гостя (Priority: P1)

Разработчик запускает smoke-тесты для проверки flow профилирования гостя. Тесты эмулируют выбор интересов, ввод ключевых слов, подтверждение профиля.

**Why this priority**: Профилирование гостя — ключевой flow для персонализации. Без него гости не получат рекомендации.

**Independent Test**: Можно запустить только тесты профилирования гостя и убедиться, что профиль создаётся и сохраняется корректно.

**Acceptance Scenarios**:

1. **Given** пользователь с ролью "гость", **When** он проходит flow профилирования (выбор тегов, ввод интересов), **Then** тест проверяет создание GuestProfile с корректными данными
2. **Given** гость уже имеет профиль, **When** он выбирает "Исправить интересы", **Then** тест проверяет возможность редактирования профиля
3. **Given** гость не выбрал ни одного тега, **When** он пытается подтвердить профиль, **Then** тест проверяет валидацию (минимум 1 тег)

---

### User Story 3 - Профилирование бизнес-партнёра (Priority: P1)

Разработчик запускает smoke-тесты для проверки flow профилирования бизнес-партнёра. Тесты эмулируют выбор цели, отраслей, технологий.

**Why this priority**: Бизнес-партнёры — ключевой сегмент для Demo Day. Их профилирование критично для качественных рекомендаций.

**Independent Test**: Можно запустить только тесты профилирования бизнеса и убедиться, что BusinessProfile создаётся корректно.

**Acceptance Scenarios**:

1. **Given** пользователь с ролью "бизнес", **When** он проходит flow профилирования (цель, отрасли, технологии), **Then** тест проверяет создание BusinessProfile
2. **Given** бизнес-партнёр выбирает цель "Инвестиции", **When** он продолжает профилирование, **Then** тест проверяет что objective = INVESTMENT
3. **Given** профиль уже существует, **When** партнёр хочет изменить данные, **Then** тест проверяет корректное обновление профиля

---

### User Story 4 - Получение рекомендаций (Priority: P2)

Разработчик запускает smoke-тесты для проверки генерации рекомендаций. Тесты проверяют что пользователь с профилем получает список рекомендованных проектов.

**Why this priority**: Рекомендации — основная ценность для гостей и бизнеса. Без них бот теряет смысл.

**Independent Test**: Можно запустить тесты рекомендаций с заранее созданным профилем и проектами.

**Acceptance Scenarios**:

1. **Given** гость с заполненным профилем и проекты в БД, **When** гость запрашивает рекомендации, **Then** тест проверяет возврат непустого списка проектов
2. **Given** бизнес-партнёр с профилем, **When** он запрашивает рекомендации, **Then** тест проверяет что рекомендации соответствуют его целям/отраслям
3. **Given** LLM недоступен (mock), **When** запрашиваются рекомендации, **Then** тест проверяет fallback поведение

---

### User Story 5 - Q&A Helper (Priority: P2)

Разработчик запускает smoke-тесты для проверки генерации вопросов Q&A. Тесты проверяют что пользователь получает релевантные вопросы для проекта.

**Why this priority**: Q&A helper помогает гостям и бизнесу задавать правильные вопросы. Важно для качества взаимодействия.

**Independent Test**: Можно запустить тесты Q&A с заранее созданным профилем и проектом.

**Acceptance Scenarios**:

1. **Given** гость с профилем и конкретный проект, **When** гость запрашивает вопросы для проекта, **Then** тест проверяет возврат списка вопросов
2. **Given** бизнес-партнёр (инвестор), **When** он запрашивает вопросы, **Then** тест проверяет что вопросы соответствуют типу BUSINESS_INVESTOR
3. **Given** LLM недоступен, **When** запрашиваются вопросы, **Then** тест проверяет возврат fallback вопросов

---

### Edge Cases

- Что происходит при одновременном профилировании двух пользователей с одним telegram_user_id?
- Как система обрабатывает callback от удалённого сообщения?
- Что если пользователь нажимает кнопки очень быстро (race condition)?
- Как обрабатывается таймаут при ожидании текстового ввода?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система ДОЛЖНА предоставлять smoke-тесты для проверки flow онбординга (команда /start, выбор роли)
- **FR-002**: Система ДОЛЖНА предоставлять smoke-тесты для проверки профилирования гостей
- **FR-003**: Система ДОЛЖНА предоставлять smoke-тесты для проверки профилирования бизнес-партнёров
- **FR-004**: Система ДОЛЖНА предоставлять smoke-тесты для проверки генерации рекомендаций
- **FR-005**: Система ДОЛЖНА предоставлять smoke-тесты для проверки Q&A helper
- **FR-006**: Тесты ДОЛЖНЫ использовать mock/stub для эмуляции Telegram API (не отправлять реальные сообщения)
- **FR-007**: Тесты ДОЛЖНЫ использовать изолированную тестовую БД
- **FR-008**: Тесты ДОЛЖНЫ выводить понятные сообщения об ошибках при падении
- **FR-009**: Тесты ДОЛЖНЫ запускаться одной командой
- **FR-010**: Тесты ДОЛЖНЫ mock'ать LLM вызовы для предсказуемости и скорости

### Key Entities

- **TestUser**: Эмулированный пользователь Telegram (telegram_user_id, username, роль)
- **MockUpdate**: Эмулированное Telegram Update (message, callback_query)
- **MockContext**: Эмулированный контекст бота (user_data, bot)

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Все 5 критичных flows покрыты smoke-тестами с полным покрытием acceptance scenarios (минимум 15 тестов: 3 сценария × 5 flows)
- **SC-002**: Все smoke-тесты выполняются за время не более 60 секунд
- **SC-003**: При внесении breaking change в хендлер, соответствующий тест падает
- **SC-004**: Разработчик может запустить тесты локально одной командой
- **SC-005**: Тесты не требуют реального Telegram Bot Token или LLM API ключа для запуска

## Clarifications

- Q: Какое максимальное время выполнения всех smoke-тестов считается приемлемым? → A: До 60 секунд

## Assumptions

- Используется pytest как тестовый фреймворк (уже есть в проекте)
- python-telegram-bot предоставляет способы mock'ать Update/Context
- Тесты будут запускаться в Docker контейнере с доступом к тестовой PostgreSQL
- Mock LLM будет возвращать предопределённые ответы для детерминированности тестов
