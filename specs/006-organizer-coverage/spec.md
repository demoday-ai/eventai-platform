# Feature Specification: Дашборд покрытия тематик для организатора (EPIC-006)

**Feature Branch**: `006-organizer-coverage`
**Created**: 2026-02-02
**Status**: Draft
**Input**: User description: "EPIC-006 — Организаторское покрытие тематик"

## Context

На Demo Day ~330 проектов распределены по 6-10 залам. Организатору (~5 человек) критически важно видеть, какие залы покрыты экспертами, а какие нет. Сейчас это делается вручную (звонки, таблицы), что приводит к ситуациям «у нас в комнате никто не пришёл из экспертов». Нужно 60-80 экспертов, активных ~30-40 — и организатор должен видеть реальную картину покрытия в реальном времени, чтобы вовремя добирать недостающих.

Этот эпик закрывает US-011 и US-012 из User Story Map:
- US-011: Организатор видит покрытие тематик и секций экспертами, статусы подтверждений
- US-012: Организатор видит непокрытые тематики, чтобы вовремя найти экспертов

## Assumptions

- Утверждённое расписание (approved ClusteringRun) уже существует — эпики 001-002 реализованы
- Эксперты уже приглашены в залы (EPIC-004) с указанием статуса (proposed/accepted/rejected)
- У экспертов есть теги компетенций, у проектов есть тематические теги
- Организатор идентифицируется по списку organizer_ids (как в EPIC-003)
- Бот работает в Telegram, REST API доступно для веб-интерфейса
- Покрытие зала считается достаточным при ≥2 подтверждённых экспертах

## User Scenarios & Testing

### User Story 1 — Сводка покрытия по залам (Priority: P1)

Организатор запрашивает сводку покрытия командой `/coverage` в Telegram-боте. Бот показывает таблицу: каждый зал, сколько экспертов подтверждено / приглашено / всего проектов, и цветовой статус (покрыт / частично / не покрыт). Организатор сразу видит, какие залы требуют внимания.

**Why this priority**: Это основная потребность — быстро оценить общую картину. Без этого организатор не знает, какие залы в зоне риска. Закрывает Pain #3 (Expert Shortage) и JTBD-2 (Ensure Expert Coverage).

**Independent Test**: Создать событие с 6 залами, назначить экспертов с разными статусами. Отправить `/coverage` — проверить что в ответе видны все залы с корректными счётчиками и статусами.

**Acceptance Scenarios**:

1. **Given** событие с утверждённым расписанием и 6 залами, **When** организатор отправляет `/coverage`, **Then** бот показывает список всех залов с количеством подтверждённых/приглашённых экспертов и проектов
2. **Given** зал с 0 подтверждённых экспертов, **When** организатор смотрит сводку, **Then** этот зал отмечен как «не покрыт»
3. **Given** зал с ≥2 подтверждёнными экспертами, **When** организатор смотрит сводку, **Then** зал отмечен как «покрыт»
4. **Given** зал с 1 подтверждённым экспертом, **When** организатор смотрит сводку, **Then** зал отмечен как «частично покрыт»

---

### User Story 2 — Детализация по залу (Priority: P1)

Организатор нажимает на зал в сводке и видит детали: список экспертов (имя, статус приглашения, теги компетенций, релевантность), список проектов зала с их тегами. Видно, какие тематики проектов покрыты экспертной компетенцией, а какие нет.

**Why this priority**: Без детализации организатор не может принять решение — кого именно добирать и куда. Дополняет US1 как drill-down.

**Independent Test**: Создать зал с 3 экспертами (accepted/proposed/rejected) и 10 проектами с тегами. Нажать на зал — проверить что видны все эксперты со статусами и совпадение тегов.

**Acceptance Scenarios**:

1. **Given** зал с назначенными экспертами, **When** организатор нажимает на зал, **Then** видит список экспертов с именем, статусом приглашения и тегами
2. **Given** зал с проектами, **When** организатор просматривает детали зала, **Then** видит количество проектов и основные тематики (теги)
3. **Given** эксперт со статусом «rejected», **When** организатор смотрит детали, **Then** отклонённый эксперт отображается с соответствующей пометкой

---

### User Story 3 — Непокрытые тематики и рекомендации (Priority: P2)

Организатор запрашивает список непокрытых тематик — система показывает, в каких залах есть проекты по темам, для которых нет подтверждённого эксперта с соответствующей компетенцией. Для каждого пробела показываются рекомендации: свободные эксперты с подходящими тегами.

**Why this priority**: Это позволяет проактивно закрывать пробелы, а не реагировать на проблемы в день DD. Прямо закрывает US-012.

**Independent Test**: Создать зал с проектами по тегу «NLP», но без эксперта с тегом «NLP». Запросить непокрытые тематики — проверить что «NLP» в зале X указан как пробел и предложен свободный NLP-эксперт.

**Acceptance Scenarios**:

1. **Given** зал с проектами по теме «NLP» и без эксперта с тегом «NLP», **When** организатор запрашивает непокрытые тематики, **Then** система показывает «NLP» как непокрытую тему в данном зале
2. **Given** непокрытая тема в зале, и свободный эксперт с подходящим тегом, **When** организатор видит пробел, **Then** система рекомендует этого эксперта как кандидата
3. **Given** все темы во всех залах покрыты, **When** организатор запрашивает пробелы, **Then** система сообщает «Все тематики покрыты»

---

### User Story 4 — REST API для веб-дашборда (Priority: P2)

Организатор через веб-интерфейс (или любой HTTP-клиент) получает данные покрытия: общая сводка, детали по залу, список пробелов. Данные возвращаются в структурированном формате для визуализации.

**Why this priority**: Дублирует функциональность бота через REST API для будущего веб-дашборда. Данные те же, канал другой.

**Independent Test**: Вызвать эндпоинт покрытия — проверить что возвращает JSON со всеми залами, счётчиками экспертов и статусами.

**Acceptance Scenarios**:

1. **Given** авторизованный организатор, **When** вызывает эндпоинт сводки покрытия, **Then** получает список залов с метриками покрытия
2. **Given** запрос с параметром room_id, **When** вызывает эндпоинт детализации, **Then** получает экспертов и проекты конкретного зала
3. **Given** неавторизованный пользователь, **When** вызывает эндпоинт, **Then** получает ошибку 403

---

### Edge Cases

- Что если нет утверждённого расписания (нет approved ClusteringRun)? — Показать сообщение «Нет утверждённого расписания. Сначала запустите кластеризацию»
- Что если в зале нет проектов? — Зал показывается с 0 проектов (возможно ошибка в расписании)
- Что если у экспертов нет тегов? — Эксперт показывается, но без информации о тематическом соответствии
- Что если у проектов нет тегов? — Проект показывается, но тематическое покрытие для него не рассчитывается
- Что если организатор запрашивает покрытие во время изменения расписания? — Показывать актуальные данные на момент запроса
- Что если эксперт назначен в несколько залов? — Показывать во всех назначенных залах с соответствующими статусами

## Requirements

### Functional Requirements

- **FR-001**: Система ДОЛЖНА предоставлять сводку покрытия по всем залам текущего утверждённого расписания по команде `/coverage`
- **FR-002**: Для каждого зала сводка ДОЛЖНА содержать: название зала, количество проектов, количество подтверждённых экспертов, количество приглашённых (pending) экспертов, статус покрытия
- **FR-003**: Статус покрытия зала ДОЛЖЕН определяться: «покрыт» (≥2 подтверждённых эксперта), «частично» (1 подтверждённый), «не покрыт» (0 подтверждённых)
- **FR-004**: Организатор ДОЛЖЕН иметь возможность перейти от сводки к детализации конкретного зала
- **FR-005**: Детализация зала ДОЛЖНА содержать: список экспертов с именем, статусом приглашения и тегами компетенций; количество проектов и основные тематики (топ-теги)
- **FR-006**: Система ДОЛЖНА определять непокрытые тематики — темы, по которым в зале есть проекты, но нет подтверждённого эксперта с соответствующим тегом
- **FR-007**: Для каждого непокрытого тега система ДОЛЖНА показать список свободных (не назначенных в этот зал) экспертов с таким тегом как кандидатов
- **FR-008**: Все данные покрытия ДОЛЖНЫ быть доступны через REST API (сводка, детализация по залу, пробелы)
- **FR-009**: Доступ к данным покрытия ДОЛЖЕН быть ограничен организаторами (по списку organizer_ids)
- **FR-010**: Команда `/coverage` и REST API ДОЛЖНЫ работать только при наличии утверждённого расписания; при отсутствии — сообщать об этом

### Key Entities

- **CoverageStatus**: Агрегированный статус покрытия зала — название, количество проектов, подтверждённые/приглашённые эксперты, цветовой статус, основные теги проектов
- **RoomDetail**: Расширенная информация по залу — список экспертов (имя, статус, теги, match_score), список тематик проектов, непокрытые теги
- **CoverageGap**: Тематический пробел — зал, непокрытый тег, список рекомендованных экспертов-кандидатов
- **ExpertCandidate**: Свободный эксперт для закрытия пробела — имя, теги, текущие назначения

## Clarifications

### Session 2026-02-02

No critical ambiguities detected. Structured scan across 11 taxonomy categories found all Clear:

- Tag matching between experts and projects uses shared tag identifiers — exact match, no hierarchy needed
- "Free expert" (FR-007) = expert with matching tag NOT assigned to the specific room in question

- Scope is read-only dashboard — no reassignment actions included
- Access control reuses existing organizer_ids pattern from EPIC-003

## Success Criteria

### Measurable Outcomes

- **SC-001**: Организатор получает полную картину покрытия всех залов за одно действие (1 команда / 1 запрос)
- **SC-002**: Организатор может определить все непокрытые залы менее чем за 30 секунд после запроса
- **SC-003**: Для каждого пробела система предлагает хотя бы одного кандидата-эксперта (при наличии свободных экспертов с подходящим тегом)
- **SC-004**: 100% залов отображаются в сводке с корректными счётчиками (подтверждённые / приглашённые / проекты)
- **SC-005**: Организаторы могут сократить время обнаружения непокрытых залов с часов (ручной обзвон) до секунд (запрос к боту)
