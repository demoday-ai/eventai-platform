## User Scenarios & Testing *(mandatory)*

### User Story 1 - Пустой Dashboard: первый вход организатора (Priority: P1)

Организатор впервые открывает админку. Событие ещё не создано. Dashboard показывает понятный empty state с объяснением, что нужно сделать, и кнопкой перехода на страницу Импорта для создания события. Организатор понимает, с чего начать, без чтения инструкции.

**Why this priority**: Это первое, что видит новый пользователь. Без понятного empty state пользователь не знает, куда нажать, и может уйти. Формирует первое впечатление о продукте.

**Independent Test**: Достаточно открыть приложение без созданного события — Dashboard должен показать карточку с текстом и кнопкой. Ценность: пользователь сразу понимает, с чего начать.

**Acceptance Scenarios**:

1. **Given** событие не создано, **When** организатор открывает Dashboard, **Then** отображается центрированная карточка с текстом «Нет активного мероприятия» и кнопкой «Перейти к импорту»
2. **Given** событие не создано, **When** организатор нажимает кнопку «Перейти к импорту», **Then** происходит переход на страницу Импорта, таб «Событие»
3. **Given** событие не создано, **When** организатор открывает Dashboard, **Then** Global Stepper отображается с тремя фазами, все в статусе «не начата» (серые)

---

### User Story 2 - Global Stepper: видимый прогресс подготовки (Priority: P1)

На каждой странице приложения (включая Dashboard) организатор видит горизонтальную полосу прогресса с 3 фазами: Данные, Распределение, Запуск. Каждая фаза имеет статус (завершена / в процессе / не начата). Клик на фазу ведёт на соответствующую страницу. Организатор в любой момент понимает, на каком этапе находится подготовка.

**Why this priority**: Global Stepper — ключевой UX-элемент, превращающий набор страниц в wizard-подобный интерфейс. Без него пользователь не видит общего прогресса и не знает, что делать дальше.

**Independent Test**: Можно протестировать, последовательно проходя шаги подготовки и проверяя, что статусы фаз обновляются. Ценность: организатор всегда видит, где он находится в процессе.

**Acceptance Scenarios**:

1. **Given** событие создано, проекты, студенты, эксперты загружены, **When** организатор смотрит на Stepper, **Then** фаза «Данные» отображается как завершённая (зелёная), остальные — не начаты (серые)
2. **Given** кластеризация одобрена, матчинг одобрён, расписание одобрено, **When** организатор смотрит на Stepper, **Then** фазы «Данные» и «Распределение» — завершены, «Запуск» — в процессе
3. **Given** напоминания настроены и брифинг отправлен, **When** организатор смотрит на Stepper, **Then** все три фазы отображаются как завершённые
4. **Given** любая фаза видна в Stepper, **When** организатор кликает на неё, **Then** происходит переход на первую незавершённую страницу этой фазы
5. **Given** фаза «Данные» не завершена (не загружены эксперты), **When** организатор наводит курсор на фазу, **Then** видны подшаги с их статусами (событие — готово, проекты — готово, студенты — готово, эксперты — не загружены)

---

### User Story 3 - Quick Action: следующий шаг (Priority: P1)

На Dashboard отображается подсказка с ближайшим незавершённым шагом и кнопкой перехода на нужную страницу. Организатор не думает «а что дальше?» — система сама говорит.

**Why this priority**: Ключевой элемент wizard-подхода. Убирает когнитивную нагрузку — организатор всегда знает следующее действие.

**Independent Test**: Загрузить часть данных и проверить, что Quick Action показывает конкретный следующий шаг с кнопкой. Ценность: пользователь никогда не «застревает».

**Acceptance Scenarios**:

1. **Given** событие создано, но проекты не загружены, **When** организатор открывает Dashboard, **Then** Quick Action показывает: «Загрузите проекты на странице Импорта» + кнопка «Перейти»
2. **Given** проекты загружены, кластеризация не запущена, **When** организатор открывает Dashboard, **Then** Quick Action: «Кластеризация не одобрена. Запустите кластеризацию по залам» + кнопка «Перейти» → Кластеризация
3. **Given** кластеризация одобрена, матчинг не одобрён, **When** организатор открывает Dashboard, **Then** Quick Action: «Распределите экспертов по залам» + кнопка «Перейти» → Эксперты
4. **Given** расписание одобрено, напоминания не настроены, **When** организатор открывает Dashboard, **Then** Quick Action: «Настройте напоминания и отправьте брифинг» + кнопка «Перейти» → Напоминания
5. **Given** все шаги подготовки завершены, **When** организатор открывает Dashboard, **Then** Quick Action не отображается

---

### User Story 4 - Метрики мероприятия (Priority: P2)

Организатор видит на Dashboard ключевые числа: количество проектов, студентов, экспертов, партнёров, залов. Это даёт мгновенное понимание масштаба и готовности мероприятия.

**Why this priority**: Важно для ежедневного мониторинга, но не является критическим для первого запуска. Без метрик пользователь всё равно может пользоваться системой.

**Independent Test**: Загрузить данные и проверить, что 5 карточек показывают корректные числа. Ценность: организатор оценивает масштаб одним взглядом.

**Acceptance Scenarios**:

1. **Given** событие создано, загружены 127 проектов, 412 студентов, 18 экспертов, 23 партнёра, 6 залов, **When** организатор открывает Dashboard, **Then** отображаются 5 карточек с числами: 127, 412, 18, 23, 6
2. **Given** данные не загружены, **When** организатор открывает Dashboard, **Then** все карточки показывают 0
3. **Given** другой организатор загрузил новых партнёров, **When** прошло 30-60 секунд, **Then** число партнёров на Dashboard текущего организатора обновляется автоматически (без перезагрузки страницы)

---

### User Story 5 - Дата и обратный отсчёт (Priority: P2)

На Dashboard отображается дата мероприятия и обратный отсчёт в днях. Рядом — ссылка для редактирования на странице Импорта.

**Why this priority**: Создаёт ощущение срочности и контекст. Не блокирует работу, но мотивирует.

**Independent Test**: Создать событие с датой в будущем, проверить, что дата и обратный отсчёт отображаются корректно.

**Acceptance Scenarios**:

1. **Given** событие создано с датой 15 марта 2026, сегодня 8 февраля 2026, **When** организатор открывает Dashboard, **Then** отображается «Demo Day: 15 марта 2026» и «через 35 дней»
2. **Given** дата мероприятия — сегодня, **When** организатор открывает Dashboard, **Then** обратный отсчёт показывает «сегодня»
3. **Given** дата мероприятия в прошлом, **When** организатор открывает Dashboard, **Then** обратный отсчёт показывает «завершено» или количество дней назад
4. **Given** дата отображается, **When** организатор кликает на ссылку редактирования, **Then** переход на страницу Импорта, таб «Событие»

---

### User Story 6 - Покрытие залов экспертами (Priority: P2)

На Dashboard отображается сводная таблица покрытия залов экспертами: зал, количество проектов, количество экспертов, статус (покрыт / пробел). Кнопка «Детали» ведёт к конкретному залу на странице Экспертов.

**Why this priority**: Покрытие экспертами — главная боль организатора (выявлено в Journey Map, score 2). Таблица на Dashboard позволяет быстро увидеть пробелы, не заходя на отдельную страницу.

**Independent Test**: Провести кластеризацию и матчинг, проверить, что таблица показывает корректные данные по залам.

**Acceptance Scenarios**:

1. **Given** кластеризация одобрена, 6 залов с проектами, эксперты назначены, **When** организатор открывает Dashboard, **Then** таблица показывает все залы с колонками: название, количество проектов, количество экспертов, статус покрытия
2. **Given** зал с 0 экспертами, **When** организатор смотрит строку этого зала, **Then** статус — «Пробел» (красный индикатор)
3. **Given** зал с 1 экспертом, **When** организатор смотрит строку, **Then** статус — «Частично» (жёлтый индикатор)
4. **Given** зал с 2 экспертами, **When** организатор смотрит строку, **Then** статус — «Покрыт» (зелёный индикатор)
5. **Given** зал с 3 экспертами, **When** организатор смотрит строку, **Then** статус — «Отлично» (зелёный индикатор)
6. **Given** зал с >3 экспертами, **When** организатор смотрит строку, **Then** статус — «Перебор» (информационный индикатор)
7. **Given** таблица покрытия видна, **When** организатор нажимает «Детали» на конкретном зале, **Then** переход на страницу Экспертов → Покрытие с фокусом на этом зале
8. **Given** кластеризация не проведена, **When** организатор открывает Dashboard, **Then** таблица покрытия не отображается (нет залов)

---

### User Story 7 - Auto-refresh данных (Priority: P3)

Данные на Dashboard обновляются автоматически каждые 30-60 секунд без перезагрузки страницы. Обновление незаметно для пользователя и не сбрасывает позицию скролла.

**Why this priority**: Важно для параллельной работы 3-5 организаторов, но не критично для одиночного пользователя.

**Independent Test**: Открыть Dashboard в двух браузерах, изменить данные в одном — проверить, что второй обновился автоматически.

**Acceptance Scenarios**:

1. **Given** Dashboard открыт, **When** проходит 30-60 секунд, **Then** метрики, покрытие и Quick Action обновляются без видимого мерцания или перезагрузки
2. **Given** Dashboard прокручен вниз до таблицы покрытия, **When** происходит auto-refresh, **Then** позиция скролла не меняется
3. **Given** данные изменились (другой организатор загрузил проекты), **When** происходит auto-refresh, **Then** число проектов на Dashboard обновляется до актуального

---

### Edge Cases

- Что происходит, если событие удалено после загрузки данных? Dashboard возвращается в empty state с кнопкой «Перейти к импорту».
- Что если один организатор редактирует событие, пока другой смотрит Dashboard? Auto-refresh покажет актуальные данные через 30-60 секунд.
- Что если сервер недоступен при auto-refresh? Данные остаются прежними, без сообщения об ошибке при фоновом обновлении. При восстановлении связи данные обновляются при следующем цикле.
- Что если количество залов > 10? Таблица покрытия скроллится вертикально внутри content area.
- Что если обратный отсчёт достигает 0 (мероприятие сегодня)? Отображается «сегодня» вместо «через 0 дней».
- Что если пользователь кликает на уже завершённую фазу в Stepper? Переход на первую страницу этой фазы (для просмотра / перепроверки).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Dashboard MUST отображать empty state с текстом «Нет активного мероприятия» и кнопкой перехода на Импорт, когда событие не создано
- **FR-002**: Dashboard MUST отображать Global Stepper с тремя фазами (Данные, Распределение, Запуск), каждая с визуальным статусом (завершена / в процессе / не начата)
- **FR-003**: Global Stepper MUST обеспечивать навигацию: клик на фазу → переход к первой незавершённой странице этой фазы
- **FR-004**: Global Stepper MUST показывать подшаги фазы при наведении/клике (детализация прогресса)
- **FR-005**: Dashboard MUST отображать Quick Action — подсказку с ближайшим незавершённым шагом и кнопкой перехода
- **FR-006**: Quick Action MUST динамически определять ближайший незавершённый шаг в порядке pipeline: событие → проекты → студенты → эксперты → кластеризация → матчинг → расписание → напоминания → брифинг
- **FR-007**: Quick Action НЕ ДОЛЖЕН отображаться, когда все шаги подготовки завершены
- **FR-008**: Dashboard MUST отображать 5 карточек метрик: количество проектов, студентов, экспертов, партнёров (все источники: бот + импорт), залов
- **FR-009**: Dashboard MUST отображать дату мероприятия и обратный отсчёт в днях
- **FR-010**: Dashboard MUST предоставлять ссылку для перехода к редактированию даты события (Импорт → таб «Событие»)
- **FR-011**: Dashboard MUST отображать сводную таблицу покрытия залов экспертами (зал, проекты, эксперты, статус по 5-уровневой шкале: 0=пробел, 1=частично, 2=покрыт, 3=отлично, >3=перебор)
- **FR-012**: Таблица покрытия MUST предоставлять ссылку «Детали» для каждого зала с переходом на страницу Экспертов/Покрытие
- **FR-013**: Данные Dashboard MUST обновляться автоматически каждые 30-60 секунд без перезагрузки страницы
- **FR-014**: Auto-refresh НЕ ДОЛЖЕН сбрасывать позицию скролла пользователя
- **FR-015**: Auto-refresh НЕ ДОЛЖЕН вызывать видимое мерцание или перерисовку интерфейса

### Key Entities

- **Событие (Event)**: мероприятие с названием, датой начала, датой окончания. Одно активное событие в системе.
- **Фаза подготовки (Phase)**: логический этап (Данные / Распределение / Запуск) с набором подшагов и статусом завершённости.
- **Метрика (Metric)**: агрегированное числовое значение (количество проектов, студентов, экспертов, партнёров, залов).
- **Покрытие зала (Room Coverage)**: связь зала с количеством проектов и экспертов, статус покрытия по 5-уровневой шкале (0=пробел, 1=частично, 2=покрыт, 3=отлично, >3=перебор).
- **Партнёр (Partner)**: контакт с флагом источника (бот / импорт). Единая таблица, фильтр по источнику в UI.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Новый организатор, впервые открыв Dashboard, понимает первый шаг и нажимает кнопку перехода к импорту в течение 10 секунд
- **SC-002**: Организатор определяет текущий этап подготовки (в какой фазе находится) за 3 секунды, глядя на Global Stepper
- **SC-003**: Организатор ни разу не задаёт вопрос «Что делать дальше?» — Quick Action всегда указывает следующий шаг
- **SC-004**: Данные на Dashboard актуальны с задержкой не более 60 секунд при параллельной работе нескольких организаторов
- **SC-005**: Организатор находит зал с пробелом в экспертном покрытии и переходит к деталям в течение 5 секунд
- **SC-006**: Dashboard загружается и отображает все данные в течение 2 секунд после авторизации

## Assumptions

- Событие создаётся на странице Импорта (таб «Событие»), а не на Dashboard
- В системе одно активное событие одновременно
- Авторизация уже реализована и не входит в scope этого эпика
- Backend API для метрик требует доработки: добавить projects_count и partners_count в GET /api/v1/admin/dashboard
- Новый endpoint GET /api/v1/admin/pipeline-status для статусов фаз Global Stepper
- Sidebar и общий layout реализуются отдельным эпиком (EPIC-002), Dashboard использует существующий layout
- Визуальные принципы: минимализм, Lucide-иконки, функциональный цвет (зелёный/жёлтый/серый), одна primary-кнопка на экран
