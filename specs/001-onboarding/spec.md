# Feature Specification: Онбординг и выбор роли

**Feature Branch**: `001-onboarding`
**Created**: 2026-02-02
**Status**: Draft
**Input**: USM v2.2 — EPIC-001 (US-001, US-002, SS-001)

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Первый вход и выбор роли (Priority: P1)

Новый пользователь открывает бота в Telegram и отправляет `/start`. Бот приветствует его, кратко объясняет своё назначение (AI-куратор Demo Day) и предлагает выбрать роль из пяти вариантов через inline-кнопки: Организатор, Эксперт, Студент, Гость, Бизнес-партнёр. После нажатия кнопки бот подтверждает выбор и показывает главное меню выбранной роли.

**Why this priority**: Без онбординга ни одна другая функция бота невозможна. Это единая точка входа для всех 435 участников Demo Day — архитектурная основа (RICE 88, но блокирует все остальные фичи). Ключевое отличие от Demo-Hero, который не различал роли.

**Independent Test**: Можно полностью протестировать, отправив `/start` в бота и выбрав любую роль. Пользователь видит приветствие и меню — фича доставляет ценность как самостоятельный каркас бота.

**Acceptance Scenarios**:

1. **Given** пользователь никогда не взаимодействовал с ботом, **When** отправляет `/start`, **Then** бот показывает приветственное сообщение с описанием и 5 inline-кнопок выбора роли.
2. **Given** бот показал кнопки выбора роли, **When** пользователь нажимает "Студент", **Then** бот подтверждает "Вы выбрали роль: Студент" и показывает главное меню студента.
3. **Given** бот показал кнопки выбора роли, **When** пользователь нажимает "Организатор", **Then** бот проверяет, есть ли Telegram ID пользователя в whitelist организаторов. Если да — подтверждает роль и показывает меню организатора. Если нет — сообщает "Роль организатора доступна только по приглашению" и предлагает выбрать другую роль.
4. **Given** бот показал кнопки выбора роли, **When** пользователь вводит произвольный текст вместо нажатия кнопки, **Then** бот повторяет предложение выбрать роль через кнопки.

---

### User Story 2 — Выбор подтипа гостя (Priority: P2)

После выбора роли "Гость" бот показывает второй экран с выбором подтипа: Абитуриент, AI-практик, Другое. Подтип сохраняется для аналитики организатора (сколько абитуриентов, сколько AI-практиков пришло). Подтип не влияет на функциональность бота для гостя.

**Why this priority**: Подтипы нужны для аналитики организатора (RICE 66). Без них бот работает для гостей, но организатор не видит срез аудитории. По данным прошлого DD: ~30% абитуриенты, ~45% AI-практики, ~25% другое.

**Independent Test**: Выбрать роль "Гость", увидеть экран подтипов, нажать кнопку — бот подтверждает и переходит к главному меню гостя.

**Acceptance Scenarios**:

1. **Given** пользователь выбрал роль "Гость", **When** бот показывает экран подтипов, **Then** отображаются 3 inline-кнопки: "Абитуриент", "AI-практик", "Другое".
2. **Given** бот показал кнопки подтипов, **When** пользователь нажимает "AI-практик", **Then** бот сохраняет подтип, подтверждает выбор и показывает главное меню гостя.
3. **Given** пользователь выбрал роль "Бизнес-партнёр", **When** бот обрабатывает выбор, **Then** экран подтипов НЕ показывается — сразу переход к меню бизнес-партнёра.

---

### User Story 3 — Возврат и смена роли (Priority: P3)

Пользователь, который уже выбрал роль, повторно отправляет `/start` или команду `/role`. Бот предлагает продолжить с текущей ролью или сменить её. При смене роли предыдущий выбор обновляется.

**Why this priority**: Обеспечивает гибкость для пользователей, которые ошиблись при выборе или хотят попробовать другую роль. Менее критично для демо, но важно для UX в продакшене.

**Independent Test**: Выбрать роль, затем отправить `/role` — увидеть текущую роль и предложение сменить.

**Acceptance Scenarios**:

1. **Given** пользователь ранее выбрал роль "Эксперт", **When** отправляет `/start`, **Then** бот показывает: "Вы зарегистрированы как Эксперт. Продолжить? [Продолжить] [Сменить роль]".
2. **Given** бот предложил продолжить или сменить роль, **When** пользователь нажимает "Сменить роль", **Then** бот показывает 5 кнопок выбора роли (как при первом входе).
3. **Given** пользователь был "Гость (AI-практик)" и сменил роль на "Студент", **When** смена завершена, **Then** бот обновляет роль, подтип сбрасывается, показывается меню студента.
4. **Given** пользователь ранее выбрал роль, **When** отправляет `/role`, **Then** бот показывает текущую роль и предложение сменить (аналогично повторному `/start`).

---

### Edge Cases

- Пользователь нажимает кнопку роли повторно (double-tap) — бот обрабатывает только первое нажатие, на второе отвечает "Роль уже выбрана".
- Telegram ID пользователя изменился (пересоздание аккаунта) — бот воспринимает как нового пользователя.
- Организатор удалён из whitelist после выбора роли — при следующем входе бот предлагает сменить роль, роль "Организатор" недоступна.
- Бот получает update без текста (стикер, фото) до выбора роли — бот повторяет предложение выбрать роль через кнопки.
- Одновременный вход 200+ пользователей в начале DD (пик) — система обрабатывает параллельные запросы без потери данных (NFR: 200 concurrent sessions).

## Clarifications

### Session 2026-02-02

- Q: Может ли пользователь иметь несколько ролей одновременно? → A: Нет. Одна роль на событие. При смене — предыдущая заменяется.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST показывать приветственное сообщение с описанием бота при получении команды `/start` от нового пользователя.
- **FR-002**: Система MUST предлагать выбор роли через 5 inline-кнопок: Организатор, Эксперт, Студент, Гость, Бизнес-партнёр.
- **FR-003**: Система MUST проверять Telegram ID пользователя по whitelist при выборе роли "Организатор" и отклонять неавторизованных.
- **FR-004**: Система MUST показывать экран выбора подтипа (Абитуриент / AI-практик / Другое) только при выборе роли "Гость".
- **FR-005**: Система MUST сохранять Telegram ID, имя пользователя, выбранную роль и подтип (для гостей) при первом выборе.
- **FR-006**: Система MUST показывать главное меню, соответствующее выбранной роли, после завершения онбординга.
- **FR-007**: Система MUST предлагать "Продолжить" или "Сменить роль" при повторном `/start` от зарегистрированного пользователя.
- **FR-008**: Система MUST поддерживать команду `/role` для просмотра текущей роли и возможности смены.
- **FR-009**: Система MUST игнорировать нетекстовые сообщения (стикеры, фото) до завершения онбординга и повторять предложение выбрать роль.
- **FR-010**: Система MUST обрабатывать повторное нажатие inline-кнопки (callback query) без создания дублирующих записей.

### Key Entities

- **Пользователь (User)**: Telegram ID (уникальный идентификатор), полное имя, username, организация. Имеет ровно одну роль в рамках события; при смене роли предыдущая заменяется.
- **Роль (Role)**: Код роли (organizer, student, expert, guest, business), отображаемое название. Справочник из 5 записей.
- **Привязка роли (User Role)**: Связь пользователя с ролью в рамках конкретного события. Содержит подтип гостя (если применимо).
- **Событие (Event)**: Demo Day как сущность с датами и настройками. Онбординг привязывает пользователя к текущему событию.

### Assumptions

- Whitelist организаторов задаётся заранее через конфигурацию (переменные окружения или seed-данные), не через интерфейс бота.
- Для демо 6 февраля — одно событие, предзагруженное в систему. Мультисобытийность — Release 1.1.
- Имя пользователя берётся из Telegram-профиля (first_name + last_name), не запрашивается отдельно.
- Главное меню каждой роли — заглушка (текст + кнопки-placeholder) до реализации соответствующих EPIC.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Пользователь завершает онбординг (от `/start` до меню роли) не более чем за 30 секунд и 3 нажатия кнопок.
- **SC-002**: 100% пользователей, отправивших `/start`, видят экран выбора роли без ошибок.
- **SC-003**: Система корректно обрабатывает 200 одновременных `/start`-запросов без потери данных и без увеличения времени отклика сверх 2 секунд.
- **SC-004**: Неавторизованный пользователь не может получить роль "Организатор" — 100% попыток отклоняются.
- **SC-005**: При повторном входе пользователь видит свою текущую роль и может сменить её за 2 нажатия.
