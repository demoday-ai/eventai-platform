## User Scenarios & Testing

### User Story 1 - Запуск и просмотр матчинга экспертов (Priority: P1)

Организатор открывает страницу «Эксперты» → таб «Матчинг». Если кластеризация одобрена и эксперты загружены, организатор видит wizard с шагами: Параметры → Результат → Одобрение. Организатор запускает автоматическое распределение экспертов по залам, просматривает результат (какой эксперт в каком зале) и при необходимости перераспределяет экспертов вручную.

**Why this priority**: Матчинг экспертов — ключевое действие эпика. Без распределения экспертов невозможно перейти к расписанию.

**Independent Test**: Можно протестировать изолированно — при наличии залов и экспертов запустить матчинг и убедиться, что все залы получили экспертов.

**Acceptance Scenarios**:

1. **Given** кластеризация одобрена и эксперты загружены, **When** организатор открывает таб «Матчинг», **Then** отображается wizard с параметрами запуска
2. **Given** wizard на шаге «Параметры», **When** организатор нажимает «Запустить», **Then** система запускает автоматическое распределение и показывает прогресс
3. **Given** матчинг завершён, **When** организатор просматривает результат, **Then** отображаются залы с назначенными экспертами, их компетенциями и количеством проектов
4. **Given** результат матчинга отображён, **When** организатор перемещает эксперта в другой зал, **Then** статистика покрытия обновляется
5. **Given** кластеризация НЕ одобрена, **When** организатор открывает таб «Матчинг», **Then** отображается empty state: «Для матчинга одобрите кластеризацию» со ссылкой на страницу кластеризации
6. **Given** эксперты НЕ загружены, **When** организатор открывает таб «Матчинг», **Then** отображается empty state: «Загрузите экспертов на странице Импорта» со ссылкой

### User Story 2 - Одобрение матчинга (Priority: P1)

Организатор, удовлетворённый распределением экспертов, одобряет матчинг. При нажатии кнопки «Одобрить» появляется диалог подтверждения. После подтверждения матчинг фиксируется, отображается подсказка о следующем шаге (генерация расписания) со ссылкой, а глобальный прогресс обновляется.

**Why this priority**: Одобрение — критический шаг, без которого невозможно перейти к следующей фазе (расписание).

**Independent Test**: Можно протестировать отдельно — при наличии результата матчинга нажать «Одобрить», подтвердить и проверить обновление статуса.

**Acceptance Scenarios**:

1. **Given** матчинг завершён и результат отображён, **When** организатор нажимает «Одобрить», **Then** появляется диалог подтверждения с текстом «Вы уверены?» и кнопками «Подтвердить» / «Отмена»
2. **Given** диалог подтверждения открыт, **When** организатор нажимает «Подтвердить», **Then** матчинг фиксируется и отображается сообщение «Матчинг одобрен. Следующий шаг — генерация расписания» со ссылкой на страницу расписания
3. **Given** диалог подтверждения открыт, **When** организатор нажимает «Отмена», **Then** диалог закрывается без изменений
4. **Given** матчинг одобрён, **When** организатор просматривает глобальный прогресс, **Then** шаг «Матчинг экспертов» отмечен как завершённый
5. **Given** матчинг одобрён, **When** организатор просматривает страницу, **Then** отображается подсказка «Перейти к расписанию» со ссылкой на /schedule

### User Story 3 - Список экспертов с CRUD (Priority: P2)

Организатор открывает таб «Список» на странице экспертов и видит таблицу всех загруженных экспертов с поиском и фильтрами. Организатор может добавить нового эксперта, отредактировать существующего или удалить.

**Why this priority**: Управление списком экспертов важно, но вторично по отношению к матчингу — данные обычно загружаются через Импорт.

**Independent Test**: Можно протестировать отдельно — открыть таб «Список», проверить отображение таблицы, добавить эксперта, найти через поиск.

**Acceptance Scenarios**:

1. **Given** эксперты загружены, **When** организатор открывает таб «Список», **Then** отображается таблица с колонками: имя, контакт, теги/компетенции, назначенный зал
2. **Given** таблица экспертов отображена, **When** организатор вводит текст в поле поиска, **Then** таблица фильтруется по имени или компетенциям
3. **Given** таблица экспертов отображена, **When** организатор нажимает «Добавить эксперта», **Then** открывается форма с полями: имя, контакт (Telegram), теги/компетенции
4. **Given** форма добавления заполнена, **When** организатор нажимает «Сохранить», **Then** эксперт добавлен в список и таблица обновляется
5. **Given** таблица экспертов отображена, **When** организатор нажимает на эксперта для редактирования, **Then** открывается форма с предзаполненными данными
6. **Given** организатор нажимает «Удалить» у эксперта, **When** подтверждает действие, **Then** эксперт удалён из списка
7. **Given** экспертов нет, **When** организатор открывает таб «Список», **Then** отображается empty state: «Экспертов пока нет. Загрузите через Импорт или добавьте вручную»

### User Story 4 - Покрытие залов экспертами (Priority: P2)

Организатор открывает таб «Покрытие» и видит сводку: какие залы покрыты экспертами, где есть пробелы, какие залы требуют внимания (эскалации). Организатор может перейти к деталям конкретного зала.

**Why this priority**: Покрытие — индикатор качества матчинга. Важно для принятия решения об одобрении, но вторично по отношению к самому матчингу.

**Independent Test**: Можно протестировать отдельно — при наличии матчинга проверить сводку покрытия и переход к деталям.

**Acceptance Scenarios**:

1. **Given** матчинг выполнен, **When** организатор открывает таб «Покрытие», **Then** отображается сводка по залам: название зала, количество проектов, количество экспертов, статус покрытия (достаточно / недостаточно / нет экспертов)
2. **Given** зал имеет недостаточное покрытие, **When** организатор смотрит на этот зал, **Then** он выделен жёлтым (внимание) или красным (эскалация)
3. **Given** сводка покрытия отображена, **When** организатор нажимает на зал, **Then** открывается детальная страница зала с проектами и назначенными экспертами
4. **Given** матчинг НЕ выполнен, **When** организатор открывает таб «Покрытие», **Then** отображается empty state: «Запустите матчинг экспертов для просмотра покрытия» со ссылкой на таб «Матчинг»

---

### User Story 5 - Рекомендации по закрытию пробелов (Priority: P3)

На детальной странице зала с недостаточным покрытием организатор видит список рекомендуемых экспертов (подобранных по совпадению тегов/компетенций). Организатор может назначить подходящего эксперта одной кнопкой «Назначить».

**Why this priority**: Рекомендации ускоряют работу, но организатор может закрывать пробелы и вручную через перемещение на странице матчинга.

**Independent Test**: Можно протестировать отдельно — на странице зала с пробелом проверить наличие рекомендаций и назначить эксперта.

**Acceptance Scenarios**:

1. **Given** зал имеет незакрытый пробел и есть свободные эксперты с подходящими компетенциями, **When** организатор просматривает детали зала, **Then** отображается список рекомендуемых экспертов с их компетенциями
2. **Given** список рекомендаций отображён, **When** организатор нажимает «Назначить» у эксперта, **Then** эксперт добавляется в зал, статистика покрытия обновляется
3. **Given** зал имеет пробел, но нет свободных экспертов с подходящими компетенциями, **When** организатор просматривает детали зала, **Then** отображается сообщение «Нет экспертов с подходящими компетенциями»

---

### Edge Cases

- Что происходит при попытке запустить матчинг без загруженных экспертов? → Empty state с подсказкой загрузить экспертов
- Что происходит при попытке запустить матчинг без одобренной кластеризации? → Empty state с подсказкой одобрить кластеризацию
- Что если все эксперты уже назначены и свободных нет? → Рекомендации показывают «Нет свободных экспертов»
- Что если матчинг уже одобрён и организатор пытается перезапустить? → Предупреждение о сбросе одобрения
- Что если зал пуст (нет проектов)? → Зал отображается, но эксперты не назначаются автоматически

## Requirements

### Functional Requirements

- **FR-001**: Система ДОЛЖНА предоставлять wizard-flow для матчинга экспертов (Параметры → Результат → Одобрение)
- **FR-002**: Система ДОЛЖНА проверять наличие одобренной кластеризации перед запуском матчинга
- **FR-003**: Система ДОЛЖНА проверять наличие загруженных экспертов перед запуском матчинга
- **FR-004**: Система ДОЛЖНА автоматически распределять экспертов по залам на основе их компетенций и тематики залов
- **FR-005**: Система ДОЛЖНА позволять вручную перемещать экспертов между залами после автоматического матчинга
- **FR-006**: Система ДОЛЖНА показывать диалог подтверждения при одобрении матчинга
- **FR-007**: После одобрения матчинга система ДОЛЖНА обновлять глобальный прогресс и показывать подсказку со ссылкой на следующий шаг (расписание)
- **FR-008**: Система ДОЛЖНА предоставлять таблицу экспертов с поиском и фильтрами (таб «Список»)
- **FR-009**: Система ДОЛЖНА позволять добавлять, редактировать и удалять экспертов (CRUD)
- **FR-010**: Система ДОЛЖНА отображать сводку покрытия залов экспертами (таб «Покрытие»)
- **FR-011**: Система ДОЛЖНА визуально выделять залы с недостаточным покрытием (жёлтый — внимание, красный — эскалация)
- **FR-012**: Система ДОЛЖНА позволять переход к детальной странице конкретного зала
- **FR-013**: Система ДОЛЖНА показывать рекомендации по экспертам для закрытия пробелов покрытия (по совпадению тегов)
- **FR-014**: Система ДОЛЖНА позволять назначить рекомендуемого эксперта в зал кнопкой «Назначить»
- **FR-015**: Система ДОЛЖНА показывать empty state с подсказкой и ссылкой при отсутствии необходимых данных (нет экспертов, нет кластеризации, нет матчинга)

### Key Entities

- **Эксперт**: Представляет оценщика/модератора. Ключевые атрибуты: имя, контакт (Telegram), теги/компетенции, назначенный зал
- **Зал (Room)**: Тематическая группа проектов. Связан с проектами через кластеризацию и с экспертами через матчинг
- **Матчинг (MatchingRun)**: Результат автоматического распределения. Содержит статус (pending/completed/approved), связи эксперт↔зал
- **Покрытие (Coverage)**: Агрегированная метрика: количество экспертов на зал, статус (достаточно/недостаточно/нет)
- **Рекомендация**: Предложение эксперта для зала на основе совпадения тегов

## Success Criteria

### Measurable Outcomes

- **SC-001**: Организатор может запустить матчинг и увидеть результат за менее чем 30 секунд
- **SC-002**: Организатор может одобрить матчинг за 3 клика (кнопка → подтверждение → готово)
- **SC-003**: 100% залов с недостаточным покрытием визуально выделены в сводке
- **SC-004**: Рекомендации по экспертам отображаются для каждого зала с пробелом (при наличии подходящих экспертов)
- **SC-005**: Организатор может найти нужного эксперта через поиск за менее чем 5 секунд
- **SC-006**: Все страницы без данных показывают понятный empty state с подсказкой и ссылкой на следующее действие
- **SC-007**: После одобрения матчинга глобальный прогресс обновляется без перезагрузки страницы

## Assumptions

- Кластеризация (EPIC-005) уже реализована и предоставляет одобренные залы с проектами
- Импорт экспертов (EPIC-003) уже реализован и предоставляет список экспертов с тегами
- Алгоритм матчинга (backend) уже реализован — эпик покрывает только frontend-wizard
- Максимальное количество экспертов — до 100, залов — до 20
- Один эксперт может быть назначен только в один зал одновременно

## Scope Boundaries

**В scope**:
- Frontend wizard для матчинга экспертов
- CRUD для экспертов (таб «Список»)
- Сводка покрытия (таб «Покрытие»)
- Детальная страница зала
- Рекомендации по экспертам для пробелов
- Диалог подтверждения одобрения
- Подсказка о следующем шаге
- Обновление глобального прогресса
- Empty states для всех табов

**Вне scope**:
- Backend алгоритм матчинга (уже реализован)
- Импорт экспертов из файлов (EPIC-003)
- Уведомления экспертам (EPIC-008/009)
- Drag-and-drop для перемещения экспертов (используем кнопки)
