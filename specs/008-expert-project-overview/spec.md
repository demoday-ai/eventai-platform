# Feature Specification: Expert Project Overview

**Feature Branch**: `008-expert-project-overview`
**Created**: 2026-02-03
**Status**: Draft

## Overview

За день до Demo Day система автоматически отправляет каждому эксперту персональный брифинг с карточками проектов его комнаты. Это позволяет эксперту подготовиться: изучить проекты, продумать вопросы, понять технический стек участников.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Автоматическая отправка брифинга эксперту (Priority: P1)

За день до Demo Day система отправляет эксперту в Telegram сообщение с карточками всех проектов его комнаты. Эксперт получает структурированную информацию по каждому проекту: название, автор, краткое описание, технологический стек, теги.

**Why this priority**: Основная ценность эпика — дать экспертам возможность подготовиться к оценке проектов. Без этого эксперты приходят "вслепую".

**Independent Test**: Эксперт с назначенной комнатой получает сообщение за день до DD с карточками проектов — может прочитать и подготовиться.

**Acceptance Scenarios**:

1. **Given** эксперт подтвердил участие (status=confirmed) и назначен на комнату, **When** наступает день перед DD (за 24 часа до начала), **Then** эксперт получает Telegram-сообщение с карточками всех проектов его комнаты
2. **Given** эксперт получил брифинг, **When** он открывает сообщение, **Then** он видит по каждому проекту: название, автор, описание (до 200 символов), стек, теги
3. **Given** в комнате 15 проектов, **When** система формирует брифинг, **Then** все 15 проектов включены в сообщение (с пагинацией если нужно)

---

### User Story 2 - Информация о GitHub-репозитории (Priority: P2)

Карточка проекта включает информацию о наличии GitHub-репозитория и базовую статистику активности (есть/нет, последний коммит).

**Why this priority**: GitHub-статус помогает эксперту понять серьёзность проекта и готовность команды, но не критичен для базовой подготовки.

**Independent Test**: Эксперт видит в карточке проекта статус GitHub — "Репозиторий: ✓ активный" или "Репозиторий: нет".

**Acceptance Scenarios**:

1. **Given** у проекта есть GitHub-ссылка в данных, **When** формируется карточка, **Then** отображается "GitHub: ✓" с датой последнего коммита (если доступно)
2. **Given** у проекта нет GitHub-ссылки, **When** формируется карточка, **Then** отображается "GitHub: —"
3. **Given** GitHub-ссылка есть, но репозиторий недоступен, **When** формируется карточка, **Then** отображается "GitHub: ⚠️ недоступен"

---

### User Story 3 - Ссылки на артефакты проекта (Priority: P3)

Карточка проекта включает ссылки на ключевые артефакты: презентация (если загружена), демо-видео или ссылка на демо.

**Why this priority**: Артефакты полезны для глубокой подготовки, но большинство экспертов ограничится базовой информацией из US1.

**Independent Test**: Эксперт видит в карточке кликабельные ссылки на презентацию и демо (если есть).

**Acceptance Scenarios**:

1. **Given** у проекта есть ссылка на презентацию, **When** формируется карточка, **Then** отображается кликабельная ссылка "📊 Презентация"
2. **Given** у проекта есть демо-видео или ссылка на демо, **When** формируется карточка, **Then** отображается кликабельная ссылка "🎬 Демо"
3. **Given** у проекта нет артефактов, **When** формируется карточка, **Then** секция артефактов не отображается

---

### Edge Cases

- **Эксперт без комнаты**: Эксперты без назначенной комнаты не получают брифинг (логируется как warning)
- **Пустая комната**: Если в комнате 0 проектов, эксперт получает сообщение "В вашей комнате пока нет проектов"
- **Длинный брифинг**: Если сообщение превышает лимит Telegram (4096 символов), разбивается на несколько сообщений
- **Заблокированный бот**: Если эксперт заблокировал бота, ошибка логируется, организатор видит это в отчёте
- **Отменивший эксперт**: Эксперты со статусом declined/cancelled не получают брифинг
- **GitHub API недоступен**: Если GitHub API не отвечает или rate limit, отображается "GitHub: ⚠️ статус недоступен"

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система ДОЛЖНА отправлять брифинг экспертам автоматически за 24 часа до начала Demo Day
- **FR-002**: Система ДОЛЖНА включать в брифинг только проекты комнаты, на которую назначен эксперт
- **FR-003**: Карточка проекта ДОЛЖНА содержать: название, автор (имя студента/команды), описание (до 200 символов), теги
- **FR-004**: Система ДОЛЖНА отображать технологический стек проекта (если указан в данных)
- **FR-005**: Система ДОЛЖНА проверять GitHub API при формировании брифинга и отображать статус репозитория (есть/нет/недоступен) с датой последнего коммита
- **FR-006**: Система ДОЛЖНА включать ссылки на артефакты (презентация, демо) если они есть
- **FR-007**: Система ДОЛЖНА разбивать длинные брифинги на несколько сообщений при превышении лимита Telegram
- **FR-008**: Система ДОЛЖНА отправлять брифинг только экспертам со статусом confirmed
- **FR-009**: Система ДОЛЖНА логировать все отправки и ошибки доставки
- **FR-010**: Организатор ДОЛЖЕН иметь возможность вручную запустить рассылку брифингов через команду `/briefing`

### Key Entities

- **ExpertBriefing**: Запись об отправке брифинга эксперту (expert_id, room_id, sent_at, status, project_count)
- **ProjectCard**: Структура карточки проекта для отображения (project_id, title, author, description, tags, stack, github_status, artifacts)

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% экспертов со статусом confirmed получают брифинг за 24 часа до DD
- **SC-002**: Каждый эксперт видит все проекты своей комнаты (полнота 100%)
- **SC-003**: Брифинг доставляется в течение 5 минут после запуска рассылки
- **SC-004**: Организатор видит отчёт о доставке с указанием успешных/неуспешных отправок

## Assumptions

- Данные о проектах (название, описание, теги, стек) уже есть в системе из EPIC-002 (Project Clustering)
- GitHub-ссылки и артефакты берутся из данных контрольных рубежей (если загружены)
- Telegram chat_id экспертов есть в системе из EPIC-004 (Expert Assignment)
- Рассылка использует существующую инфраструктуру напоминаний из EPIC-007

## Out of Scope

- Интерактивные элементы в брифинге (кнопки, inline-клавиатуры)
- Персонализация карточек под профиль эксперта
- Уведомления об изменениях в проектах после отправки брифинга
