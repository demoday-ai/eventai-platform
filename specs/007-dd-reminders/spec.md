# Feature Specification: Напоминания перед Demo Day (EPIC-007)

**Feature Branch**: `007-dd-reminders`
**Created**: 2026-02-03
**Status**: Draft
**Input**: User description: "EPIC-007 — Напоминалки"

## Context

На прошлом Demo Day 15 проектов не пришли, 80 студентов не заполнили форму, часть экспертов не явилась — всё выясняется в последний момент. Гости пропускали интересные проекты из-за сдвигов тайминга. Нет единого механизма напоминаний для всех ролей.

EPIC-007 (RICE 780 — наивысший приоритет) реализует систему автоматических напоминаний всем 5 ролям перед Demo Day: за день и за час до события. Это закрывает SS-008 из User Story Map и адресует Pain #2 (неподтверждённые студенты), Pain #3 (нехватка экспертов), а также JTBD-7 (знать свой слот).

Частично инфраструктура уже существует: напоминания студентам (за 5 дней до DD с кнопкой «Ознакомлен») и напоминания экспертам (через 3 дня после просмотра приглашения). EPIC-007 добавляет новый слой: напоминания «за день» и «за час» для всех ролей, а также уведомления гостям/бизнесу с их персональной программой.

## Assumptions

- Событие (Event) имеет дату начала (`start_date`), по которой рассчитываются таймеры напоминаний
- У студентов, экспертов, гостей и бизнес-партнёров есть telegram_user_id для отправки сообщений
- У гостей и бизнес-партнёров уже есть сохранённая персональная программа (EPIC-005)
- Напоминания отправляются через Telegram-бота (единственный канал для MVP)
- Организаторы не получают напоминания о себе — они получают сводку «кто не подтвердил» (уже реализовано в EPIC-003)
- Слоты по времени не привязаны к конкретным проектам (нет точного расписания по часам внутри зала в текущей модели данных) — напоминание указывает зал и общее время работы зала
- Запуск напоминаний: организатор вручную через команду `/remind` или автоматически по расписанию

## User Scenarios & Testing

### User Story 1 — Напоминание «за день» студентам и экспертам (Priority: P1)

За день до Demo Day система отправляет персональное напоминание каждому студенту и каждому эксперту с подтверждённым участием: «Завтра, Зал X. Готовься!» Студент видит своё время и зал, эксперт — свой зал и количество проектов. Это критически снижает неявки.

**Why this priority**: Неявки студентов и экспертов — главная боль организаторов (Pain #2, #3, RICE 780). Минимальное действие с максимальным эффектом: одно сообщение каждому за день.

**Independent Test**: Создать событие с start_date = завтра, назначить студентов и экспертов в залы. Запустить отправку напоминаний — проверить что каждый получил персональное сообщение с названием зала.

**Acceptance Scenarios**:

1. **Given** событие с start_date = завтра и студент с подтверждённым слотом, **When** запускается рассылка «за день», **Then** студент получает сообщение: «Завтра ты выступаешь! Зал «X», проект «Y»»
2. **Given** эксперт со статусом confirmed и назначением в зал, **When** запускается рассылка «за день», **Then** эксперт получает: «Завтра, Зал «X». N проектов ждут вашей оценки»
3. **Given** студент без подтверждения (не нажал «Ознакомлен»), **When** запускается рассылка, **Then** студент всё равно получает напоминание с пометкой «Пожалуйста, подтвердите участие»
4. **Given** эксперт со статусом declined, **When** запускается рассылка, **Then** отклонившему эксперту напоминание НЕ отправляется

---

### User Story 2 — Напоминание «за день» гостям и бизнес-партнёрам (Priority: P1)

Гости и бизнес-партнёры получают напоминание за день до DD с кратким содержанием их персональной программы: топ-проекты из рекомендаций, залы, которые стоит посетить. Это повышает вовлечённость и помогает подготовиться.

**Why this priority**: Гости — ключевая аудитория DD (бизнес-партнёры, инвесторы, HR). Без напоминания часть не придёт или придёт без плана.

**Independent Test**: Создать гостя с профилем и сохранённой программой. Запустить рассылку «за день» — проверить что получил сообщение с топ-проектами из программы.

**Acceptance Scenarios**:

1. **Given** гость с сохранённой программой, **When** запускается рассылка «за день», **Then** гость получает: «Завтра DD! Ваша программа: Проект A (Зал X), Проект B (Зал Y), ...»
2. **Given** бизнес-партнёр с профилем, **When** запускается рассылка, **Then** получает напоминание с топ-5 проектами из его подборки
3. **Given** гость без сохранённой программы, **When** запускается рассылка, **Then** получает общее напоминание: «Завтра DD! Пройдите профилирование, чтобы получить персональную программу»

---

### User Story 3 — Напоминание «за час» всем участникам (Priority: P2)

За час до начала DD система отправляет короткое напоминание: студенту — «Через час — твоё выступление», эксперту — «Через час — Зал X», гостям — «DD начинается через час. Вот ваша программа».

**Why this priority**: CustDev подтвердил: «напоминалка за день, за полдня, за час» (100% confidence). Напоминание за час — финальный толчок для тех, кто забыл.

**Independent Test**: Установить start_date на текущий день, время начала через 1 час. Запустить рассылку «за час» — проверить что все роли получили краткое напоминание.

**Acceptance Scenarios**:

1. **Given** событие начинается через 1 час, **When** запускается рассылка «за час», **Then** студент получает: «Через час — твоё выступление! Зал «X»»
2. **Given** эксперт confirmed, **When** рассылка «за час», **Then** получает: «Через час — Зал «X». Первый проект: «Y»»
3. **Given** гость с программой, **When** рассылка «за час», **Then** получает: «DD начинается через час! Рекомендуем начать с Зала «X»»

---

### User Story 4 — Организатор управляет рассылкой (Priority: P2)

Организатор запускает рассылку напоминаний вручную командой `/remind` в боте. Бот показывает превью: сколько сообщений будет отправлено каждой роли. Организатор подтверждает — рассылка начинается. После завершения организатор видит отчёт: отправлено / не удалось.

**Why this priority**: Организатор контролирует момент отправки и видит охват. Без этого нет механизма запуска.

**Independent Test**: Отправить `/remind` как организатор — увидеть превью с количеством получателей по ролям. Подтвердить — получить отчёт.

**Acceptance Scenarios**:

1. **Given** организатор отправляет `/remind`, **When** есть событие с датой DD завтра, **Then** бот показывает: «Напоминания: N студентов, M экспертов, K гостей, L бизнес. Отправить?»
2. **Given** организатор подтверждает рассылку, **When** отправка завершена, **Then** бот показывает отчёт: «Отправлено: X. Ошибки: Y»
3. **Given** организатор отправляет `/remind`, **When** нет события в ближайшие 2 дня, **Then** бот сообщает: «Нет события в ближайшие дни»
4. **Given** не-организатор отправляет `/remind`, **Then** бот отвечает: «Команда доступна только организаторам»

---

### Edge Cases

- Что если у пользователя нет telegram_user_id? — Пропустить, включить в отчёт как «не отправлено (нет Telegram)»
- Что если бот не может отправить сообщение (пользователь заблокировал бота)? — Зафиксировать ошибку, продолжить рассылку остальным, включить в отчёт
- Что если напоминание уже было отправлено (повторный запуск)? — Предупредить организатора: «Напоминания уже отправлены X минут назад. Отправить повторно?»
- Что если событие сегодня, а не завтра? — Организатор может выбрать тип: «за день» или «за час»
- Что если у студента нет назначенного зала (нет room_project)? — Пропустить, включить в отчёт
- Что если рассылка прервалась (ошибка сети)? — Сохранять прогресс, при повторном запуске отправлять только тем, кому ещё не отправлено
- Что если Telegram rate limit (30 msg/sec)? — Отправлять с задержкой между сообщениями

## Requirements

### Functional Requirements

- **FR-001**: Система ДОЛЖНА отправлять персональные напоминания «за день» студентам с указанием зала и проекта
- **FR-002**: Система ДОЛЖНА отправлять персональные напоминания «за день» экспертам с указанием зала и количества проектов
- **FR-003**: Система ДОЛЖНА отправлять напоминания «за день» гостям с содержанием их персональной программы (топ-проекты)
- **FR-004**: Система ДОЛЖНА отправлять напоминания «за день» бизнес-партнёрам с их персональной подборкой проектов
- **FR-005**: Система ДОЛЖНА отправлять краткие напоминания «за час» всем активным участникам
- **FR-006**: Организатор ДОЛЖЕН иметь возможность запустить рассылку командой `/remind`
- **FR-007**: Перед отправкой система ДОЛЖНА показать превью: количество получателей по каждой роли
- **FR-008**: После отправки система ДОЛЖНА показать отчёт: количество отправленных и ошибок
- **FR-009**: Система ДОЛЖНА предупреждать о повторной отправке, если напоминания уже были отправлены
- **FR-010**: Система ДОЛЖНА пропускать пользователей без telegram_user_id и фиксировать их в отчёте
- **FR-011**: Система ДОЛЖНА соблюдать лимиты Telegram при отправке (задержка между сообщениями)
- **FR-012**: Доступ к команде `/remind` ДОЛЖЕН быть ограничен организаторами

### Key Entities

- **Notification**: Запись об отправленном напоминании — кому (user_id), тип (day_before / hour_before), статус (pending / sent / failed), время отправки
- **ReminderBatch**: Пакет рассылки — кто запустил, тип напоминания, количество получателей, количество отправленных/ошибок, время начала/окончания

## Success Criteria

### Measurable Outcomes

- **SC-001**: Все активные участники (студенты с подтверждением, эксперты confirmed, гости и бизнес с профилем) получают напоминание за день до DD за одно действие организатора
- **SC-002**: Организатор видит полный отчёт о рассылке в течение 5 минут после запуска (для ~400 получателей)
- **SC-003**: Не менее 95% целевых получателей успешно получают напоминание (ошибки только из-за блокировки бота)
- **SC-004**: Время от запуска до завершения рассылки: менее 3 минут для 400 получателей
- **SC-005**: Организатор может предотвратить случайную повторную рассылку благодаря предупреждению о дубликате
