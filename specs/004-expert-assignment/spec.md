# Feature Specification: Распределение экспертов

**Feature Branch**: `004-expert-assignment`
**Created**: 2026-02-02
**Status**: Draft
**Input**: User description: "[EPIC-004] Распределение экспертов"

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Автоматический матчинг экспертов по комнатам (Priority: P1)

Организатор загружает список экспертов (294 эксперта, 31 тег) и список комнат с тематиками. Система сопоставляет теги интересов экспертов с тематиками комнат и предлагает распределение: какой эксперт в какую комнату. Организатор видит предложение, может скорректировать вручную и утвердить.

**Why this priority**: Ручное распределение — корневая проблема ("Инна знает контекст, бот — нет"). Автоматический матчинг — базовая ценность фичи (RICE H3 = 55, 100% confidence). Без него всё остальное (приглашения, подтверждения) невозможно.

**Independent Test**: Можно протестировать, загрузив список экспертов с тегами и список комнат с тематиками, и проверив, что система предлагает релевантное распределение (эксперт с тегами NLP/LLM попадает в NLP-комнату, а не в FinTech).

**Acceptance Scenarios**:

1. **Given** загружен список экспертов с тегами и список комнат с тематиками, **When** организатор запускает автоматический матчинг, **Then** система предлагает распределение: для каждой комнаты — список рекомендованных экспертов с указанием степени совпадения тегов.
2. **Given** система предложила распределение, **When** организатор видит, что эксперт назначен не в ту комнату, **Then** может перенести эксперта в другую комнату вручную, и система пересчитывает покрытие.
3. **Given** у эксперта слишком широкие теги (10+ из 31), **When** система матчит, **Then** учитывает приоритет (вес) тегов, а не просто количество совпадений. Если приоритеты не заданы — использует наиболее редкие теги эксперта.
4. **Given** комната имеет слабое покрытие экспертами (например, FinTech: 8 экспертов, Security: 2), **When** система матчит, **Then** выделяет эти комнаты как "требующие внимания" и предлагает экспертов со смежными тегами.

---

### User Story 2 — Персональное приглашение эксперта (Priority: P1)

После утверждения распределения система отправляет каждому эксперту персональное приглашение в Telegram: "Приглашаем на DD! По вашим интересам (NLP, Agents) подходит Зал 3. Вот проекты этой комнаты." Эксперт может подтвердить, запросить другую комнату или отказаться.

**Why this priority**: Без приглашения невозможно получить подтверждение. CustDev: "бот должен сопоставлять" (Инна), "могу тайминг расставить" (Рустем). Эксперт хочет видеть конкретику, а не абстрактное "приходи".

**Independent Test**: Можно протестировать, утвердив распределение и проверив, что эксперт с Telegram-контактом получает сообщение с названием комнаты, списком тематик и кнопками действий.

**Acceptance Scenarios**:

1. **Given** распределение утверждено организатором и у эксперта указан Telegram-контакт, **When** система запускает рассылку приглашений, **Then** эксперт получает сообщение: "Приглашаем на DD [дата]! По вашим интересам ([теги]) подходит [Зал N]. [Кол-во] проектов." с кнопками "Иду" / "Хочу другую комнату" / "Не смогу".
2. **Given** эксперт нажимает "Иду", **When** система обрабатывает ответ, **Then** эксперт получает подтверждение "Вы записаны в [Зал N]" и его статус обновляется на "подтверждён".
3. **Given** эксперт нажимает "Хочу другую комнату", **When** система обрабатывает запрос, **Then** предлагает список альтернативных комнат с тематиками и количеством проектов.
4. **Given** эксперт нажимает "Не смогу", **When** система обрабатывает отказ, **Then** статус обновляется на "отказ", организатор видит обновлённую статистику покрытия.
5. **Given** у эксперта нет Telegram-контакта, **When** система запускает рассылку, **Then** эксперт попадает в список "без контакта" для ручной связи организатором.

### User Story 3 — Дашборд покрытия для организатора (Priority: P1)

Организатор видит сводную картину: по каждой комнате — сколько экспертов подтвердили, сколько отказались, сколько не ответили, сколько нужно. Комнаты с недостаточным покрытием выделены визуально.

**Why this priority**: Организатор — главный бенефициар. Цитата Инны: "Если я закрою все секции экспертами и участниками вовремя — событие уже можно считать успешным." Без обзорной картины организатор не может действовать.

**Independent Test**: Можно протестировать, создав комнаты с разным уровнем подтверждений и проверив, что организатор видит корректную сводку с цветовой индикацией.

**Acceptance Scenarios**:

1. **Given** рассылка приглашений запущена, **When** организатор запрашивает покрытие, **Then** видит по каждой комнате: "Зал N: [тематика] — Подтверждено: X / Нужно: 2, Отказ: Z, Нет ответа: W" с индикацией: 2+ подтверждённых = покрыто (зелёный), 1 = частично (жёлтый), 0 = не покрыто (красный).
2. **Given** комната не покрыта (0 подтверждённых экспертов), **When** организатор видит дашборд, **Then** комната выделена как критическая, и система предлагает экспертов со смежными тегами, которые ещё не подтвердили участие.
3. **Given** эксперт меняет статус (подтверждает/отказывается), **When** организатор обновляет дашборд, **Then** видит актуальную статистику в реальном времени.

---

### User Story 4 — Выбор временных слотов экспертом (Priority: P2)

Эксперт при подтверждении может указать, в какие временные слоты ему удобно (совмещение с работой). Система учитывает это при финальном распределении.

**Why this priority**: CustDev: "могу тайминг расставить, какие слоты могу, а какие нет" (Рустем). Повышает конверсию подтверждений, но не блокирует базовый флоу (можно запустить без слотов).

**Independent Test**: Можно протестировать, предложив эксперту слоты и проверив, что выбранные слоты сохраняются и учитываются.

**Acceptance Scenarios**:

1. **Given** эксперт нажал "Иду", **When** система спрашивает о временных предпочтениях, **Then** эксперт видит список доступных временных слотов дня DD и может выбрать удобные.
2. **Given** эксперт выбрал слоты с 14:00 до 17:00, **When** организатор смотрит покрытие по времени, **Then** видит, что с 17:00 до 19:30 в этой комнате нет эксперта.

---

### User Story 5 — Эскалация и напоминания неотвечающим экспертам (Priority: P2)

Если эксперт не ответил на приглашение, система автоматически напоминает, а затем эскалирует организатору список молчащих экспертов.

**Why this priority**: ~20% экспертов не приходят. Без эскалации организатор узнаёт об этом слишком поздно. Каждый неподтверждённый эксперт — потенциально непокрытая комната.

**Independent Test**: Можно протестировать, создав эксперта без ответа и проверив, что через заданный срок он получает напоминание, а организатор — алерт.

**Acceptance Scenarios**:

1. **Given** эксперт не ответил в течение 3 дней, **When** наступает дедлайн напоминания, **Then** эксперт получает повторное сообщение: "Напоминаем: DD [дата], вас ждём в [Зал N]. Подтвердите участие."
2. **Given** эксперт не ответил в течение 5 дней (или за 2 дня до DD — что наступит раньше), **When** наступает дедлайн эскалации, **Then** организатор получает алерт со списком неответивших экспертов и комнат, которые остаются без покрытия.
3. **Given** общее количество сообщений эксперту, **When** система отправляет напоминания, **Then** не превышает 4 сообщения за весь цикл DD (приглашение, напоминание, овервью, напоминалка в день DD).

---

### Edge Cases

- Что если эксперт подтвердил, но в день DD не пришёл (no-show)? — Организатор может вручную отметить no-show. Комната помечается как "без эксперта" в реальном времени.
- Что если у эксперта 10+ тегов (ему "всё интересно")? — Система использует наиболее редкие теги (FinTech: 8 экспертов важнее, чем LLM: 29) или запрашивает у эксперта приоритизацию при онбординге.
- Что если в комнате Security всего 2 эксперта и оба отказались? — Система алертит организатора, предлагает экспертов со смежными тегами (Cybersecurity, ML in Industry). Организатор решает: привлечь замену или объединить с другой комнатой.
- Что если эксперт подтвердил одну комнату, а потом хочет сменить? — Эксперт может запросить смену до дедлайна. Организатор видит изменение в дашборде.
- Что если один эксперт нужен в двух комнатах одновременно? — Эксперт назначается в одну основную комнату. Переход между комнатами — вне скоупа MVP.
- Что если Telegram-контакт эксперта указан некорректно? — Ошибка доставки логируется. Эксперт попадает в список "без контакта".

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST предзагружать экспертов из seed-файла (`data/expert-mapping.json`) при первом запуске, если данных нет. Организатор MUST иметь возможность загрузить новый файл (CSV/JSON) через бот для замены seed-данных. Список комнат с тематиками берётся из утверждённой кластеризации (EPIC-002).
- **FR-002**: Система MUST предлагать распределение экспертов по комнатам на основе совпадения тегов экспертов с тематиками комнат.
- **FR-003**: При матчинге система MUST учитывать приоритет (вес) тегов эксперта, а не только количество совпадений. При отсутствии приоритетов — использовать редкость тега (меньше экспертов с тегом = выше приоритет).
- **FR-004**: Система MUST выделять комнаты с недостаточным покрытием и предлагать экспертов со смежными тегами. Смежность определяется через LLM-промпт: система передаёт список тегов и просит определить семантически близкие в контексте AI/ML проектов.
- **FR-005**: Организатор MUST иметь возможность скорректировать предложенное распределение вручную (перенести эксперта между комнатами).
- **FR-006**: После утверждения распределения организатор получает ссылку-приглашение для общего чата экспертов. Когда эксперт приходит к боту через `/start`, система MUST распознать его по Telegram-username из seed-данных и показать персональное приглашение с назначенной комнатой, тематикой и количеством проектов.
- **FR-007**: Приглашение MUST содержать кнопки: "Иду", "Хочу другую комнату", "Не смогу".
- **FR-008**: При выборе "Хочу другую комнату" система MUST показать список альтернативных комнат с тематиками.
- **FR-009**: Организатор MUST видеть дашборд покрытия: по каждой комнате — количество подтверждённых, отказавшихся, неответивших экспертов с индикацией статуса (покрыто / частично / не покрыто).
- **FR-010**: Система MUST формировать список экспертов без Telegram-контакта и показывать его организатору.
- **FR-011**: Система MUST отправить напоминание эксперту (который уже начал диалог с ботом), если он не ответил на приглашение в течение 3 дней. Экспертам, не начавшим диалог — только через список организатору.
- **FR-012**: Система MUST эскалировать организатору список неответивших экспертов (включая тех, кто не пришёл к боту), если они не ответили в течение 5 дней (или за 2 дня до DD — что наступит раньше).
- **FR-013**: Частота сообщений эксперту MUST быть ограничена: не более 4 сообщений за весь цикл DD.
- **FR-014**: Эксперт MUST иметь возможность сменить комнату после подтверждения (до дедлайна).

### Key Entities

- **Expert Profile**: Профиль эксперта — теги интересов с весами/приоритетами, Telegram-контакт, статус DD (подтверждён / отказ / не ответил / no-show), назначенная комната.
- **Room (Section)**: Комната/секция DD — тематика, теги, количество проектов, требуемое число экспертов, текущее покрытие.
- **Expert-Room Assignment**: Связь эксперта с комнатой — предложена системой или назначена организатором вручную, степень совпадения тегов, статус подтверждения.
- **Escalation**: Запись об эскалации — неответивший эксперт, дата эскалации, комната без покрытия, статус (открыта / решена).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 80%+ приглашённых экспертов с Telegram-контактом дают ответ (подтверждение или отказ) за 5+ дней до DD. Текущее состояние: подтверждения собираются вручную, ~20% no-show.
- **SC-002**: 100% комнат DD покрыты минимум одним подтверждённым экспертом за 3 дня до DD. Текущее состояние: организатор узнаёт о непокрытых комнатах в последний момент.
- **SC-003**: 70%+ экспертов считают назначенную комнату релевантной их компетенциям (post-DD опрос). Текущее состояние: ручное назначение, "зачем я вообще пришёл, потратил своё время?"
- **SC-004**: Организатор видит полную картину покрытия в одном месте, а не в нескольких таблицах и чатах.
- **SC-005**: No-show экспертов снижается с ~20% до менее 10%.

## Clarifications

### Session 2026-02-02

- Q: Как система определяет "смежные теги" для матчинга? → A: Смысловая близость через LLM с конкретным промптом (не эмбеддинги, не ручные группы). Система отправляет список тегов в LLM и просит определить семантически близкие к целевому тегу в контексте AI/ML проектов.
- Q: Минимальное число экспертов на комнату для дашборда покрытия? → A: Фиксированное: 2 эксперта на любую комнату. Покрыто = 2+, частично = 1, не покрыто = 0.
- Q: Откуда берутся данные экспертов для матчинга? → A: Seed из `data/expert-mapping.json` при старте (аналогично EPIC-002 seed) + возможность загрузить новый файл (CSV/JSON) через бот.
- Q: Как организатор запускает рассылку приглашений? → A: Двухэтапно: после утверждения распределения организатор видит превью рассылки (кол-во экспертов, пример сообщения), затем подтверждает отправку.
- Q: Как матчинг связан с кластеризацией EPIC-002? → A: Матчинг требует утверждённую (approved) кластеризацию. Если нет утверждённого расписания — система блокирует и предлагает сначала утвердить расписание.
- Q: Как бот доставляет приглашения экспертам, которые ещё не начали диалог? → A: Организатор отправляет ссылку-приглашение в общий чат экспертов. Эксперты сами приходят к боту через `/start`, бот распознаёт их по Telegram-username из seed-данных и показывает персональное приглашение с назначенной комнатой.

## Assumptions

- Список экспертов (294 записи) с тегами (31 тег) предзагружается из seed-файла `data/expert-mapping.json` при старте. Организатор может заменить данные, загрузив новый CSV/JSON файл. Теги нормализованы (синонимы объединены).
- Список комнат с тематиками берётся из утверждённой кластеризации EPIC-002 (status=approved). Матчинг невозможен без утверждённой кластеризации.
- Каждой комнате нужно минимум 2 эксперта. Покрытие в дашборде: 2+ подтверждённых = покрыто (зелёный), 1 = частично (жёлтый), 0 = не покрыто (красный). При 10 комнатах — требуется 20+ подтверждённых экспертов из 76 приглашённых.
- Эксперт назначается в одну основную комнату (без переходов между комнатами в MVP).
- Эксперт взаимодействует через Telegram-бота. Нет отдельного веб-интерфейса для экспертов.
- Организатор имеет финальное право корректировки любого назначения.
- Сроки эскалации (3 дня — напоминание, 5 дней — алерт) адаптируются в зависимости от оставшегося времени до DD.

## Dependencies

- **EPIC-001 (Онбординг)**: Эксперт должен быть зарегистрирован в системе с ролью "эксперт" и заполненными тегами интересов.
- **EPIC-002 (Кластеризация проектов)**: Матчинг экспертов требует утверждённую (approved) кластеризацию. Без неё матчинг заблокирован — система предлагает организатору сначала утвердить расписание.
- **Данные экспертов**: Список из `data/expert-mapping.json` (294 эксперта, Telegram-контакты, теги).
