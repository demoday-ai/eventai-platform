# Feature Specification: Import Data Tabs

**Feature Branch**: `023-import-tabs`
**Created**: 2026-02-09
**Status**: Draft
**Input**: EPIC-003 — Табификация страницы Import с табами Событие/Проекты/Эксперты/Гости, форма создания/редактирования мероприятия, подсказки о порядке загрузки данных.

## User Scenarios & Testing

### User Story 1 - Табы на странице импорта (Priority: P1)

Как организатор, я хочу видеть страницу импорта данных, разделённую на логические табы (Событие, Проекты, Эксперты, Гости), чтобы быстро находить нужную секцию и понимать структуру данных.

**Why this priority**: Табы — основа UX-реструктуризации страницы. Без них остальные улучшения невозможны.

**Independent Test**: Открыть страницу /import — должны отображаться 4 таба с нумерацией, переключение между ними работает, содержимое каждого таба соответствует его названию.

**Acceptance Scenarios**:

1. **Given** организатор открывает страницу «Импорт данных», **When** страница загрузилась, **Then** отображаются 4 таба: «1. Событие», «2. Проекты», «3. Эксперты», «4. Гости»
2. **Given** организатор на табе «Проекты», **When** кликает на таб «Эксперты», **Then** отображается содержимое таба «Эксперты», таб «Эксперты» визуально выделен как активный
3. **Given** организатор загрузил проекты на табе «Проекты», **When** переключается на другой таб и возвращается, **Then** результат загрузки (количество проектов, статус) сохраняется

---

### User Story 2 - Создание и редактирование мероприятия (Priority: P1)

Как организатор, я хочу создать или отредактировать мероприятие прямо на странице импорта (таб «Событие»), чтобы начать подготовку без перехода в настройки.

**Why this priority**: Создание события — первый шаг пайплайна. Без события невозможны остальные действия.

**Independent Test**: На табе «Событие» создать новое мероприятие с названием, датами начала/окончания — оно должно появиться на Dashboard.

**Acceptance Scenarios**:

1. **Given** нет активного мероприятия, **When** организатор заполняет форму (название, дата начала, дата окончания) и нажимает «Создать», **Then** мероприятие создаётся, отображается подтверждение, Dashboard показывает новое событие
2. **Given** мероприятие уже существует, **When** организатор открывает таб «Событие», **Then** форма заполнена текущими данными (название, даты, описание) и доступна для редактирования
3. **Given** организатор редактирует дату окончания, **When** нажимает «Сохранить», **Then** изменения сохраняются, Dashboard обновляет обратный отсчёт
4. **Given** организатор вводит дату окончания раньше даты начала, **When** нажимает «Сохранить», **Then** отображается ошибка валидации, сохранение не происходит

---

### User Story 3 - Подсказки о порядке загрузки данных (Priority: P2)

Как организатор, я хочу видеть подсказки о рекомендуемом порядке загрузки данных, чтобы не ошибиться с последовательностью и понимать зависимости между данными.

**Why this priority**: Подсказки снижают когнитивную нагрузку и предотвращают ошибки порядка загрузки.

**Independent Test**: Открыть страницу импорта без данных — на каждом табе отображается подсказка, что нужно сделать до этого шага.

**Acceptance Scenarios**:

1. **Given** событие не создано, **When** организатор открывает таб «Проекты», **Then** отображается подсказка: «Сначала создайте мероприятие на вкладке Событие» с ссылкой на таб «Событие»
2. **Given** событие создано, но проекты не загружены, **When** организатор открывает таб «Эксперты», **Then** загрузка экспертов доступна (эксперты не зависят от проектов), подсказка не отображается
3. **Given** все данные загружены, **When** организатор открывает любой таб, **Then** подсказки о зависимостях не отображаются, загрузка доступна
4. **Given** табы пронумерованы (1-4), **When** организатор видит страницу, **Then** нумерация подсказывает рекомендуемый порядок действий

---

### Edge Cases

- Что происходит при ошибке сети во время создания мероприятия? — Отображается сообщение об ошибке, данные формы сохраняются для повторной попытки.
- Что если организатор загружает проекты до создания события? — Загрузка проектов автоматически создаёт событие (существующее поведение сохраняется).
- Что если организатор переключает табы во время загрузки файла? — Загрузка продолжается в фоне, результат отображается при возврате на таб.
- Что если два организатора одновременно редактируют событие? — Последнее сохранение побеждает (стандартное поведение).

## Requirements

### Functional Requirements

- **FR-001**: Страница импорта ДОЛЖНА отображать 4 таба: «Событие», «Проекты», «Эксперты», «Гости»
- **FR-002**: Каждый таб ДОЛЖЕН быть пронумерован (1-4) для подсказки рекомендуемого порядка
- **FR-003**: Переключение между табами ДОЛЖНО сохранять состояние каждого таба (результаты загрузки, введённые данные формы)
- **FR-004**: Таб «Событие» ДОЛЖЕН содержать форму с полями: название, дата начала, дата окончания, описание
- **FR-005**: Если мероприятие не создано, форма ДОЛЖНА работать в режиме создания с кнопкой «Создать»
- **FR-006**: Если мероприятие существует, форма ДОЛЖНА быть предзаполнена текущими данными с кнопкой «Сохранить»
- **FR-007**: Система ДОЛЖНА валидировать, что дата окончания не раньше даты начала
- **FR-008**: Табы «Проекты», «Эксперты», «Гости» ДОЛЖНЫ содержать существующую функциональность загрузки файлов без изменений
- **FR-009**: При отсутствии события табы загрузки ДОЛЖНЫ показывать подсказку с рекомендацией сначала создать мероприятие
- **FR-010**: После успешного создания/редактирования мероприятия данные Dashboard ДОЛЖНЫ обновляться автоматически
- **FR-011**: Таб «Событие» ДОЛЖЕН быть активным по умолчанию при первом открытии страницы

### Key Entities

- **Мероприятие (Event)**: Центральная сущность — название, дата начала, дата окончания, описание. Одно активное мероприятие в системе.
- **Проект**: Привязан к мероприятию. Название, описание, автор, теги.
- **Эксперт**: Привязан к мероприятию. Имя, Telegram, компетенции.
- **Гость**: Привязан к мероприятию. Имя, Telegram, тип (инвестор, партнёр, ментор и др.).

## Success Criteria

### Measurable Outcomes

- **SC-001**: Организатор может создать мероприятие и загрузить все данные (проекты, эксперты, гости) на одной странице за один визит без перехода в настройки
- **SC-002**: Все 4 таба доступны и переключаемы, содержимое каждого таба отображается корректно
- **SC-003**: Организатор-новичок понимает рекомендуемый порядок действий по нумерации табов без дополнительных инструкций
- **SC-004**: Существующая функциональность загрузки файлов (проекты, эксперты, гости) работает без регрессий после реструктуризации
- **SC-005**: Форма создания/редактирования мероприятия валидирует данные и отображает ошибки до отправки

## Assumptions

- Событие создаётся также через Settings — форма на импорте дублирует эту возможность для удобства, не заменяет её
- Эксперты не зависят от проектов (могут быть загружены в любом порядке)
- Загрузка гостей не зависит от проектов или экспертов
- Нумерация табов (1-4) — рекомендация, не строгое ограничение
