## Overview

Доработки и полировка функциональности напоминаний Demo Day (EPIC-007). Основной функционал реализован, требуется:
1. Защита от превышения лимита сообщений Telegram (4096 символов)
2. Восстановление после прерывания рассылки
3. End-to-end валидация всех сценариев

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Безопасная отправка длинных сообщений (Priority: P1)

Организатор отправляет напоминания гостям с персональной программой. Если список проектов слишком длинный, система должна корректно обрезать сообщение, сохранив читаемость.

**Why this priority**: Без этого длинные сообщения вызовут ошибку Telegram API и рассылка не дойдёт до получателей.

**Independent Test**: Создать гостя с программой из 20+ проектов, отправить напоминание — сообщение должно быть доставлено и читаемо.

**Acceptance Scenarios**:

1. **Given** гость с программой из 5 проектов, **When** отправляется напоминание, **Then** сообщение содержит все 5 проектов
2. **Given** гость с программой из 25 проектов, **When** отправляется напоминание, **Then** сообщение содержит топ-N проектов и текст "...и ещё X проектов"
3. **Given** сообщение превышает 4000 символов, **When** система форматирует сообщение, **Then** итоговый текст не превышает 4096 символов

---

### User Story 2 - Восстановление прерванной рассылки (Priority: P2)

Организатор начал рассылку, но она прервалась (перезапуск бота, сетевой сбой). При следующем запуске `/remind` система предлагает возобновить прерванную рассылку.

**Why this priority**: Предотвращает дублирование сообщений и потерю прогресса при технических сбоях.

**Independent Test**: Прервать рассылку на середине, перезапустить бот, выполнить `/remind` — получить предложение возобновить.

**Acceptance Scenarios**:

1. **Given** рассылка со статусом `in_progress`, **When** организатор вызывает `/remind`, **Then** система показывает информацию о прерванной рассылке и кнопки "Возобновить" / "Начать заново"
2. **Given** организатор выбирает "Возобновить", **When** система продолжает рассылку, **Then** отправляются только сообщения получателям без статуса `sent`
3. **Given** организатор выбирает "Начать заново", **When** система отменяет старую рассылку, **Then** статус старой рассылки меняется на `cancelled` и начинается новая

---

### User Story 3 - E2E валидация сценариев (Priority: P3)

Проверка всех 10 сценариев из quickstart.md для подтверждения корректной работы системы.

**Why this priority**: Финальная валидация перед релизом.

**Independent Test**: Последовательное выполнение сценариев 1-10 с проверкой ожидаемых результатов.

**Acceptance Scenarios**:

1. **Given** тестовые данные загружены, **When** выполняются все 10 сценариев, **Then** каждый сценарий завершается успешно
2. **Given** любой сценарий, **When** проверяется результат, **Then** результат соответствует описанию в quickstart.md

---

### Edge Cases

- **Пустое сообщение**: Система не отправляет пустые сообщения, получатель пропускается со статусом `skipped`
- **Заблокировавший бота**: Ошибка отправки логируется, получатель помечается как `failed`
- **Все получатели обработаны**: Batch автоматически переводится в статус `completed`
- **Несколько прерванных batch**: Показывается только самая последняя (newest) прерванная рассылка; старые остаются в статусе `in_progress` до ручной отмены через REST API

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система ДОЛЖНА проверять длину каждого сообщения перед отправкой
- **FR-002**: Система ДОЛЖНА обрезать список проектов в сообщении, если итоговая длина превышает 4000 символов
- **FR-003**: При обрезке система ДОЛЖНА добавлять текст "...и ещё N проектов" с указанием количества пропущенных
- **FR-004**: Система ДОЛЖНА обнаруживать рассылки со статусом `in_progress` при запуске `/remind`
- **FR-005**: Система ДОЛЖНА предлагать организатору выбор: возобновить или начать заново
- **FR-006**: При возобновлении система ДОЛЖНА пропускать получателей со статусом `sent` в notifications
- **FR-007**: При отмене система ДОЛЖНА установить статус прерванной рассылки в `cancelled`
- **FR-008**: Система ДОЛЖНА логировать все операции обрезки сообщений и возобновления рассылок
- **FR-009**: При наличии нескольких прерванных рассылок система ДОЛЖНА показывать только самую последнюю по времени

### Key Entities

- **ReminderBatch**: Добавляется обработка статуса `in_progress` для детекции прерванных рассылок
- **Notification**: Используется статус `sent` для определения уже обработанных получателей

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% сообщений доставляется без ошибок Telegram API (включая длинные программы)
- **SC-002**: Прерванные рассылки обнаруживаются в течение 1 секунды после вызова `/remind`
- **SC-003**: При возобновлении рассылки ни один получатель не получает дублирующее сообщение
- **SC-004**: Все 10 сценариев quickstart.md проходят успешно

## Assumptions

- Лимит Telegram API составляет 4096 символов (документировано)
- Прерванные рассылки возникают редко (аварийные ситуации)
- Организатор принимает решение о возобновлении/отмене вручную
- Топ-проекты для гостей уже отсортированы по релевантности

## Out of Scope

- Автоматическое возобновление рассылок без участия организатора
- Разбиение длинных сообщений на несколько частей
- Изменение структуры сообщений (только обрезка списка проектов)
