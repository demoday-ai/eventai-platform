# Specification: Contact Requests (EPIC-010)

**Version:** 1.0
**Date:** 2026-02-03
**Dependencies:** EPIC-005/006 (Profiling), EPIC-003 (Student confirmation)

---

## Problem Statement

Бизнес-партнёры, гости и эксперты хотят продолжить общение с авторами понравившихся проектов после Demo Day. Однако контактные данные студентов не должны передаваться без их согласия (152-ФЗ).

**Примечание по CustDev:** Функция полноценного планирования встреч 1:1 получила низкую оценку (1/5, "слишком как-то круто"). Для MVP реализуем упрощённый вариант: запрос контакта с согласием студента.

---

## User Stories

### US-020: Запрос контакта (все роли кроме организатора)
**Как гость/бизнес-партнёр/эксперт**, я хочу запросить контакт автора понравившегося проекта, чтобы продолжить общение после Demo Day.

**Acceptance Criteria:**
- Кнопка "Связаться с автором" у каждого проекта в программе/подборке
- Запрос отправляется студенту через бота
- Студент видит информацию о запрашивающем (имя, роль)
- Контакт передаётся только с явного согласия студента

### US-021: Согласие на передачу контакта (студент)
**Как студент**, я хочу решить, делиться ли контактом с запрашивающим, чтобы контролировать свои данные.

**Acceptance Criteria:**
- Студент получает уведомление: "[Имя] ([роль]) хочет связаться по проекту [название]"
- Кнопки: "Разрешаю" / "Не сейчас"
- При согласии — обмен контактами через бота (Telegram username)
- При отказе — вежливое уведомление запрашивающему

---

## Functional Requirements

### FR-001: Кнопка запроса контакта
- Доступна для ролей: Гость, Бизнес/партнёр, Эксперт
- Не доступна для: Организатор, сам студент
- Отображается в карточке проекта и в рекомендациях

### FR-002: Создание запроса
- Сохраняется в БД: requester_id, project_id, student_id, status
- Статусы: pending, approved, rejected, expired
- Отправляется уведомление студенту

### FR-003: Обработка согласия
- Студент получает inline-кнопки
- При "Разрешаю":
  - Статус → approved
  - Обоим отправляется контакт друг друга
- При "Не сейчас":
  - Статус → rejected
  - Запрашивающему: "Автор пока не готов к общению"

### FR-004: Защита от спама
- Один запрос от пользователя к одному проекту (не дублировать)
- Показывать статус существующего запроса вместо повторной кнопки
- Expire pending requests after 7 days

## Data Model

### ContactRequest (новая сущность)
| Field | Type | Description |
|-------|------|-------------|
| id | UUID | PK |
| requester_id | UUID | FK → users (кто запрашивает) |
| project_id | UUID | FK → projects |
| student_user_id | UUID | FK → users (автор проекта) |
| status | Enum | pending, approved, rejected, expired |
| requester_message | String | Опциональное сообщение |
| created_at | DateTime | Время создания |
| responded_at | DateTime | Время ответа |

## User Scenarios

### Scenario 1: Бизнес-партнёр запрашивает контакт
1. Партнёр смотрит проект в своей подборке
2. Нажимает "Связаться с автором"
3. Бот подтверждает: "Запрос отправлен. Ожидаем ответа автора."
4. Студент получает уведомление
5. При согласии — обмен контактами

### Scenario 2: Студент обрабатывает запрос
1. Студент получает: "Инвестор [имя] хочет связаться по проекту [название]"
2. Нажимает "Разрешаю"
3. Бот отправляет студенту: "Контакт передан. Telegram партнёра: @username"
4. Бот отправляет партнёру: "Автор согласен. Telegram: @student"

### Scenario 3: Студент отказывает
1. Студент получает запрос
2. Нажимает "Не сейчас"
3. Партнёр получает: "Автор пока не готов к общению. Попробуйте позже."

---

## Success Criteria

- 80% запросов обрабатываются студентами в течение 24 часов
- Студенты контролируют передачу своих данных (100% с согласия)
- Запрос контакта занимает <30 секунд
- Обмен контактами происходит мгновенно после согласия

---

## Out of Scope

- Планирование конкретных временных слотов встреч
- Календарная интеграция
- Видеозвонки через бота
- Групповые встречи

---

## Assumptions

- У студентов есть публичный Telegram username
- Студенты активны в боте и ответят на запрос
- Данные о проектах связаны с telegram_contact из формы рубежа
