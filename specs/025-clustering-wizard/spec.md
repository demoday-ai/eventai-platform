# Feature Specification: Clustering Wizard

**Feature Branch**: `025-clustering-wizard`
**Created**: 2026-02-09
**Status**: Draft
**Input**: EPIC-005 Кластеризация: Wizard-flow для AI-кластеризации проектов по залам

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Запуск AI-кластеризации (Priority: P1)

Организатор открывает страницу Кластеризации, задаёт количество залов и опциональные критерии (фидбэк для LLM, названия тем залов), нажимает «Запустить». Система запускает фоновую задачу кластеризации и показывает прогресс. По завершении — показывает результат: залы с распределёнными проектами.

**Why this priority**: Без запуска кластеризации невозможно перейти к матчингу экспертов и расписанию — это фундаментальный шаг пайплайна.

**Independent Test**: Организатор задаёт 6 залов, нажимает «Запустить», ждёт результата — видит 6 залов с проектами внутри.

**Acceptance Scenarios**:

1. **Given** проекты загружены (≥1), **When** организатор задаёт количество залов и нажимает «Запустить», **Then** система запускает кластеризацию и показывает прогресс выполнения.
2. **Given** кластеризация завершена, **When** организатор смотрит результат, **Then** видит список залов с названиями, описаниями и количеством проектов в каждом.
3. **Given** проекты не загружены, **When** организатор открывает страницу, **Then** видит empty state с подсказкой «Загрузите проекты» и ссылкой на Импорт.

---

### User Story 2 — Просмотр и ручная корректировка (Priority: P1)

Организатор просматривает результат кластеризации, видит залы с проектами. Если какой-то проект оказался не в том зале — организатор перемещает его в другой зал. Статистика обновляется.

**Why this priority**: Корректировка AI-решений критична для качества — организатор знает контекст лучше LLM.

**Independent Test**: Организатор видит проект в зале А, перемещает в зал Б — проект исчезает из А, появляется в Б, счётчики обновляются.

**Acceptance Scenarios**:

1. **Given** результат кластеризации отображается, **When** организатор раскрывает зал, **Then** видит список проектов с названием, автором, тегами.
2. **Given** проект в зале А, **When** организатор выбирает «Переместить» и целевой зал Б, **Then** проект перемещается, счётчики обоих залов обновляются.
3. **Given** результат кластеризации, **When** организатор смотрит на зал, **Then** видит название зала, описание темы и количество проектов.

---

### User Story 3 — Одобрение кластеризации (Priority: P1)

Организатор удовлетворён распределением и нажимает «Одобрить». Кластеризация фиксируется. Global Stepper обновляется. Появляется подсказка о следующем шаге — матчинге экспертов.

**Why this priority**: Одобрение — финальная точка эпика, без неё пайплайн не продвигается дальше.

**Independent Test**: Организатор нажимает «Одобрить» — видит подтверждение, Global Stepper обновляется, появляется ссылка на следующий шаг.

**Acceptance Scenarios**:

1. **Given** результат кластеризации просмотрен, **When** организатор нажимает «Одобрить», **Then** система просит подтверждение.
2. **Given** подтверждение дано, **When** кластеризация одобрена, **Then** Global Stepper обновляет статус шага «Кластеризация» на «завершён».
3. **Given** кластеризация одобрена, **When** организатор видит результат, **Then** отображается подсказка «Следующий шаг — распределение экспертов» со ссылкой.

---

### Edge Cases

- Что если организатор запускает кластеризацию повторно (уже есть результат)? — Показать предупреждение, что предыдущий результат будет заменён.
- Что если кластеризация не завершилась (ошибка LLM)? — Показать сообщение об ошибке с кнопкой «Повторить».
- Что если все проекты попали в один зал? — Показать результат как есть, организатор может перераспределить вручную.
- Что если кластеризация уже одобрена и организатор хочет перезапустить? — Разрешить перезапуск, старое одобрение сбрасывается.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST отображать wizard-flow для кластеризации: Параметры → Результат → Одобрение.
- **FR-002**: Организатор MUST иметь возможность задать количество залов (числовое поле, ≥2).
- **FR-003**: Организатор SHOULD иметь возможность задать текстовый фидбэк/критерии для LLM и названия тем залов.
- **FR-004**: Система MUST запускать кластеризацию одной кнопкой и показывать статус выполнения (polling).
- **FR-005**: Система MUST отображать результат: залы с названиями, описаниями тем и списками проектов.
- **FR-006**: Организатор MUST иметь возможность переместить проект из одного зала в другой.
- **FR-007**: После перемещения статистика (количество проектов в залах) MUST обновляться.
- **FR-008**: Система MUST предоставить кнопку «Одобрить» с диалогом подтверждения.
- **FR-009**: После одобрения Global Stepper MUST обновить статус шага кластеризации.
- **FR-010**: После одобрения MUST отображаться подсказка «Следующий шаг — распределение экспертов» со ссылкой.
- **FR-011**: Если проекты не загружены, страница MUST показывать empty state с ссылкой на Импорт.

### Key Entities

- **ClusteringRun**: Запуск кластеризации — содержит параметры (количество залов, фидбэк), статус, результат.
- **Room**: Зал — название, описание темы, список проектов.
- **Project**: Проект — название, автор, теги, привязка к залу.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Организатор может запустить кластеризацию и увидеть результат за один визит страницы (без перезагрузки).
- **SC-002**: Перемещение проекта между залами отражается мгновенно (обновление счётчиков без задержки).
- **SC-003**: Одобрение кластеризации обновляет Global Stepper и показывает следующий шаг.
- **SC-004**: Все 3 user stories проходят acceptance scenarios без ошибок.

## Assumptions

- Backend API для кластеризации уже реализован (POST /clustering/run, GET /clustering/current, POST /clustering/{id}/move, POST /clustering/{id}/approve).
- Polling для статуса job уже реализован (POST /clustering/run возвращает job_id, GET /clustering/job/{id}).
- Страница Clustering.tsx уже частично реализована — требуется проверка соответствия USM и доработка при необходимости.
