# Feature Specification: Загрузка и AI-кластеризация проектов

**Feature Branch**: `002-project-clustering`
**Created**: 2026-02-02
**Status**: Draft
**Input**: EPIC-002 из User Story Map v2.2
**RICE**: H1, Score 335 (Reach 335, Impact 3, Confidence 100%, Effort 3 нед)
**Pain Map**: Организатор Pain 1 — «Хаотичное расписание» (Priority Score 50, highest)

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Загрузка данных о проектах (Priority: P1)

Организатор загружает в бот файл с данными о проектах (название, описание, теги, автор, Telegram-контакт). Бот валидирует данные и сообщает итог: сколько проектов загружено, у скольких есть проблемы (нет описания, нет тегов).

**Why this priority**: Без данных о проектах невозможна кластеризация и всё дальнейшее — расписание, рассылка слотов, подборки для гостей.

**Independent Test**: Организатор отправляет файл в чат → бот парсит и подтверждает загрузку → данные доступны для просмотра.

**Acceptance Scenarios**:

1. **Given** организатор авторизован с ролью «организатор», **When** он отправляет файл (CSV или JSON) с данными о проектах, **Then** бот парсит файл и отвечает: «Загружено N проектов. M проектов без описания — проверьте».
2. **Given** файл содержит дубликаты (одинаковое название проекта), **When** бот обрабатывает файл, **Then** бот сообщает: «Найдено K дубликатов» и перечисляет их. Дубликаты не загружаются.
3. **Given** организатор отправляет файл невалидного формата (например, PDF), **When** бот получает файл, **Then** бот отвечает: «Поддерживаемые форматы: CSV, JSON. Пожалуйста, отправьте файл в одном из них».
4. **Given** проекты уже загружены ранее, **When** организатор загружает новый файл, **Then** бот предлагает: «Заменить предыдущие данные (N проектов) новыми (M проектов)?» с кнопками Да / Нет.

---

### User Story 2 — AI-кластеризация проектов по залам (Priority: P1)

После загрузки проектов организатор запускает кластеризацию. Система автоматически распределяет проекты по тематическим залам на основе описаний и тегов. Результат — список залов с проектами и обоснованием тематики каждого зала.

**Why this priority**: Это core-фича, решающая главную боль организатора (Pain #1, score 50). Заменяет ручной процесс с ChatGPT, который «кластеризует фигово» и допускает дубликаты.

**Independent Test**: Организатор нажимает «Запустить кластеризацию» → получает распределение проектов по залам → может оценить результат.

**Acceptance Scenarios**:

1. **Given** проекты загружены (N ≥ 10), **When** организатор нажимает «Запустить кластеризацию», **Then** бот показывает параметры: «Кластеризовать N проектов на K залов?» с возможностью изменить количество залов.
2. **Given** кластеризация запущена, **When** процесс завершён, **Then** бот показывает результат: для каждого зала — название тематики, список проектов, количество проектов в зале.
3. **Given** результат кластеризации готов, **When** организатор просматривает его, **Then** разница между самым большим и самым маленьким залом составляет не более 5 проектов (балансировка).
4. **Given** результат кластеризации готов, **When** организатор просматривает его, **Then** каждый проект присутствует ровно в одном зале (0 дубликатов).
5. **Given** результат кластеризации готов, **When** организатор просматривает зал, **Then** для каждого зала указано обоснование тематики (например: «NLP + Agents: 8 проектов связаны с обработкой языка и автономными агентами»).

---

### User Story 3 — Просмотр и корректировка кластеризации (Priority: P2)

Организатор просматривает результат кластеризации и может вручную перенести проект из одного зала в другой или запросить повторную кластеризацию с другими параметрами. После корректировки — утверждает расписание.

**Why this priority**: AI предлагает, человек решает. Без возможности корректировки организатор не доверит автоматике.

**Independent Test**: Организатор видит кластеризацию → переносит проект → утверждает финальное расписание.

**Acceptance Scenarios**:

1. **Given** результат кластеризации показан, **When** организатор выбирает проект и нажимает «Перенести», **Then** бот предлагает список доступных залов и организатор может переместить проект.
2. **Given** организатор хочет перегенерировать результат, **When** он нажимает «Перегенерировать» и вводит обратную связь на естественном языке (например: «разделите NLP и агентов»), **Then** бот запускает новую кластеризацию с учётом фидбэка (ручные правки сбрасываются, бот предупреждает об этом). Фидбэк опционален — можно перегенерировать без комментария.
3. **Given** организатор доволен результатом, **When** он нажимает «Утвердить расписание», **Then** расписание фиксируется и становится доступно для рассылки (EPIC-003) и других функций бота.
4. **Given** расписание уже утверждено, **When** организатор хочет внести изменения, **Then** бот предупреждает: «Расписание уже утверждено. Изменения затронут рассылку. Продолжить?» с кнопками Да / Нет.

### User Story 4 — Предзагруженные данные для демо (Priority: P1)

Для демонстрации 6 февраля система содержит предзагруженные данные о проектах прошлого Demo Day (~305 проектов из checkpoint-форм с описаниями и контактами). Организатор может сразу запустить кластеризацию без ручной загрузки.

**Why this priority**: Дедлайн демо — 6 февраля. Без предзагруженных данных демонстрация невозможна.

**Independent Test**: Организатор открывает бот → видит, что данные уже загружены → запускает кластеризацию.

**Acceptance Scenarios**:

1. **Given** организатор входит в бот впервые, **When** он переходит в раздел «Проекты», **Then** бот показывает: «Загружено 305 проектов (демо-данные). Запустить кластеризацию?»
2. **Given** демо-данные загружены, **When** организатор загружает собственный файл, **Then** демо-данные заменяются на загруженные.

---

### Edge Cases

- Что если загружен файл с 0 проектов (пустой)? → Бот сообщает: «Файл пуст. Загрузите файл с данными о проектах».
- Что если загружен файл с менее чем 10 проектами? → Кластеризация возможна, но бот предупреждает: «Мало проектов для кластеризации. Рекомендуем не менее 10».
- Что если в файле отсутствует обязательное поле? → Бот сообщает: «Строка N: отсутствует [название поля]. Исправьте и загрузите заново». Обязательные поля: название, описание, теги, автор, Telegram-контакт.
- Что если кластеризация занимает больше 30 секунд? → Бот показывает индикатор прогресса: «Кластеризация... это займёт до минуты».
- Что если организатор запускает кластеризацию повторно при уже утверждённом расписании? → Бот предупреждает о последствиях (сброс утверждения).
- Что если количество залов больше количества проектов? → Бот предупреждает: «Слишком много залов для N проектов. Рекомендуем не более M залов».
- Что если у всех проектов одинаковые теги? → Кластеризация использует описания для дифференциации, даже при одинаковых тегах.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система ДОЛЖНА принимать файлы с данными о проектах в форматах CSV и JSON.
- **FR-002**: Система ДОЛЖНА валидировать загруженные данные: наличие всех обязательных полей (название, описание, теги, автор, Telegram-контакт), выявление дубликатов по названию. Строки с пропущенными полями отклоняются с указанием номера строки и отсутствующего поля.
- **FR-003**: Система ДОЛЖНА сообщать организатору итог загрузки: количество загруженных проектов, количество с ошибками, список дубликатов.
- **FR-004**: Система ДОЛЖНА предоставлять AI-кластеризацию проектов по тематическим залам на основе описаний и тегов.
- **FR-005**: Система ДОЛЖНА обеспечивать балансировку залов: разница между самым большим и самым маленьким залом — не более 5 проектов.
- **FR-006**: Система ДОЛЖНА гарантировать отсутствие дубликатов: каждый проект — ровно в одном зале.
- **FR-007**: Система ДОЛЖНА предоставлять обоснование тематики каждого зала (краткое описание + ключевые теги).
- **FR-008**: Организатор ДОЛЖЕН иметь возможность вручную переносить проект из одного зала в другой.
- **FR-009**: Организатор ДОЛЖЕН иметь возможность запустить повторную кластеризацию с указанием обратной связи на естественном языке (например: «разделите NLP и агентов»). Система ДОЛЖНА учитывать этот фидбэк при перегенерации.
- **FR-010**: Организатор ДОЛЖЕН иметь возможность утвердить расписание, после чего оно фиксируется для использования другими функциями.
- **FR-011**: Система ДОЛЖНА предупреждать об изменениях утверждённого расписания.
- **FR-012**: Система ДОЛЖНА содержать предзагруженные демо-данные (~305 проектов прошлого Demo Day из checkpoint-форм) для демонстрации. Источник: `data/test/checkpoint12_anon.xlsx` (название, описание, автор, TG) + `docs/00-research/past-demoday-projects.md` (теги, где есть).
- **FR-013**: Загрузка данных и управление кластеризацией ДОЛЖНЫ быть доступны только пользователям с ролью «организатор».
- **FR-014**: Система ДОЛЖНА вести организатора через пошаговый wizard: загрузка → кластеризация → корректировка → утверждение. После каждого шага бот предлагает следующий. Возврат к предыдущим шагам — через кнопку «Назад».

### Key Entities

- **Проект (Project)**: название, описание, теги (тематики), автор, Telegram-контакт автора, зал (после кластеризации), время слота (после составления расписания).
- **Зал (Room)**: номер/название, тематика (автоматическая), список проектов, количество проектов.
- **Расписание (Schedule)**: привязка проектов к залам, статус (черновик / утверждено), дата утверждения.
- **Кластеризация (Clustering Run)**: параметры (количество залов), результат (проект → зал), обоснование тематик, статус (черновик / утверждён / отменён).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Организатор может загрузить данные о 305 проектах и получить результат кластеризации менее чем за 2 минуты.
- **SC-002**: Кластеризация распределяет проекты по залам с 0 дубликатами (каждый проект ровно в одном зале).
- **SC-003**: Балансировка залов: разница между самым большим и самым маленьким залом не превышает 5 проектов.
- **SC-004**: Не менее 80% проектов попадают в тематически релевантный зал (по оценке организатора).
- **SC-005**: Организатор может утвердить расписание за одну сессию (без необходимости возвращаться позже).
- **SC-006**: Ручная корректировка (перенос проекта) занимает менее 30 секунд на один проект.

## Assumptions

- Для демо 6 февраля используются предзагруженные данные из `data/test/checkpoint12_anon.xlsx` (~305 проектов) + теги из `docs/00-research/past-demoday-projects.md` где есть совпадение.
- Количество залов по умолчанию — 6 (как на прошлом Demo Day), но организатор может изменить.
- Кластеризация выполняется по тематикам (NLP, CV, Agents, FinTech, ...), а не по трекам (EdTech, стартап, индустрия, наука) — треки описывают формат обучения.
- Временные слоты внутри зала (порядок выступлений) в данном EPIC не определяются — это задача составления детального расписания (может быть отдельным EPIC или расширением).
- Загрузка файлов доступна только через Telegram-бот (организатор отправляет файл в личный чат с ботом).
- Один проект может иметь несколько тегов-тематик, но назначается ровно в один зал.

## Dependencies

- **EPIC-001 (Онбординг)**: пользователь должен быть авторизован с ролью «организатор».
- **Данные**: `data/test/checkpoint12_anon.xlsx` (основной источник демо-данных: ~305 проектов) + `docs/00-research/past-demoday-projects.md` (тематические теги).
- **EPIC-003 (Подтверждение участия)**: использует утверждённое расписание для рассылки слотов студентам.

## Clarifications

### Session 2026-02-02

- Q: Какие поля в файле проектов обязательные, а какие опциональные? → A: Все 5 полей обязательны (название, описание, теги, автор, Telegram-контакт). Файл с пропущенным полем отклоняется построчно с указанием ошибки.
- Q: Как организатор оценивает тематическую релевантность кластеризации (SC-004)? → A: Субъективно при просмотре. Если не устраивает — организатор может перегенерировать кластеризацию, указав обратную связь на естественном языке (например: «разделите NLP и агентов», «слишком много проектов в зале 3»). Система учитывает фидбэк при повторной кластеризации.
- Q: Как организатор навигирует по workflow в боте? → A: Пошаговый wizard: загрузка → кластеризация → корректировка → утверждение. Бот ведёт организатора последовательно, предлагая следующий шаг после завершения текущего. Возврат к предыдущим шагам возможен через кнопку «Назад».
- Q: Как формировать демо-данные для seed? → A: Seed из checkpoint-форм (контрольные рубежи): ~305 проектов с полями название, описание (план работы + ожидаемый результат), автор, Telegram-контакт. Тематические теги — из past-demoday-projects.md где есть совпадение по названию, остальные — пустые. Кластеризация по описаниям компенсирует неполные теги.

- Интеграция с формой контрольного рубежа (Release 1.1, SS-015).
