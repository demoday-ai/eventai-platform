# Research: Профилирование и программа для гостей (EPIC-005)

**Date**: 2026-02-02
**Branch**: `005-guest-profiling`

## R1: Алгоритм ранжирования проектов для гостей

**Decision**: Двухступенчатый подход — IDF tag overlap + LLM re-ranking.

**Rationale**:
- Этап 1 (tag overlap): переиспользует IDF-формулу из `matching_service.py`. Быстрый, детерминированный, работает без LLM.
- Этап 2 (LLM re-ranking): берёт top-20 кандидатов из этапа 1 и ранжирует с учётом свободного текста профиля гостя. Позволяет учесть нюансы ("антифрод в финтехе" vs просто "финтех").
- При недоступности LLM → только tag overlap (graceful degradation).

**Alternatives considered**:
- Только tag overlap — не учитывает свободный текст, грубое ранжирование.
- Только LLM — медленнее (330 проектов в один prompt нереально), дороже, нет fallback.
- Embedding similarity — требует дополнительной инфраструктуры (vector DB), overkill для ~330 проектов.

## R2: IDF-формула для гостевого профиля

**Decision**: Адаптировать `compute_idf_weights()` из matching_service, но считать IDF по проектам (не по экспертам).

**Rationale**:
- Формула: `idf(tag) = log(total_projects / projects_with_tag)` — редкий тег получает больший вес.
- Гость с интересом "Security" (мало проектов) получит более точное ранжирование, чем гость с "NLP" (много проектов).
- Score проекта: `sum(idf(tag) * 1.0 для точных совпадений)`. Adjacent tags не используются (в отличие от экспертов) — для гостей важнее точное совпадение.

**Alternatives considered**:
- Равные веса тегов — "NLP" (50+ проектов) забивает сигнал от редких тегов.
- TF-IDF с учётом количества тегов проекта — избыточная сложность для MVP.

## R3: Извлечение интересов из свободного текста

**Decision**: Один LLM-вызов для парсинга текста → структурированный JSON с тематиками, технологиями, отраслями.

**Rationale**:
- Используем existing `llm_client.send_chat_completion()` с json_mode=True.
- System prompt задаёт список допустимых тегов (из БД) + разрешает добавить ключевые слова.
- Результат: `{"tags": ["NLP", "FinTech"], "keywords": ["антифрод", "hiring automation"]}`.
- Keywords сохраняются отдельно для LLM re-ranking (этап 2).

**Alternatives considered**:
- Regex / keyword matching — не понимает синонимы и контекст ("антимошенничество" → "антифрод").
- Embedding + nearest tag — требует vector DB.
- Спрашивать гостя уточняющие вопросы — удлиняет flow (SC-001: ≤2 мин).

## R4: LLM-генерация кратких описаний проектов

**Decision**: Batch-генерация через один LLM-вызов для top-15 проектов.

**Rationale**:
- System prompt: "Ты помогаешь гостю Demo Day. Сгенерируй 2-3 предложения о каждом проекте, адаптированные под интересы гостя."
- User prompt: профиль гостя + JSON массив из {title, description, tags} для 10-15 проектов.
- Один вызов вместо 15 отдельных → укладываемся в 10 секунд (SC-002).
- При недоступности LLM → используем первые 2 предложения из оригинального description.

**Alternatives considered**:
- Показывать оригинальное описание — не адаптировано под гостя, длинное.
- Per-project LLM call — 15 вызовов × 3-5 сек = слишком долго.
- Pre-generate summaries — не учитывает профиль конкретного гостя.

## R5: Telegram UX для мульти-выбора тегов

**Decision**: Inline-кнопки с toggle (✓/пусто) + текстовый ввод в том же состоянии ConversationHandler.

**Rationale**:
- python-telegram-bot поддерживает edit_message_reply_markup для toggle кнопок.
- Callback data: `tag:{tag_name_short}` (≤64 байт). Для длинных тегов — использовать короткий id.
- Гость нажимает кнопки → они toggle. Пишет текст → принимается как свободный ввод.
- Кнопка "Готово" завершает выбор.
- Показывать не все 31 тег — группировать по 3×5 сеткой, показывать top-15 по количеству проектов.

**Alternatives considered**:
- ReplyKeyboardMarkup — не поддерживает toggle, занимает место.
- Несколько шагов (сначала кнопки, потом текст) — удлиняет flow.
- Tinder-стиль (как Demo-Hero) — медленно, не масштабируется на 31 тег.

## R6: Интеграция с онбордингом (auto-trigger)

**Decision**: После `subtype_chosen()` в start.py, если роль == Guest, отправить сообщение с предложением профилирования.

**Rationale**:
- В `subtype_chosen()` уже установлена роль и подтип гостя.
- Добавляем: после "Добро пожаловать!" → через 1 сек send message: "Укажите интересы для персональной программы" + кнопка "Начать".
- Кнопка запускает отдельный ConversationHandler для профилирования.
- Команда /profile также доступна для повторного входа.

**Alternatives considered**:
- Встроить профилирование прямо в onboarding ConversationHandler — усложняет существующий код, нарушает SRP.
- Только по команде /profile — гость может не знать о ней.

## R7: Учёт параллельности залов

**Decision**: Помечать конфликтующие проекты (из разных залов) в подборке, но не исключать.

**Rationale**:
- FR-009: "не включать два конфликтующих по времени проекта без пометки".
- Так как временных слотов нет (Assumptions: EPIC-003/отложено), используем залы как proxy.
- Проекты из одного зала не конфликтуют. Проекты из разных залов помечаются: "⚠ Зал 3 и Зал 5 — параллельно".
- Не исключаем проекты — гость сам решает приоритеты.

**Alternatives considered**:
- Жёсткое ограничение (один проект на зал) — слишком ограничительно, гость может перейти между залами.
- Игнорировать конфликты — не даёт полезной информации гостю.
