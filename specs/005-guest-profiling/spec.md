# Feature Specification: Профилирование и программа для гостей (EPIC-005)

**Feature Branch**: `005-guest-profiling`
**Created**: 2026-02-02
**Status**: Draft
**Input**: User description: "EPIC-005 -- Профилирование и программа для гостей"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Указание интересов гостем (Priority: P1)

Гость приходит в Telegram-бота после онбординга (роль "Гость" + подтип уже выбраны в EPIC-001). Бот предлагает указать интересы двумя способами: через интерактивные кнопки тематик (NLP, CV, Автономные агенты, EdTech, FinTech, Security, MedTech и т.д.) и через свободный текстовый ввод. Гость может выбрать несколько тематик кнопками, а затем уточнить интересы текстом (например, "интересует антифрод в финтехе и hiring automation"). AI анализирует текст и извлекает профиль: тематики, технологии, отрасли. Бот показывает подтверждение: "Я понял, вас интересует: NLP, CV, антифрод. Верно?" Гость подтверждает или корректирует. Профиль сохраняется.

**Why this priority**: Без профиля невозможно сгенерировать персональную программу. Это входная точка для всей персонализации. CustDev подтверждает: "варианты плюс возможность текста, поклацать" (Настя, интервью #4).

**Independent Test**: Гость с ролью "AI-практик" заходит в бота, нажимает кнопки NLP и CV, пишет "интересует антифрод". Бот подтверждает: "NLP, CV, антифрод". Гость нажимает "Да". Профиль сохранён в БД с привязкой к пользователю.

**Acceptance Scenarios**:

1. **Given** гость прошёл онбординг (роль "Гость" + подтип выбраны), **When** онбординг завершён, **Then** бот автоматически предлагает "Укажите интересы для персональной программы" и показывает клавиатуру с тематиками (мульти-выбор) и предложение написать текстом. Команда /profile также доступна для повторного входа.
2. **Given** гость выбрал 2 тематики кнопками и написал свободный текст, **When** бот обрабатывает ввод, **Then** AI извлекает ключевые слова и показывает подтверждение с полным списком интересов.
3. **Given** бот показал "Вас интересует: X, Y, Z. Верно?", **When** гость нажимает "Да", **Then** профиль интересов сохраняется, бот предлагает получить персональную программу.
4. **Given** бот показал подтверждение, **When** гость нажимает "Нет, изменить", **Then** бот возвращается к выбору тематик с предыдущими выборами сохранёнными.
5. **Given** гость вводит только свободный текст без кнопок, **When** бот обрабатывает текст, **Then** AI корректно извлекает тематики из свободного текста и показывает подтверждение.

### User Story 2 - Генерация персональной программы (Priority: P1)

После подтверждения профиля система генерирует персональную программу из 10-15 проектов. Программа ранжирована по релевантности к профилю гостя. Каждый проект содержит: название, AI-сгенерированное краткое описание (2-3 предложения, адаптированное под профиль гостя), теги, автор, зал. Проекты разделены на "обязательно посетить" (top-5 по релевантности) и "если останется время" (остальные). Учитывается параллельность залов -- бот не рекомендует два проекта из разных залов одновременно, если нет физической возможности посетить оба.

**Why this priority**: Это основная ценность фичи. Персональная программа -- главная потребность гостей (5/5 по CustDev). Без неё гость видит <20% проектов и теряет время.

**Independent Test**: Создать гостя с профилем "NLP, антифрод". Запустить генерацию. Убедиться: top-5 содержат NLP/антифрод проекты, программа не превышает 15 проектов, проекты из одного зала не конфликтуют.

**Acceptance Scenarios**:

1. **Given** гость подтвердил профиль интересов, **When** нажимает "Получить программу" или бот предлагает автоматически, **Then** система генерирует программу из 10-15 проектов с ранжированием по релевантности.
2. **Given** программа сгенерирована, **When** бот отправляет её гостю, **Then** каждый проект содержит: название, AI-сгенерированное краткое описание (2-3 предложения, адаптированное под профиль), теги, автор, зал и оценку релевантности.
3. **Given** программа содержит 10+ проектов, **When** бот отображает её, **Then** проекты разделены на "обязательно" (top-5) и "по возможности" (остальные).
4. **Given** у гостя профиль, не совпадающий ни с одним проектом, **When** система генерирует программу, **Then** система предлагает наиболее популярные проекты с пояснением "по общей релевантности".

### User Story 3 - Просмотр деталей проекта из подборки (Priority: P2)

Гость видит в подборке список проектов с краткими описаниями. По каждому проекту можно запросить подробности: полное описание, автор, контакт, зал. Это позволяет гостю подготовиться к визиту и сформулировать вопросы заранее.

**Why this priority**: Расширяет ценность подборки, но не блокирует MVP. Краткие описания уже дают достаточно информации для принятия решения.

**Independent Test**: Гость получил подборку из 12 проектов. Нажимает на проект #3. Бот показывает полное описание, теги, автора, зал.

**Acceptance Scenarios**:

1. **Given** гость получил подборку проектов, **When** нажимает на кнопку конкретного проекта, **Then** бот показывает расширенный бриф: полное описание, все теги, автор, telegram-контакт, зал.
2. **Given** гость просматривает детали проекта, **When** нажимает "Назад", **Then** бот возвращает к списку подборки на той же позиции.

---

### User Story 4 - Обновление профиля и перегенерация (Priority: P3)

Гость может обновить свой профиль интересов в любое время (до и во время мероприятия) и запросить новую подборку. Это полезно, если гость уже посетил часть проектов или изменились интересы.

**Why this priority**: Nice-to-have для повторных визитов. Основной сценарий -- однократное профилирование перед DD.

**Independent Test**: Гость с существующим профилем вводит /profile. Бот показывает текущий профиль и предлагает изменить. Гость меняет. Новая подборка генерируется.

**Acceptance Scenarios**:

1. **Given** гость уже имеет профиль и подборку, **When** вводит /profile, **Then** бот показывает текущие интересы и предлагает обновить.
2. **Given** гость обновил профиль, **When** подтверждает изменения, **Then** система генерирует новую подборку на основе обновлённого профиля.

---

### Edge Cases

- Что, если гость не выбрал ни одной тематики и не ввёл текст? -- Бот напоминает, что нужно указать хотя бы один интерес, не сохраняет пустой профиль.
- Что, если AI не может извлечь осмысленные тематики из свободного текста? -- Бот показывает: "Не удалось определить интересы. Попробуйте выбрать из списка тематик." Предлагает кнопки повторно.
- Что, если проектов для подборки меньше 10? -- Показать все подходящие (сколько есть), дополнить популярными проектами до 10.
- Что, если в БД нет проектов с тегами, совпадающими с профилем? -- Показать top-15 проектов по общему рейтингу с пояснением.
- Что, если гость ещё не прошёл онбординг (нет роли)? -- Перенаправить на /start для выбора роли.
- Что, если гость уже просмотрел подборку и запрашивает повторно? -- Показать ту же подборку (кэшировать до обновления профиля).
- Что, если AI-сервис недоступен (таймаут, ошибка)? -- Graceful degradation: для извлечения текста — использовать только выбранные кнопками теги; для ранжирования — использовать tag overlap без LLM re-ranking. Гость получает результат в любом случае.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система ДОЛЖНА предоставлять гостю гибридный интерфейс профилирования: inline-кнопки с тематиками (мульти-выбор) и поле свободного текста.
- **FR-002**: Система ДОЛЖНА поддерживать мульти-выбор тематик через inline-кнопки (toggle on/off) с визуальным выделением выбранных.
- **FR-003**: Система ДОЛЖНА анализировать свободный текст гостя с помощью AI и извлекать: тематики, технологии, отрасли. При недоступности AI система использует только теги, выбранные кнопками, и предупреждает гостя: "Текст не удалось обработать, используем выбранные тематики."
- **FR-004**: Система ДОЛЖНА показывать подтверждение извлечённых интересов перед сохранением: "Вас интересует: X, Y, Z. Верно?"
- **FR-005**: Система ДОЛЖНА сохранять профиль интересов гостя (выбранные тематики + извлечённые ключевые слова) с привязкой к пользователю и событию.
- **FR-006**: Система ДОЛЖНА генерировать персональную программу из 10-15 проектов, ранжированных по релевантности к профилю гостя. Ранжирование: 1) отбор кандидатов по совпадению тегов (IDF-взвешенный tag overlap), 2) LLM re-ranking top-20 кандидатов с учётом свободного текста профиля гостя.
- **FR-007**: Система ДОЛЖНА разделять программу на "обязательно посетить" (top-5) и "если останется время" (остальные).
- **FR-008**: Система ДОЛЖНА для каждого проекта в подборке отображать: название, AI-сгенерированное краткое описание (2-3 предложения, адаптированное под профиль гостя), теги, автор, зал.
- **FR-009**: Система ДОЛЖНА учитывать параллельность залов при формировании программы -- не включать два конфликтующих по времени проекта без пометки.
- **FR-010**: Система ДОЛЖНА позволять гостю запросить подробности по любому проекту из подборки.
- **FR-011**: Система ДОЛЖНА поддерживать повторное профилирование: гость может обновить интересы и получить новую подборку.
- **FR-012**: Система ДОЛЖНА обрабатывать случай, когда AI не может извлечь интересы из текста, предлагая вернуться к выбору кнопками.
- **FR-013**: Система ДОЛЖНА дополнять подборку популярными проектами, если проектов по профилю меньше 10.
- **FR-014**: Система ДОЛЖНА ограничивать доступ к профилированию только пользователям с ролью "Гость" или "Бизнес/партнёр".

### Key Entities

- **Профиль интересов (GuestProfile)**: Связь с пользователем и событием. Содержит: выбранные тематики (из кнопок), извлечённые ключевые слова (из текста), сырой текст ввода, дату создания/обновления.
- **Рекомендация (Recommendation)**: Связь с профилем и проектом. Содержит: оценку релевантности, категорию ("must_visit" / "if_time"), порядок в подборке.
- **Проект (Project)** (существующий): Название, описание, автор, теги, зал. Уже реализован в EPIC-002.
- **Тег (Tag)** (существующий): Название. Переиспользуется из EPIC-002/004.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Гость завершает профилирование (от начала до подтверждения) за 2 минуты или менее.
- **SC-002**: Персональная программа содержит 10-15 проектов и генерируется за 10 секунд или менее после подтверждения профиля.
- **SC-003**: Не менее 70% проектов в подборке "обязательно посетить" (top-5) содержат теги, совпадающие с профилем гостя.
- **SC-004**: Гость может просмотреть детали любого проекта из подборки в 1 клик.
- **SC-005**: Не менее 80% гостей, начавших профилирование, успешно его завершают (не бросают на середине).
- **SC-006**: Система корректно обрабатывает все 3 подтипа гостей (абитуриент, AI-практик, другое) с одинаковым качеством подборки.

## Clarifications

### Session 2026-02-02

- Q: Алгоритм ранжирования проектов для подборки? → A: Tag overlap + LLM re-ranking (отбор кандидатов по совпадению тегов с IDF-весами, затем LLM ранжирует top-20 с учётом свободного текста гостя).
- Q: Триггер профилирования — когда гость впервые видит предложение? → A: Автоматически после онбординга — после выбора роли "Гость" + подтипа бот сразу предлагает указать интересы.
- Q: Поведение при недоступности LLM? → A: Graceful degradation — использовать только tag overlap (без LLM re-ranking и AI-извлечения из текста), предупредить гостя если текст не был обработан.
- Q: Краткое описание проектов в подборке — откуда берётся? → A: LLM-саммари — AI генерирует краткое описание (2-3 предложения) из полного, адаптированное под профиль гостя.

## Assumptions

- Проекты уже загружены и распределены по залам (EPIC-002 реализован).
- Тематики для кнопок берутся из существующих тегов проектов в БД (31 уникальный тег из EPIC-002/004).
- Подтип гостя (абитуриент / AI-практик / другое) уже выбран при онбординге (EPIC-001) и доступен в профиле пользователя.
- AI-анализ свободного текста использует существующий LLM-клиент (OpenRouter) из EPIC-002.
- Параллельность залов определяется по номеру зала из распределения (EPIC-002), а не по временным слотам (которые ещё не реализованы).
- Профилирование доступно как гостям, так и бизнес-партнёрам (EPIC-006 расширит для бизнеса, но базовый механизм общий).

## Scope

### In Scope

- Гибридное профилирование (кнопки + текст) в Telegram-боте
- AI-извлечение интересов из свободного текста
- Генерация персональной программы из 10-15 проектов
- Ранжирование и разделение на "обязательно" / "по возможности"
- Просмотр деталей проекта из подборки
- Обновление профиля и перегенерация

### Out of Scope

- Q&A-помощник (отдельный EPIC-007)
- Follow-up пакет после мероприятия (EPIC-014)
- Бизнес-специфичное профилирование (задача/отрасль/стадия -- EPIC-006)
- Временные слоты и точное расписание (EPIC-003/отложено)
- Запрос контактов авторов проектов (отдельная фича)
- Оценка проектов и фидбэк (отдельная фича)

## Dependencies

- **EPIC-001** (Онбординг): роль и подтип гостя уже выбраны
- **EPIC-002** (Кластеризация проектов): проекты загружены, теги назначены, залы распределены
