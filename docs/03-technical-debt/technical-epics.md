### TECH-001: Smoke-тесты на критичные хендлеры

**Проблема:**
Основной flow (онбординг, профилирование, рекомендации) не покрыт тестами. Баг в хендлере = ручная отладка на проде.

**Scope:**
- Тесты на онбординг (выбор роли, регистрация)
- Тесты на профилирование гостей
- Тесты на профилирование бизнеса
- Тесты на получение рекомендаций
- Тесты на Q&A helper

**Acceptance Criteria:**
- [ ] Покрытие критичных хендлеров smoke-тестами
- [ ] Тесты запускаются локально и в CI
- [ ] Документация по запуску тестов

**Оценка:** 2-3 часа
**Риск без реализации:** Высокий

### TECH-002: Идемпотентность загрузки seed data

**Проблема:**
`load_seed_projects` и `load_seed_experts` вызываются в lifespan при каждом старте. Нет проверки "уже загружено".

**Текущее поведение:**
```python
# main.py:40-55
async def lifespan(app: FastAPI):
    await load_seed_projects()  # Каждый раз!
    await load_seed_experts()   # Каждый раз!
```

**Решение:**
- Добавить `ON CONFLICT DO NOTHING` в INSERT
- Или флаг `seeded` в таблице settings
- Или проверка `SELECT COUNT(*) > 0`

**Acceptance Criteria:**
- [ ] Повторный запуск не дублирует данные
- [ ] Логирование "seed already loaded" при пропуске

**Оценка:** 30 минут
**Риск без реализации:** Средний (дубликаты при рестарте)

### TECH-003: Graceful degradation для LLM

**Проблема:**
Если OpenRouter лежит — рекомендации, кластеризация, Q&A не работают. Нет fallback.

**Текущее поведение:**
- LLM недоступен → ошибка → пользователь видит "что-то пошло не так"

**Решение:**
- Fallback на cached результаты (если есть)
- Fallback на простые эвристики (tag matching для рекомендаций)
- Fallback вопросы для Q&A (уже есть `get_fallback_questions`)
- Уведомление организаторам при недоступности LLM

**Acceptance Criteria:**
- [ ] При недоступности LLM система продолжает работать
- [ ] Пользователь получает результат (пусть менее качественный)
- [ ] Алерт организаторам о проблемах с LLM

**Оценка:** 1-2 часа
**Риск без реализации:** Средний (DD без рекомендаций если LLM ляжет)

### TECH-004: Rate limiting для LLM клиента

**Проблема:**
Есть retry с backoff, но нет глобального rate limiter. Массовый вызов кластеризации/рекомендаций может упереться в лимиты OpenRouter.

**Текущее поведение:**
```python
# llm_client.py
# Retry есть, rate limiting нет
```

**Решение:**
- Добавить `asyncio.Semaphore` для ограничения concurrent requests
- Или использовать `aiolimiter` для token bucket
- Конфигурируемый лимит (env: `LLM_RATE_LIMIT_RPM=60`)

**Acceptance Criteria:**
- [ ] Не более N запросов в минуту к LLM
- [ ] Очередь запросов при превышении лимита
- [ ] Метрика количества запросов к LLM

**Оценка:** 1 час
**Риск без реализации:** Низкий (OpenRouter лимиты высокие)

### TECH-005: Мониторинг и health checks

**Проблема:**
Нет способа узнать что бот жив и отвечает, кроме как написать ему.

**Текущее состояние:**
- `/health` endpoint есть (возвращает `{"status": "ok"}`)
- Нет проверки что бот реально работает
- Нет алертов

**Решение:**
- Расширить `/health` проверкой DB connection
- Добавить `/health/bot` — проверка что polling активен
- Простой мониторинг: cron job с curl + алерт в Telegram

**Acceptance Criteria:**
- [ ] `/health` проверяет DB
- [ ] `/health/bot` проверяет статус бота
- [ ] Алерт в Telegram организаторам при падении

**Оценка:** 1 час
**Риск без реализации:** Средний (не узнаем о падении вовремя)

**Acceptance Criteria:**

- [ ] История выполнения джобов в БД

**Оценка:** 2 часа

### TECH-009: Кеширование LLM ответов (расширенное)

**Проблема:**
Каждый вызов LLM = токены. Нет глобального кеширования.

**Текущее состояние:**
- Q&A кешируется (1 час TTL) ✓
- Рекомендации гостей кешируются ✓
- Кластеризация НЕ кешируется
- Бизнес-рекомендации НЕ кешируются

**Решение:**
- Redis для централизованного кеша
- TTL конфигурируемый по типу запроса
- Cache invalidation при изменении данных

**Acceptance Criteria:**
- [ ] Все LLM ответы кешируются
- [ ] Метрика cache hit/miss
- [ ] Экономия токенов >50%

**Оценка:** 3-4 часа
**Риск:** Низкий (оптимизация)
