# ADR-001: Объединение системы поддержки на thread-модели

**Статус:** Частично заменён (см. ниже)
**Дата:** 2026-06-03
**Контекст решения:** Дмитрий Горбунов

> **Заменено решением о хранилище.** Это ADR выбрал `support_messages` как стор
> сообщений. Позже (см. `docs/superpowers/specs/2026-06-03-unified-conversations-design.md`)
> единым источником правды стал **`chat_messages`** (роли user/assistant/organizer),
> чтобы организатор видел весь диалог гостя с ботом и мог перехватывать чат.
> `support_messages` депрекейтнут (не пишется). Метаданная-часть этого ADR
> (`support_threads`: needs_attention/status, + новое `taken_over`) остаётся в силе.

## Проблема

В проекте сосуществуют две несвязанные системы поддержки:

| | Система A: `support_threads` + `support_messages` | Система B: `support_log` |
|---|---|---|
| Появилась | 24 мар 2026, PR #34 (старый python-telegram-bot) | 5 мая 2026, миграция 038 (новый aiogram-бот) |
| Модель данных | thread + N сообщений (полноценный диалог) | одна строка: question + answer |
| Кто пишет | никто (бот отвязан после замены на aiogram) | бот через `/support` |
| Кто читает | веб-админка `SupportChat.tsx` | group-handler в Telegram-группе |
| Доставка ответа юзеру | `bot_messenger` (работает) | реплай в Telegram-группе |
| Состояние на проде | API работает | `ORGANIZER_CHAT_ID=0` → форвард уходит в никуда |

**Следствие:** обращение юзера через бота не видно в админке; организатор без настроенной
Telegram-группы не получает обращения вообще, при этом бот отвечает юзеру «отправлено,
ответ придёт».

## Находки research

1. В `backend/app/services/admin/support_service.py` есть три функции —
   `get_or_create_thread`, `add_user_message`, `get_thread_messages_for_context` —
   которые **не вызываются нигде** (0 вызовов вне определения). Это «мост для бота»,
   который написали, но не подключили.
2. `frontend/src/pages/SupportChat.tsx` уже умеет рендерить `sender_type === "bot"`
   (фиолетовый фон, лейбл «AI-агент»). UI готов к объединённой системе.
3. `support_messages.sender_type` — свободная строка `String(20)` без CHECK, примет
   `user`/`organizer`/`bot`.
4. Спека `specs/031-bot-replacement/brief.md:89,150` прямо предусматривает единую систему
   («SupportChat сохраняется + поддерживается ботом… ответ доходит до гостя»).
   Критерий приёмки не был выполнен.

## Решение

**Единый источник истины — thread-модель (`support_threads` + `support_messages`).
`support_log` депрекейтим. Единственный канал ответа организатора — веб-админка.**

Почему thread-модель, а не `support_log`:
- Нормальная модель диалога (тред + N сообщений). `support_log` вырожден — один
  вопрос / один ответ, многоходовую переписку не выражает.
- Веб-админка и доставка ответа юзеру под неё уже построены и работают.
- Мостовые функции под бота уже написаны — надо просто вызвать их.

Почему только веб-админка (без Telegram-группы):
- Убирает дублирующий канал и необходимость двусторонней синхронизации.
- Снимает зависимость от `ORGANIZER_CHAT_ID`.

## План реализации

1. **Бот: зеркальные ORM-модели** `SupportThread` / `SupportMessage`
   (`bot/src/models/`, `id`/`created_at` объявить явно — bot `Base` пустой).
   Зарегистрировать в `bot/src/models/__init__.py`.
2. **Бот: запись в треды.** В `support_text` (`bot/src/bot/routers/support.py`)
   писать через `get_or_create_thread` + `add_user_message`, ставить
   `needs_attention=True`. Перестать писать в `support_log`.
3. **Бот: история для AI-агента.** Подтягивать через эквивалент
   `get_thread_messages_for_context`, а не из `support_log`.
4. **Бот: убрать Telegram-группу.** Удалить `organizer_reply` group-handler,
   его регистрацию в `main.py`, настройку `organizer_chat_id`.
5. **Backend:** `reply_to_thread` уже доставляет юзеру через `bot_messenger` —
   снять `needs_attention` при ответе организатора (уже частично есть).
6. **Депрекейт `support_log`:** таблицу оставить (исторические данные), писать перестать.
   Старые записи не мигрируем (тестовые).

## Последствия

- Админка становится единственной точкой работы с поддержкой; организатор видит все
  обращения из бота и отвечает оттуда.
- Удаляется код group-handler и зависимость `ORGANIZER_CHAT_ID`.
- `support_log` остаётся в схеме как deprecated; отдельной задачей можно удалить позже.
