# Brief: UX/UI Админ-панели EventAI

> **Версия:** 1.0
> **Дата:** 2026-02-08
> **Статус:** Утверждён
> **Источники:** UX-спецификация admin-panel-ux.md, основной бриф проекта v3.0, существующая реализация (15 страниц React)
> **Неопределённость:** 0.01

---

## Что нужно сделать?

**Описание задачи:**
Переработать UX/UI веб-админки EventAI до коммерческого уровня качества. Текущая панель функционально работает, но выросла органически — навигация неочевидна, шаги подготовки мероприятия не связаны в единый поток, пользователь должен сам знать порядок действий. Нужно превратить набор страниц в wizard-подобный интерфейс, который ведёт организатора за руку от загрузки данных до запуска мероприятия.

**Job Story:**
Когда я (организатор) впервые открываю админку EventAI, я хочу видеть понятный пошаговый процесс подготовки мероприятия, чтобы пройти весь путь от загрузки проектов до отправки приглашений без инструкции и не задавая вопрос «а что дальше?».

---

## Полезное действие (Зачем?)

**Базовая проблема пользователя:**
Организатор Demo Day — не разработчик. Это 3-5 человек из core-команды Хаба, которые работают параллельно. Им нужен инструмент, в котором каждый шаг очевиден, прогресс виден, а следующее действие подсказано. Сейчас они должны самостоятельно помнить порядок: сначала импорт, потом кластеризация, потом матчинг, потом расписание, потом рассылки. Если шаг пропущен — ошибки видны только по результату.

**Ценность для бизнеса:**
- Снижение порога входа: новый организатор справляется без обучения
- Уменьшение ошибок: система не даёт пропустить шаг или запустить процесс без предварительных данных
- Готовность к коммерческому использованию: продукт можно предложить сторонним организаторам мероприятий
- Параллельная работа: 3-5 человек могут работать одновременно, видя общий прогресс

## Жёсткие ограничения (Constraints)

*То, что нельзя изменить — передаётся дизайнеру/разработчику как данность.*

- **Бюджет:** Внутренняя разработка, без внешних дизайнеров
- **Сроки:** Нет жёсткого дедлайна. Приоритет — качество, не скорость
- **Команда:** 3 разработчика + AI-ассистент. Frontend: React 19 + TypeScript + Vite + Tailwind CSS + shadcn/ui
- **Инфраструктура:** Существующий backend (FastAPI, 87 endpoints) не меняется. Вся работа — на стороне фронтенда
- **Доступ:** Только организаторы (авторизация уже реализована)
- **Иконки:** Только Lucide React (SVG), никаких unicode-эмоджи в UI
- **Данные:** Все данные получаются через существующий REST API (`/api/v1/`)

## Что нужно учесть?

- **Формат результата:** Веб-приложение (SPA), десктопный браузер
- **UX-парадигма:** Wizard/onboarding-flow (как Stripe, Intercom) — ведёт за руку шаг за шагом, но не блокирует свободную навигацию
- **Существующий UX-ориентир:** Файл `admin-panel-ux.md` содержит детальную спецификацию целевого состояния (Global Stepper, реорганизация sidebar, миграция страниц). Использовать как отправную точку, но не как жёсткое ТЗ — возможны улучшения
- **Текущее состояние:** 15 работающих страниц (Dashboard, Import, Clustering, ExpertMatching, ExpertList, Coverage, CoverageRoomDetail, Schedule, Messaging, GuestList, ProjectsList, RoomDetail, Settings, Landing, Login). Функционал протестирован, покрыт Vitest-тестами
- **Параллельная работа:** Несколько организаторов одновременно — UI должен корректно отображать общий прогресс (auto-refresh)
- **Согласования:** Финальный дизайн согласовывается с тимлидом (Дмитрий Горбунов)

## Приоритет (RICE)

- **Reach:** 3-5 организаторов на каждое мероприятие, потенциально десятки при коммерческом масштабировании
- **Impact:** 3/3 (высокий) — напрямую определяет первое впечатление и удобство основного рабочего инструмента
- **Confidence:** 90% — проблемы UX очевидны из текущей реализации, UX-спека уже проработана
- **Effort:** Средний — фронтенд-рефакторинг без изменения backend
- **RICE Score:** (10 × 3 × 0.9) / 5 = **5.4** (высокий приоритет)

---

## Критерии приёмки

- [ ] Новый организатор (не из команды разработки) может подготовить мероприятие от начала до конца без инструкции и обучения
- [ ] Весь путь «загрузил проекты → отправил приглашения» проходится за одну сессию без потери контекста
- [ ] Ни на одном шаге пользователь не задаёт вопрос «а что дальше?» — следующее действие всегда очевидно
- [ ] На каждой странице без данных отображается понятный empty state со ссылкой на нужный шаг
- [ ] Глобальный прогресс подготовки мероприятия виден с любой страницы
- [ ] Навигация группирует функции по смыслу (подготовка → распределение → запуск)
- [ ] 3-5 организаторов могут работать параллельно, видя актуальный общий прогресс
- [ ] Все существующие функции сохранены — ничего не потеряно при реструктуризации
- [ ] UI визуально аккуратный, единообразный, без «собранного из разных мест» ощущения

## Out of Scope

- Мобильная адаптация (только десктопный браузер)
- Мультиязычность (только русский)
- Тёмная тема (dark mode)
- Ролевая модель внутри админки (разные уровни доступа для разных организаторов)
- Изменения backend API (только фронтенд)
- Новый функционал — только реструктуризация и улучшение UX существующего
- Онбординг-тур (tooltips) — отдельная задача после завершения UI
- Горячие клавиши / shortcuts

## Оценка рисков

**Продуктовые риски:**
- Реструктуризация навигации может сломать привычки текущей команды — митигация: вовлечь команду в ревью на ранних этапах
- Wizard-подход может казаться ограничивающим для опытных пользователей — митигация: wizard + свободная навигация через sidebar

**Бизнес-риски:**
- Большой объём фронтенд-работы при маленькой команде — митигация: инкрементальный подход, приоритезация по фазам (сначала Global Stepper + Sidebar, потом миграция страниц)

**Общий вывод:** Риски управляемы. Фронтенд-рефакторинг не затрагивает backend, все изменения обратимы через git.
