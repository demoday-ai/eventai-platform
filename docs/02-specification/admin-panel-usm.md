# User Story Map: UX/UI Админ-панели EventAI

> **Версия:** 1.0
> **Дата:** 2026-02-08
> **Бриф:** admin-panel-ux-brief.md
> **Роль:** Организатор (3-5 чел, core-команда)
> **Неопределённость:** 0.01

## Backbone (путь организатора)

```
Данные → Распределение → Запуск
```

Параллельно на всех этапах доступны: **Dashboard**, **Рассылки**, **Аудитория**.

---

## Принципы визуала

1. **Минимализм** — каждый элемент на странице должен нести смысл. Нет декоративных элементов, нет визуального шума. Если элемент можно убрать без потери понимания — убрать.

2. **Иконки — только Lucide React (SVG)** — никаких unicode-эмоджи, никаких растровых иконок. Если нужна пиктограмма — Lucide. Если в Lucide нет подходящей — кастомный SVG в том же стиле (stroke, 24×24).

3. **Типографика как навигация** — иерархия читается через размер и вес шрифта, не через цвет и рамки. Заголовки, подзаголовки, body text — 3 уровня максимум.

4. **Цвет — функциональный** — цвет используется только для передачи статуса и действий:
   - Зелёный — завершено / успех
   - Жёлтый — требует внимания
   - Серый — не начато / неактивно
   - Акцентный — primary action (одна кнопка на экран)
   - Без градиентов, теней, декоративных подложек

5. **Воздух важнее контента** — достаточные отступы, свободное пространство между блоками. Лучше скролл, чем перегруженный экран.

6. **Одно действие на экран** — в каждый момент должно быть очевидно, что является главным действием. Одна primary-кнопка, остальные — secondary/ghost.

7. **Инструменты разработки** — при реализации использовать Claude-плагины: `frontend-design` для генерации UI-компонентов, `Playwright` для визуального тестирования, `Context7` для актуальной документации shadcn/ui и Tailwind.

## Release 1.0 — Wizard-flow от данных до запуска

### [EPIC-001] Dashboard и прогресс

---
**[US-001][Организатор] Как организатор, я хочу видеть текущий прогресс подготовки мероприятия на главной странице, чтобы понимать, на каком этапе мы находимся и что делать дальше.**
- [ ] AC: Отображается Global Stepper с 3 фазами (Данные → Распределение → Запуск)
- [ ] AC: Каждая фаза имеет статус: завершена / в процессе / не начата
- [ ] AC: При клике на фазу — переход на соответствующую страницу
- [ ] AC: Подшаги фазы видны при наведении/клике
- [ ] AC: Фаза «Данные» завершена: событие создано + проекты + студенты + эксперты загружены
- [ ] AC: Фаза «Распределение» завершена: кластеризация + матчинг + расписание одобрены
- [ ] AC: Фаза «Запуск» завершена: напоминания настроены + брифинг отправлен
---

**[US-002][Организатор] Как организатор, я хочу видеть ключевые метрики мероприятия (проекты, студенты, эксперты, партнёры, залы), чтобы оценить масштаб и готовность.**
- [ ] AC: Карточки с числами: проекты, студенты, эксперты, партнёры, залы
- [ ] AC: Данные обновляются автоматически (auto-refresh)

**[US-003][Организатор] Как организатор, я хочу видеть подсказку с ближайшим незавершённым шагом (Quick Action), чтобы не думать, что делать дальше.**

## Gap Analysis & Risks

### Пропущенные функции

- **Уведомления для организатора в реальном времени** — сейчас только auto-refresh. В будущем может понадобиться WebSocket/push для «Новый гость зарегистрировался», «Эксперт отказался»
- **Версионирование кластеризации** — возможность сравнить несколько прогонов кластеризации перед одобрением
- **Шаблоны мероприятий** — при коммерческом использовании организаторы могут захотеть клонировать настройки прошлого DD

### Риски

- [⚠️ Risk] Реструктуризация навигации сломает привычки команды — митигация: инкрементальный подход, ревью на каждом этапе
- [⚠️ Risk] Консистентность студент-проект при импорте из разных файлов — митигация: строгая валидация + понятные сообщения об ошибках
- [⚠️ Risk] Большой объём фронтенд-работы — митигация: эпики независимы, можно деливерить поштучно
