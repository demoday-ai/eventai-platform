## Что изменилось

Новые фичи добавляют три класса нагрузки, не покрытых в NFR v1.0:

1. **Полноценный чат** (бот <-> админка) - persistent messaging, до 50 одновременных диалогов
2. **Оценки экспертов через бота** - ~50 экспертов x ~15 проектов = 750 оценок за 10 часов DD
3. **Экспертный пайплайн в боте** - новый ConversationHandler с состояниями (приглашение, доступность, подтверждение)

Также: админ-панель (React) теперь подтверждена как обязательный компонент, а не "если хватит ресурсов".

## 1. Производительность (Performance Efficiency) - ДОПОЛНЕНИЯ

### 1.1 Профиль нагрузки - обновление

- **Одновременные сессии (пик):** ~200 -> **~250** (добавляется: 50 экспертов в боте с выбором слотов + чат-обращения)
- **Админ-панель:** до 5 одновременных организаторов в браузере (React SPA)

### 1.2 Пропускная способность - новые потоки

| Поток | Среднее | Пиковое | Заметки |
|-------|---------|---------|---------|
| Чат-сообщения (бот -> админка) | ~2 msg/min | ~30 msg/min | 30-50 одновременных диалогов в день DD |
| Оценки экспертов | ~1 RPS | ~5 RPS | 750 оценок за 10 часов, пик в конце сессий |
| Экспертный пайплайн (кнопки бота) | ~0.5 RPS | ~3 RPS | Приглашения, подтверждения, смена комнат |
| Множественная кластеризация | N/A | 2-3x текущего | 2-3 варианта кластеризации вместо одного |

### 1.3 Время отклика - новые операции

| Операция | Целевое время | Заметки |
|----------|---------------|---------|
| Отправка чат-сообщения из админки | < 2 000 ms | Запись в БД + отправка через Telegram API |
| Получение чат-истории | < 1 000 ms | Пагинация, последние 50 сообщений |
| Сохранение оценки эксперта | < 1 500 ms | Запись в БД + ack в боте |
| Генерация 2-3 вариантов кластеризации | < 300 000 ms | 2-3x от текущих 120 сек, прогресс-бар обязателен |
| Автогенерация расписания из кластеров | < 60 000 ms | Чисто алгоритмическая, без LLM |
| Проверка параллельности похожих тем | < 5 000 ms | При генерации расписания |

## 2. Надёжность (Reliability) - ДОПОЛНЕНИЯ

### 2.1 Целостность данных (RPO) - расширение

К существующим RPO=0 данным (подтверждения, оценки) добавляются:

- **Оценки экспертов через бота:** RPO = 0. Потеря оценки = потеря результата DD.
- **Чат-сообщения:** RPO = 0. Потеря вопроса пользователя = потеря доверия.
- **Выбор доступности эксперта:** RPO = 0. Потеря = повторный сбор, раздражение эксперта.

### 2.2 Деградация при отказе зависимостей - расширение

| Сценарий | Поведение (новое) |
|----------|-------------------|
| LLM API недоступен | Множественная кластеризация недоступна. Оценки, чат, расписание - работают (не зависят от LLM) |
| Админ-панель недоступна | Чат-ответы от организаторов невозможны. Бот продолжает принимать сообщения, копит в очередь |
| БД недоступна | Оценки и чат теряют запись. Бот кэширует в memory, retry при восстановлении |

## 3. Безопасность (Security) - ДОПОЛНЕНИЯ

### 3.1 Новая роль: Модератор

- Модератор - отдельная роль в расписании, но НЕ в системе авторизации
- Модератор не имеет доступа к админке
- Данные модераторов (ФИО, расписание) видны организаторам

### 3.2 Чат-сообщения

- Чат-сообщения содержат свободный текст от пользователей - возможен нежелательный контент
- Организатор видит все сообщения, модерация на его усмотрение
- Логи чата не содержат PII сверх Telegram username/ID (уже разрешено в NFR v1.0)

### 3.3 Оценки экспертов

- Оценки привязаны к эксперту (Telegram ID) - персональные данные
- Доступ к оценкам: только организаторы (через админку)
- Агрегированные результаты (средний балл) - могут быть публичными

## 5. Сопровождаемость (Maintainability) - ОБНОВЛЕНИЕ

### 5.1 Команда поддержки - изменение

- **Было:** 3 человека (тимлид/продукт, фронтенд/UX, бэкенд)
- **Стало:** 1 разработчик + AI-ассистент

**Последствия:**
- Решения должны быть проще, меньше движущихся частей
- Предпочитать polling вместо WebSocket (проще дебажить, не нужен отдельный сервер)
- Предпочитать расширение существующих моделей вместо новых таблиц
- Избегать отдельных микросервисов - всё в монолите

### 5.2 Сложность системы - новые компоненты

| Компонент | Сложность | Обоснование |
|-----------|-----------|-------------|
| Чат (бот + админка) | Средняя | Новая модель ChatMessage, новый ConversationHandler state, новая страница в админке |
| Оценки экспертов | Средняя | Новая модель ExpertEvaluation, новый бот-флоу, дашборд-виджет |
| Множественная кластеризация | Низкая | Расширение существующего clustering_service, хранение N вариантов |
| Экспертный пайплайн в боте | Высокая | Новый ConversationHandler, 4-5 состояний, интеграция с матчингом |
| Модераторы | Низкая | Новое поле role в schedule_slot, фильтрация при экспорте |

## 6. Наблюдаемость (Observability) - ДОПОЛНЕНИЯ

### 6.1 Новые метрики для дашборда

| Метрика | Описание | Источник |
|---------|----------|----------|
| Непрочитанные чат-сообщения | Кол-во сообщений без ответа организатора | Чат-поддержка (#8) |
| Среднее время ответа в чате | Время от вопроса до ответа организатора | Чат-поддержка (#8) |
| Оценки экспертов: прогресс | % проектов с оценкой, средний балл по залам | Обратная связь (#19) |
| Подтверждения экспертов | Подтвердил / хочет поменять / не ответил | Пайплайн (#5-7) |
| Покрытие слотов экспертами | По временным слотам, не только по залам | Матчинг UX (#4) |

### 6.2 Алерты - дополнения

- Непрочитанных чат-сообщений > 10 -> уведомление организатору
- Эксперт запросил смену комнаты -> уведомление организатору
- Генерация кластеризации завершена -> уведомление организатору

## [Сводка аналитика]

### Главные вызовы

1. **Чат = новый класс нагрузки.** 30-50 одновременных диалогов в день DD. Polling каждые 3-5 сек от админки = ~10-17 запросов/сек дополнительно к API. На одном разработчике - выбрать polling, не WebSocket.

2. **Оценки экспертов - критичные данные в боте.** 750 оценок за DD. Потеря = потеря результата мероприятия. Нужна мгновенная запись в БД, не batch.

3. **ConversationHandler растёт.** start.py уже 65KB. Добавление экспертного пайплайна + чата + оценок -> риск превышения 100KB. Рекомендуется декомпозиция на отдельные хендлеры.

4. **Один разработчик.** Все решения должны быть проще, чем в NFR v1.0 предполагалось. Предпочитать polling, монолит, расширение существующих таблиц.

### Серые зоны

1. **Формат оценочной формы экспертов.** Критерии и шкала не определены. Проверить по `data/test/experts_22jan_anon.xlsx`.

2. **Real-time чата.** Polling vs SSE vs WebSocket - архитектурное решение. Рекомендация: polling (проще, один разработчик).

### Рекомендации

1. **Декомпозировать start.py** перед добавлением новых хендлеров. Выделить expert_handler.py, chat_handler.py, feedback_handler.py.

2. **Polling для чата** с интервалом 3 сек. Проще WebSocket, достаточно для 30-50 диалогов, не требует дополнительной инфраструктуры.

3. **Расширять существующие модели** где возможно (добавить поля в Expert, ScheduleSlot) вместо создания новых таблиц.

4. **Прогресс-бар для множественной кластеризации.** 2-3 варианта = 4-5 минут. Без индикации прогресса пользователь решит, что система зависла.
