# Brief: Доработки DemoDay AI по фидбеку организатора

> **Версия:** 1.0
> **Дата:** 2026-03-17
> **Статус:** Утвержден
> **Источники:** Фидбек Инны (куратор ИТМО, организатор Demo Day) от 25-26.02.2026 - два сообщения + заметки с созвона
> **Неопределенность:** 0.02

---

### Что нужно сделать?
- **Описание задачи:** Доработать существующий продукт (AI-агент-куратор DemoDay) по фидбеку реального пользователя - Инны, куратора ИТМО и организатора Demo Day. 20 пунктов доработок в 4 группах приоритетов: quick fixes, кластеризация, экспертный пайплайн, коммуникации и аналитика.
- **Job Story:** Когда организатор готовит Demo Day на 330 проектов, он хочет управлять кластеризацией, экспертами, расписанием и коммуникациями из одного инструмента, чтобы не тратить недели на ручную работу в Excel и Telegram.

### Полезное действие (Зачем?)
- **Базовая проблема пользователя:** Продукт уже покрывает базовый сценарий, но при реальном использовании организатору не хватает гибкости в кластеризации, автоматизации работы с экспертами и обратной связи. Без этих доработок инструмент останется демо-прототипом, а не рабочим инструментом.
- **Ценность для бизнеса:** Переход от "показать на Demo Day" к "использовать на Demo Day". Инна подтвердила готовность пользоваться продуктом при реализации доработок.

### Жесткие ограничения (Constraints)
- **Бюджет:** Нет (pet-project, студенческая разработка)
- **Сроки:** Нет жестких дедлайнов
- **Команда:** Один разработчик (Дмитрий) + AI-ассистент (Claude)
- **Инфраструктура:** Существующий стек (FastAPI, React, Telegram-бот, PostgreSQL, Yandex Cloud VM)
- **Регуляторные:** Нет

### Что нужно учесть?
- **Обратная совместимость:** Все доработки дополняют существующий функционал, не ломают его. Ручное назначение экспертов остается доступным наряду с новым пайплайном через бота.
- **Один разработчик:** Решения должны быть простыми в реализации и поддержке. Никакой избыточной сложности.

### Приоритет (RICE)
- **Reach:** 5-10 организаторов Demo Day, 50-100 экспертов, 330 проектов
- **Impact:** Высокий - переводит продукт из прототипа в рабочий инструмент
- **Confidence:** Высокая - фидбек от реального пользователя, конкретные пожелания
- **Effort:** Большой - 20 пунктов, 10 спек
- **RICE Score:** (10 x 3 x 0.9) / 8 = 3.4

### Критерии приемки

**P1 - Quick fixes:**
- [ ] Кнопки кофе-брейк и секции в расписании работают
- [ ] Кнопка "Связь с организаторами" в боте
- [ ] Минимальное кол-во экспертов в зале настраивается из интерфейса
- [ ] Добавление/редактирование студентов и названия проекта
- [ ] Напоминания отправляются и студентам, и экспертам за 30 мин до слотов

**P2 - Кластеризация:**
- [ ] Организатор задает ограничения: макс. проектов в кластере, перерывы, лимит по времени
- [ ] Проекты для бизнеса/прожарки можно объединить в одну комнату независимо от тематики
- [ ] Похожие темы не ставятся в параллельные залы
- [ ] Можно сгенерировать несколько вариантов кластеризации и выбрать лучший
- [ ] Из кластеризации генерируется расписание с временными слотами и перерывами

**P3 - Экспертный пайплайн:**
- [ ] Матчинг учитывает должность эксперта, а не только теги
- [ ] В UI распределения экспертов видны временные слоты
- [ ] Эксперт через бота получает приглашение, указывает доступность, получает назначение
- [ ] Эксперт может подтвердить или запросить смену комнаты
- [ ] Организатор видит запросы на смену в админке
- [ ] Эксперты без ограничений по времени пропускают выбор слота

**P4 - Коммуникации и аналитика:**
- [ ] Модераторы добавляются в расписание, выгрузка расписания по модератору
- [ ] Полноценный чат: пользователи пишут в бота, организаторы отвечают из админки (все роли)
- [ ] Эксперты оценивают проекты через бота, результаты на дашборде

### Out of Scope
- Мобильное приложение
- Мультитенантность (несколько организаций)
- Синхронизация с внешними сервисами (Notion, Google Sheets и др.) - отложено до уточнения конкретных потребностей
- Платежи, монетизация
- Автоматическое распознавание речи на самом мероприятии

### Оценка рисков
- **Продуктовые риски:** Scope большой (20 пунктов) при одном разработчике. Митигация: приоритизация P1->P4, каждая группа приносит ценность независимо.
- **Бизнес-риски:** Следующий Demo Day может пройти до завершения всех доработок. Митигация: P1+P2 покрывают минимально жизнеспособный сценарий организатора.
