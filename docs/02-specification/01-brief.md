> **Версия:** 1.0
> **Дата:** 2026-02-01
> **Статус:** Утверждён для Discovery
> **Источники:** CustDev-интервью с организатором (#1), аналитика прошлого DD, RICE-матрица, Pain Map, JTBD

## Что нужно сделать?

**Описание задачи:**
Создать AI-платформу (Telegram-бот) для автоматизации подготовки и проведения Demo Day в AI Talent Camp: кластеризация ~330 проектов по залам, распределение экспертов, подтверждение участия студентов, напоминания, предобработка и рассылка обратной связи. Единая точка входа для всех ролей (организатор, эксперт, студент, внешний гость).

**Job Story:**
Когда до Demo Day остаётся 2-4 недели и нужно распределить ~330 проектов по 6-10 залам, обеспечить покрытие экспертами и подтвердить участие студентов, я хочу, чтобы AI-система автоматизировала кластеризацию, распределение и коммуникацию, чтобы расписание было точным, все участники подтвердили присутствие, а после DD каждый студент получил обратную связь.

---

## Полезное действие (Зачем?)

### Базовая проблема пользователя:

**Для организатора (core-команда Хаба, ~5 чел):**
- Один человек вручную составляет расписание на ~330 проектов, используя ChatGPT для кластеризации — "ChatGPT кластеризует фигово", дубликаты (один проект в двух комнатах)
- 80 студентов не заполнили форму и не нашли себя в расписании за ночь до DD
- 15 проектов просто не пришли — "люди не отвечали"
- Нужно 60-80 экспертов, набирается ~50, активных ~30-40
- Обратная связь от экспертов собирается, но не рассылается студентам: "Кто-то может написать 'Херня!' и всё"
- "Нету нормальной системы, которая это всё отслеживает"

**Для эксперта (~50 приглашённых):**
- Идёт на защиты вслепую — нет информации о проектах заранее
- Попадает на нерелевантную тематику — "Зачем я вообще пришёл, потратил своё время?"
- Забывает оценить проекты после DD, игнорирует

**Для студента (~330 проектов):**
- Не знает свой слот вовремя — расписание выходит поздно
- Не получает обратную связь — "в саппорт постоянно пишут: можно ОС увидеть?"
- Оценка несправедлива, если в зале не было эксперта

**Для внешнего гостя (вторичный, ~50 доходимость из 190 регистраций):**
- 330 проектов в 6-10 залах — видит максимум 16%
- "Максимально неудобное мероприятие для внешнего"
- Demo-Hero не помог: "фигня"

### Ценность для бизнеса:
- Сокращение времени составления расписания с дней до часов
- Снижение неявок студентов (с ~5% до <2%) и экспертов (с ~20% до <10%)
- 100% студентов получают отфильтрованную обратную связь (сейчас 0%)
- Каждый зал обеспечен экспертами по релевантной теме
- Организаторы получают дашборд вместо "ходить по залам и спрашивать"
- Замена Demo-Hero единым ботом для всех ролей

## Необходимая информация

### От организатора:
- Таблица экспертов с тегами интересов (текущий формат)

### Технические:
- Telegram Bot API: лимиты на рассылку (30 сообщений/сек для бота, 20 сообщений/мин для группового чата)
- GitHub API: rate limits (5000 req/hour с токеном)
- LLM API: стоимость обработки 330 проектов (кластеризация + обзоры + предобработка ОС)
- Формат экспорта данных из формы контрольного рубежа

## Критерии приёмки

### MVP (Must Have — RICE > 200):
- [ ] Бот принимает роль при старте (организатор / эксперт / студент / гость)
- [ ] Бот рассылает расписание студентам персонально и собирает подтверждение (Да / Нет)
- [ ] Неподтверждённые студенты эскалируются организатору за 5+ дней до DD
- [ ] Напоминалки: за день и за час до выступления (студентам, экспертам)
- [ ] AI кластеризует проекты по темам на основе описаний + GitHub README
- [ ] Организатор просматривает и подтверждает кластеризацию
- [ ] Система не допускает дубликатов (один проект — один зал)
- [ ] AI предобрабатывает ОС от экспертов: фильтрация неконструктива, группировка
- [ ] Организатор модерирует ОС → рассылка студентам через бота

### Should Have (RICE 50-200):
- [ ] Бот предлагает эксперту комнату по тегам интересов, эксперт подтверждает
- [ ] Дашборд покрытия: сколько экспертов подтвердили по каждому залу
- [ ] Эксперт получает овервью проектов своей комнаты за день до DD

### Could Have (RICE < 50):
- [ ] Быстрая оценка проектов прямо в Telegram (эксперт, после DD)
- [ ] Персональная программа для внешних гостей (профилирование + подборка)
- [ ] Дашборд реального времени для оргов (кто пришёл, пустые слоты)

### Quality Gates:
- [ ] 0 дубликатов в расписании
- [ ] ≥90% студентов подтверждают участие за 7+ дней до DD
- [ ] ≥80% экспертов подтверждают комнату за 7+ дней до DD
- [ ] ≥80% студентов получают ОС в течение 48 часов после DD
- [ ] Кластеризация тематически когерентна (оценка организатора ≥ 4/5)

## Out of Scope

### Не входит в MVP:
- Видео-конференции / стриминг (организуется отдельно через Zoom/PowerShell Chat)
- Составление расписания по времени (только кластеризация по залам — тайминг решает орг)
- Q&A-помощник для экспертов (отклонено: "предложения по вопросам — нет, это от человека должно исходить")
- Follow-up пакет с LOI/шаблонами писем (нет запроса от гостей)
- Мобильное приложение (только Telegram)
- Транскрибация записей DD (техническая возможность, но не приоритет)
- Валидация качества презентаций до DD ("насильно не заставишь")
- Интеграция с системой оценивания (руководитель ставит оценку вне бота)

### Сознательные ограничения:
- Бот не заменяет организатора — модерирует, помогает, но решения за человеком
- Бот не оценивает проекты — только собирает и рассылает оценки экспертов
- Кластеризация предлагается AI, но утверждается организатором
- ОС предобрабатывается AI, но модерируется организатором перед отправкой

## Открытые вопросы

1. **Формат данных от студентов:** Как именно выглядит форма контрольного рубежа? Какие поля парсить для кластеризации? Есть ли API к Google Forms/Sheets?

2. **Теги экспертов:** В каком формате хранится таблица тегов? Кто её обновляет? Сколько уникальных тегов/категорий?

3. **Масштаб кластеризации:** Сколько залов выделяется на следующий DD? Фиксировано ли количество (6/10) или зависит от числа проектов?

4. **Формат ОС экспертов:** Свободный текст? Структурированная форма? Шкала + комментарий? Это влияет на AI-предобработку.

5. **Интервью с другими сегментами:** Валидация болей экспертов (запланировано) и студентов (нужно провести). Текущие данные — через призму организатора.

6. **Доступ к данным прошлого DD:** Можно ли получить реальные данные (расписание, список проектов, оценки) для тестирования кластеризации?

7. **Онлайн-участие:** Часть студентов выступает онлайн (Zoom). Нужно ли учитывать гибридный формат?

## Оценка рисков

### Продуктовые риски:

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| AI кластеризует не лучше ChatGPT | Средняя | Высокое | Использовать GPT-4.1 + GitHub + описания (мультимодальный контекст); человеческая модерация |
| Студенты игнорируют бота так же, как формы | Высокая | Высокое | Эскалация кураторам, интеграция с учебным процессом, персональные (не массовые) пинги |
| Эксперты воспримут бота как спам | Средняя | Среднее | Минимум сообщений, только по делу, возможность отписаться |
| Предобработка ОС теряет суть комментария | Низкая | Среднее | Организатор модерирует перед отправкой, fallback на оригинальный текст |

### Бизнес-риски:

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Формат DD изменится кардинально | Низкая | Высокое | Модульная архитектура, каждая фича — отдельный сценарий |
| Нет доступа к данным (формы, таблицы экспертов) | Средняя | Критическое | Договориться с организаторами заранее, предусмотреть ручной импорт |
| Команда не успевает MVP к DD | Средняя | Высокое | Приоритизация по RICE: напоминалки и подтверждения — минимальный скоуп |
| Один интервью — недостаточная валидация | Высокая | Среднее | Провести интервью с экспертом и студентом до начала разработки |

### Общий вывод:
Риски управляемые. Ключевой фактор — прямой доступ к процессам AI Talent Camp (мы — команда кэмпа). Критичная зависимость — получение реальных данных (формы, таблицы экспертов, расписание прошлого DD). MVP по RICE: напоминалки + подтверждения — технически простые, высокий impact.
