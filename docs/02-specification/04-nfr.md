# Нефункциональные требования (NFR)

> **Версия:** 1.0
> **Дата:** 2026-02-02
> **На основе:** Brief v3.0, USM v2.0, CustDev #1-3, RICE v3.0
> **Контекст:** Telegram-бот, Yandex Cloud VM, демо 6 февраля 2026

## 1. Performance (Производительность)

### NFR-P1. Отклик бота на команды и кнопки

| Метрика | Целевое значение | Обоснование |
|---------|-----------------|-------------|
| Ответ на /start, /role, inline-кнопки | **< 1 сек** | UX-стандарт Telegram-ботов: пользователь ожидает мгновенной реакции |
| Ответ на CRUD-операции (подтверждение участия, оценка) | **< 2 сек** | Запись в БД + ack |
| Показ typing indicator | **< 300 мс** | Если операция дольше 1 сек — показывать "typing..." |

### NFR-P2. Отклик AI-функций (LLM)

| Операция | Целевое значение | Обоснование |
|----------|-----------------|-------------|
| Профилирование гостя/бизнеса (парсинг свободного текста) | **< 8 сек** | CustDev: "2 минуты на профилирование" — ожидание ответа AI должно быть частью, не сверх |
| Генерация подборки 10-15 проектов | **< 10 сек** | Поиск + ранжирование + формирование карточек |
| Q&A-помощник (генерация вопросов) | **< 8 сек** | Одноразовый запрос, не real-time чат |
| Матрица сравнения проектов | **< 10 сек** | Структурирование данных по нескольким проектам |
| Кластеризация 330 проектов | **< 120 сек** | Batch-операция организатора. Допустим: "Кластеризация займёт ~1-2 мин" |
| Предобработка ОС (фильтрация неконструктива) | **< 5 сек на комментарий** | Batch: 330 × 2 эксперта = ~660 комментариев. Общее время batch: < 30 мин |
| Овервью проекта (карточка из описания + GitHub) | **< 5 сек** | Batch при формировании овервью для экспертов |

### NFR-P3. Рассылка сообщений

| Операция | Целевое значение | Обоснование |
|----------|-----------------|-------------|
| Массовая рассылка напоминалок (435 пользователей) | **< 30 сек** | Telegram Bot API: 30 msg/sec → 435/30 ≈ 15 сек + overhead |
| Рассылка расписания студентам (330 шт) | **< 20 сек** | После утверждения кластеризации |
| Рассылка приглашений экспертам (50 шт) | **< 5 сек** | |

## 2. Scalability (Масштабируемость)

### NFR-S1. Пользовательская нагрузка

| Параметр | MVP (демо) | Продакшн-цель | Обоснование |
|----------|-----------|---------------|-------------|
| Одновременных пользователей | **50** | **200** | Пик: начало DD, все получают напоминалки одновременно |
| Всего зарегистрированных | **~50** (демо-данные) | **550+** | Олег: "550 студентов, растёт с каждым годом" |
| Проектов в системе | **330** | **1000** | Олег: "потенциально можно говорить о всём университете" |
| Экспертов | **50** | **100** | Рост пропорционально проектам |
| Бизнес-партнёров | **10** | **50** | По мере улучшения конверсии рассылок |

### NFR-S2. Хранение данных

| Параметр | Целевое | Обоснование |
|----------|---------|-------------|
| Размер БД (один DD) | **< 500 MB** | 330 проектов × (описание + теги + метаданные) + пользователи + оценки + ОС |
| Размер файлов артефактов (презентации, GitHub-кэш) | **< 5 GB** | 65 GB SSD на VM, нужен запас для OS и логов |
| Общий размер на диске | **< 10 GB** (из 65 GB) | Чтобы осталось место для ОС, логов, temp |

### NFR-S3. Ограничения Telegram Bot API

| Ограничение | Значение | Митигация |
|-------------|----------|-----------|
| Сообщений в секунду (один бот) | **30 msg/sec** | Очередь с rate limiter |
| Сообщений в чат в минуту | **20 msg/min** | Batch сообщений, не спамить одного юзера |
| Размер сообщения | **4096 символов** | Длинные подборки разбивать на части |
| Inline-кнопок в ряду | **8** | Дизайн кнопок с учётом лимита |
| Callback data | **64 байта** | Короткие payload-ы |

## 3. Availability (Доступность)

### NFR-A1. Uptime

| Период | Целевое | Обоснование |
|--------|---------|-------------|
| **День DD (критическое окно: 10:00-20:00)** | **99.5%** (< 6 мин даунтайма) | Во время DD бот — основной инструмент навигации |
| **За неделю до DD** | **95%** | Рассылки, подтверждения, профилирование |
| **После DD (1 неделя)** | **95%** | Follow-up, ОС, контакты |
| **Вне цикла DD** | **Не критично** | Нет активных пользователей |

### NFR-A2. Graceful degradation

| Сценарий отказа | Поведение | Обоснование |
|-----------------|-----------|-------------|
| LLM API недоступен | Бот продолжает работать: CRUD-операции (подтверждение, оценка, напоминалки) — без AI. AI-фичи (кластеризация, Q&A, профилирование) — сообщение "AI-сервис временно недоступен" | Ядро бота не должно зависеть от LLM |
| БД недоступна | Бот отвечает "Технические работы, попробуйте позже" | Критическая зависимость, быстрый перезапуск |
| Telegram API down | Ничего не работает (внешняя зависимость) | Не в нашем контроле. Мониторинг + алерт |
| VM reboot | Автозапуск бота через systemd | Uptime после перезагрузки < 30 сек |

## 4. Security & Privacy (Безопасность и приватность)

### NFR-SEC1. Защита персональных данных

| Требование | Описание | Обоснование |
|------------|----------|-------------|
| **152-ФЗ** | Персональные данные хранятся на территории РФ (Yandex Cloud) | Законодательное требование |
| **Согласие на контакт** | Контакты передаются только с согласия обеих сторон (US-020, US-021) | CustDev #3 (Олег): "Не всем ребятам будет интересно. Это будет негативный фактор" |
| **Минимизация PII** | Хранить: Telegram ID, роль, подтип. Не хранить: паспортные данные, адреса | Принцип минимизации данных |
| **PII в логах** | Логи не содержат персональных данных (Telegram ID — псевдоним, не PII) | Безопасность логирования |

### NFR-SEC2. Аутентификация и авторизация

| Требование | Описание | Обоснование |
|------------|----------|-------------|
| **Идентификация через Telegram** | Telegram ID = уникальный идентификатор пользователя. Нет отдельной регистрации | Telegram обеспечивает аутентификацию |
| **Ролевой доступ** | Организатор видит дашборд и управление. Студент — свой слот и ОС. Эксперт — свою комнату. Гость/бизнес — свою программу | 5 ролей с разными правами |
| **Смена роли** | /role — смена с подтверждением. Нет одновременного использования двух ролей | Защита от путаницы |
| **Организатор = whitelist** | Роль "Организатор" доступна только Telegram ID из whitelist-а | Защита от несанкционированного доступа к управлению |

### NFR-SEC3. Безопасность инфраструктуры

| Требование | Описание |
|------------|----------|
| SSH-доступ к VM | Только через bastion (`bastion.camp.aitalenthub.ru`), ключ |
| Telegram Bot Token | В переменной окружения (.env), не в коде |
| LLM API ключи | В переменной окружения (.env), не в коде |
| HTTPS | Webhook через Traefik с TLS (домен `team10.camp.aitalenthub.ru`) |
| Firewall | Только порты 443 (webhook), 22 (SSH через bastion) |

### NFR-SEC4. Модерация AI-выходов

| Требование | Описание | Обоснование |
|------------|----------|-------------|
| **Кластеризация** | AI предлагает → организатор утверждает | "Бот не заменяет организатора" (Brief) |
| **Обратная связь** | AI фильтрует → организатор модерирует → рассылка | CustDev: "кто-то напишет 'Херня!' и всё" |
| **Подборка для бизнеса** | AI генерирует → Олег (или орг) может промодерировать | CustDev #3: качество — фильтр перед показом бизнесу |

## 5. Usability (Удобство использования)

### NFR-U1. Каналы и платформа

| Требование | Значение | Обоснование |
|------------|---------|-------------|
| **Платформа** | Только Telegram | CustDev #1: "Я бы не пользовалась никаким приложением." CustDev #2: "Неудобно себе отдельно ставить для демо-дня" |
| **Язык интерфейса** | Русский | Основная аудитория — русскоязычная |
| **Мультиязычность** | Вне скоупа MVP | Gap Analysis: "есть ли иностранные гости?" — не приоритет |

### NFR-U2. Взаимодействие

| Требование | Значение | Обоснование |
|------------|---------|-------------|
| **Минимум текстового ввода** | Inline-кнопки для всех стандартных действий | Brief: "inline-кнопки для подтверждений" |
| **Свободный текст** | Только для профилирования гостя/бизнеса и комментариев | AI парсит текст, кнопки — для подтверждений |
| **Профилирование** | **< 2 минуты** от начала до подборки | CustDev: "2 минуты на описание запроса" |
| **Оценка проекта** | **< 2 минуты** на проект | CustDev #2: "если упаковать в пару сообщений, то да" |
| **Глубина навигации** | **< 3 нажатия** до целевого действия | /start → роль → действие |
| **Обратная навигация** | Кнопка "Назад" или /menu в каждом экране | Стандарт UX ботов |

### NFR-U3. Информационная плотность

| Требование | Значение | Обоснование |
|------------|---------|-------------|
| **Карточка проекта** | 3-5 строк: название, описание (2 предложения), стек, теги, зал/время | Telegram — узкий экран, длинные сообщения не читают |
| **Подборка** | 10-15 проектов с приоритизацией: "обязательно" vs "если время" | USM: SS-006 |
| **Сообщение-напоминалка** | 1-2 строки: дата, время, зал, ссылка | Минимум информации для действия |
| **Batch-карточки** | Не более 5 проектов в одном сообщении | Telegram-лимит 4096 символов; разбивать |

## 6. Reliability (Надёжность)

### NFR-R1. Консистентность данных

| Требование | Описание |
|------------|----------|
| **0 дубликатов в расписании** | Один проект — один зал. Системная проверка при кластеризации |
| **Все студенты в расписании** | Валидация: после кластеризации — отчёт "N студентов без слота" |
| **Идемпотентность кнопок** | Двойное нажатие "Подтверждаю" не создаёт дубль |
| **Атомарность оценок** | Оценка эксперта сохраняется целиком (баллы + комментарий) или не сохраняется |

### NFR-R2. Восстановление

| Требование | Значение |
|------------|---------|
| **Перезапуск бота** | Автовосстановление через systemd, < 30 сек |
| **Потеря сообщений** | Webhook + retry: Telegram повторяет unacked updates |
| **Откат кластеризации** | Организатор может вернуться к предыдущей версии расписания |

## 7. Infrastructure (Инфраструктура)

### NFR-I1. Вычислительные ресурсы

| Ресурс | Значение | Ограничение |
|--------|---------|-------------|
| **CPU** | 4 vCPU | VM кэмпа, не расширяется |
| **RAM** | 8 GB | Бот + БД + кэш должны уложиться |
| **Disk** | 65 GB SSD | OS (~5 GB) + приложение + данные < 15 GB |
| **OS** | Ubuntu 22.04 | Предустановлено |
| **Network** | Private subnet, NAT через Edge VM | Исходящий трафик через NAT |

### NFR-I2. Зависимости

| Зависимость | Тип | Критичность | Fallback |
|-------------|-----|-------------|----------|
| Telegram Bot API | Внешняя | Критическая | Нет (единственный канал) |
| OpenRouter (LLM API) | Внешняя | Высокая | CRUD-функции работают без LLM |
| PostgreSQL / SQLite | Локальная | Критическая | Быстрый перезапуск |
| Traefik (reverse proxy) | Инфраструктура кэмпа | Критическая | Polling mode вместо webhook |

### NFR-I3. Деплой

| Требование | Значение | Обоснование |
|------------|---------|-------------|
| **Способ деплоя** | Docker Compose или systemd service | Простота на одной VM |
| **Zero-downtime deploy** | Не требуется для MVP | Допустим < 1 мин даунтайм при обновлении |
| **Rollback** | `git revert` + перезапуск сервиса | Минимальный процесс |
| **Конфигурация** | .env файл (токены, ключи) | Не в коде, не в git |

## 8. Observability (Наблюдаемость)

### NFR-O1. Логирование

| Требование | Описание |
|------------|----------|
| **Structured logging** | JSON-формат: timestamp, level, event, user_role, action |
| **Уровни** | ERROR (сбои), WARN (retry, rate limit), INFO (действия пользователей), DEBUG (off в prod) |
| **Ротация** | Logrotate: максимум 7 дней или 500 MB |
| **Без PII** | Telegram ID — ok (псевдоним), имена/контакты — нет |

### NFR-O2. Мониторинг

| Метрика | Способ | Порог алерта |
|---------|--------|-------------|
| Бот жив | Health-check endpoint | Нет ответа > 60 сек |
| Ошибки LLM API | Счётчик 5xx/таймаутов | > 3 ошибки за 5 мин |
| Очередь рассылки | Размер очереди | > 500 сообщений (застряла) |
| Disk usage | `df` | > 80% |

### NFR-O3. Аналитика (для оргов)

| Метрика | Описание | Обоснование |
|---------|----------|-------------|
| Доходимость по ролям | % подтвердивших → пришедших | CustDev #1: "нету нормальной системы" |
| Покрытие экспертами | Залы с/без экспертов | RICE H3 |
| Воронка гостей | Регистрация → профилирование → посещение → follow-up | USM: US-023 |
| Подтипы гостей | Абитуриент / AI-практик / Другое | RICE H16 |
| Оценки проектов | % проектов с оценкой, средний балл | CustDev #1: "не все эксперты выставляют" |

| **Demo-сценарий** | E2E прогон всех 5 ролей на реальных данных | Критерий приёмки демо 6 февраля |

### NFR-M3. Документация

| Требование | Описание |

|------------|----------|--------|
| **152-ФЗ** | Данные на территории РФ (Yandex Cloud) | Выполняется |
