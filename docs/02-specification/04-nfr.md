## 1. Производительность (Performance Efficiency)

*Перевод бизнес-пожеланий «быстро» в цифры.*

### 1.1 Профиль нагрузки

- **DAU (Daily Active Users):** ~435 в день DD (330 студентов + 50 экспертов + 50 гостей + 5 организаторов). Вне цикла DD — 0.
- **MAU (Monthly Active Users):** ~435. Система событийная: активность сконцентрирована в окне ±2 недели от DD.
- **Одновременные сессии (пик):** ~200 в начале DD (все получают напоминалки одновременно, 10:00-10:30).

> [💡 Предложение] Для событийных платформ (конференции, хакатоны) пик = первые 30 мин мероприятия. Стандарт: 40-50% DAU одновременно.

### 1.2 Пропускная способность (Throughput)

- **Среднее:** ~5 RPS (событийный чат-бот: нажатия кнопок, просмотр карточек)
- **Пиковое:** ~30 RPS (момент массовой рассылки напоминалок 435 пользователям). Ограничение Telegram Bot API: 30 msg/sec.

> [💡 Предложение] Telegram-бот как единственный канал доставки — пиковая нагрузка определяется лимитом Telegram API, а не серверным RPS. Очередь сообщений обязательна.

### 1.3 Время отклика (Latency)

- **Команды и inline-кнопки:** < **1 000 ms** (95th percentile). Стандарт UX Telegram-ботов: пользователь ожидает мгновенной реакции.
- **CRUD-операции** (подтверждение участия, оценка проекта): < **2 000 ms** (95th percentile). Запись в БД + отправка ack.
- **AI-операции** (профилирование, генерация подборки, Q&A-помощник): < **10 000 ms**. CustDev: "2 минуты на профилирование" — ожидание AI-ответа должно быть частью этих 2 минут, не сверх. Typing-индикатор обязателен при ожидании > 1 сек.
- **Batch-операции** (кластеризация 330 проектов, предобработка ~660 комментариев ОС): < **120 000 ms**. Допустим прогресс-бар: "Кластеризация: 45%..."

## 2. Надёжность (Reliability)

*Перевод бизнес-страхов «всё сломалось» в SLA.*

### 2.1 Доступность (Availability)

- **SLA в день DD (10:00-20:00):** **99.5%** — допустимый простой < 6 минут за 10-часовое окно.
- **SLA за неделю до DD:** **95%** — рассылки, подтверждения, профилирование. Допустим простой до 8 часов.
- **SLA после DD (1 неделя):** **95%** — follow-up, рассылка ОС, обмен контактами.
- **Вне цикла DD:** не критично — нет активных пользователей.
- **Плановое окно обслуживания:** ночь (00:00-06:00 MSK), не в день DD.

> [💡 Предложение] Для событийных платформ 99.5% в день мероприятия — стандарт Silver. Gold (99.9%) не оправдан при 1 дне нагрузки. Bronze (99%) = 6 минут простоя — рискованно для 10-часового окна.

### 2.2 Целостность данных (RPO — Recovery Point Objective)

- **Максимально допустимая потеря данных:** **0 минут** для подтверждений участия, оценок, обратной связи.
- *Перевод:* «Если всё упадёт, мы не можем потерять ни одну оценку эксперта и ни одно подтверждение студента.» CustDev #1: "15 проектов просто не пришли" — система подтверждений должна быть надёжной.
- **Для AI-генерированного контента** (подборки, вопросы): потеря допустима — можно перегенерировать.

### 2.3 Скорость восстановления (RTO — Recovery Time Objective)

- **В день DD:** < **5 минут**. Автоматический перезапуск сервиса + replay непроцессированных обновлений Telegram.
- **Вне DD:** < **1 час**. Ручное вмешательство допустимо.
- *Перевод:* «После аварии в день DD система должна заработать за 5 минут, без потери данных.»

### 2.4 Деградация при отказе зависимостей

| Сценарий | Поведение системы |
|----------|-------------------|
| LLM API недоступен | CRUD-функции продолжают работать (подтверждения, оценки, напоминалки). AI-фичи (кластеризация, Q&A, профилирование) — сообщение "AI-сервис временно недоступен, попробуйте позже" |
| БД недоступна | Бот отвечает "Технические работы, попробуйте через минуту" |
| Telegram API down | Ничего не работает (внешняя зависимость, не в нашем контроле) |

## 3. Безопасность (Security)

*Законы, правила и защита.*

### 3.1 Регуляторные требования (Compliance)

- [x] **152-ФЗ** (персональные данные РФ) — данные хранятся на территории РФ
- [ ] GDPR — не применимо (нет европейских пользователей)
- [ ] PCI DSS — не применимо (нет платежей)
- [ ] HIPAA — не применимо
- [ ] Внутренние стандарты СБ — не применимо (университетский проект)

### 3.2 Классификация данных

- **Тип данных:** Персональные данные (Telegram ID, имя, роль, контактная информация — с согласия)
- **Уровень защищённости:** **УЗ-4** (общедоступные персональные данные, < 100 тыс. субъектов, нет спецкатегорий)

> [💡 Предложение] УЗ-4 — минимальный уровень по 152-ФЗ. Telegram ID — псевдоним, не прямой идентификатор. Имена и контакты передаются только с двустороннего согласия (CustDev #3, Олег: "Не всем ребятам будет интересно. Это будет негативный фактор").

### 3.3 Аутентификация и авторизация

- **Требования к входу:** Аутентификация через Telegram (Telegram ID = уникальный идентификатор). Нет отдельной регистрации / паролей.
- **Парольная политика:** не применимо (Telegram обеспечивает аутентификацию).
- **Сессии:** Telegram-сессия (управляется Telegram, не системой).
- **Ролевой доступ:** 5 ролей с разными правами. Организатор — только из whitelist по Telegram ID.
- **Привилегированный доступ:** роль "Организатор" доступна только заранее заданным Telegram ID (whitelist). Защита от несанкционированного доступа к управлению расписанием, рассылками, дашбордам.

### 3.4 Локализация данных

- **Где можно хранить данные:** **Только РФ** (Yandex Cloud, ЦОД в России). Требование 152-ФЗ.

### 3.5 Дополнительные требования безопасности

- Секреты (токены бота, API-ключи LLM) — не в коде, не в системе контроля версий. В переменных окружения.
- Логи не содержат персональных данных (имён, контактов). Telegram ID допустим (псевдоним).
- Все критические AI-решения модерируются человеком: кластеризация → организатор утверждает; ОС → организатор модерирует; подборка для бизнеса → организатор может промодерировать. CustDev #1: "кто-то напишет 'Херня!'". Brief: "бот не заменяет организатора".

## 4. Масштабируемость (Scalability)

*Как система будет расти.*

### 4.1 Ожидаемый рост

- **Текущее состояние (февраль 2026):** 550 студентов, ~330 проектов, ~50 экспертов, ~50 гостей.
- **Через 6 месяцев:** ~600 студентов, ~350 проектов. Рост органический, без скачков. CustDev #3 (Олег): "С каждым годом уровень у ребят растёт."
- **Через 1 год:** ~700 студентов, ~400 проектов, ~70 экспертов. Возможно расширение на другие факультеты ИТМО.
- **Через 3 года:** до ~2000 студентов, ~1000 проектов. CustDev #3 (Олег): "Потенциально можно говорить о всём университете, либо о какой-то общности решения этой задачи."

### 4.2 Требования к масштабированию

- **Горизонтальное:** Не нужно для MVP. Опционально при переходе на общеуниверситетский масштаб (1000+ проектов).
- **Пределы:** Система должна работать при **1000 проектов** и **2000 пользователей** без деградации производительности сверх заявленных метрик (раздел 1).
- **Ограничение:** VM кэмпа — 4 vCPU, 8 GB RAM, 65 GB SSD. Не расширяется в рамках кэмпа. При переходе в продакшн — пересмотреть.

## 5. Сопровождаемость (Maintainability)

*Как будем поддерживать после запуска.*

### 5.1 Команда поддержки

- **Кто поддерживает:** Внутренняя команда (3 человека: тимлид/продукт, фронтенд/UX, бэкенд/бизнес-логика).
- **Компетенции команды:** Python, Telegram Bot API, LLM API, базовый DevOps. *[Constraint: Команда знает Python — зафиксировано как ограничение, не как выбор стека.]*
- **Режим работы:** По запросу. В день DD — онлайн 10:00-20:00 (критическое окно).

### 5.2 Деплоймент

- **Частота обновлений:** По релизам (перед каждым DD-циклом). Между DD — патчи по необходимости.
- **Допустимый даунтайм при деплое:** < 1 минута. Не в день DD.
- **Откат:** Ручной откат допустим (< 5 мин). Автоматический откат не требуется для MVP.

## 6. Наблюдаемость (Observability)

*Как узнаем, что что-то сломалось.*

### 6.1 Мониторинг

- **Ключевые метрики:** доступность бота (health check), ошибки LLM API (5xx/timeout), размер очереди рассылки, использование диска.
- **Алерты:** Уведомление организатору в Telegram при: бот не отвечает > 60 сек; LLM API > 3 ошибок за 5 мин; очередь > 500 сообщений.
- **Дашборды:** Для MVP — не требуется. Для продакшна — дашборд доходимости и покрытия экспертами (RICE H9).

### 6.2 Логирование

- **Уровень детализации:** Стандартный (действия пользователей, ошибки, ключевые события). Подробный (debug) — только при диагностике.
- **Срок хранения логов:** **30 дней** (покрывает полный цикл DD: подготовка + DD + follow-up + анализ).
- **Требования аудита:** Audit trail для критических действий: утверждение кластеризации организатором, модерация ОС, передача контактов. Кто, когда, что подтвердил.

### 6.3 Бизнес-аналитика (для организаторов)

| Метрика | Описание | Источник |
|---------|----------|----------|
| Доходимость по ролям | % подтвердивших → реально пришедших (студенты, эксперты) | CustDev #1: "нету нормальной системы" |
| Покрытие экспертами | Залы с/без экспертов, баланс | RICE H3 |
| Воронка гостей | Регистрация → профилирование → посещение → follow-up | USM: US-023 |
| Подтипы гостей | Абитуриент / AI-практик / Другое | RICE H16 |
| Оценки проектов | % проектов с оценкой, средний балл | CustDev #1: "не все эксперты выставляют" |

## 7. Совместимость (Compatibility)

*С чем и как должно работать.*

### 7.1 Клиентские устройства

- **Браузеры:** Не применимо (Telegram-бот, без веб-интерфейса для пользователей).
- **Мобильные:** Telegram на iOS 14+ и Android 8+. Также Telegram Desktop (Windows, macOS, Linux).
- **Специфические устройства:** Нет.

> [💡 Предложение] Telegram как единственный канал — кроссплатформенность обеспечивается Telegram, не нами. CustDev #1: "Я бы не пользовалась никаким приложением." CustDev #2: "Неудобно себе отдельно ставить для демо-дня."

### 7.2 Сеть

- **Доступ:** Публичный интернет (Telegram).
- **Офлайн-режим:** Не нужен. Telegram сам кэширует сообщения при потере связи.

### 7.3 Интеграции

- **Обязательные:** Telegram Bot API (единственный канал), LLM API (кластеризация, профилирование, Q&A, предобработка ОС).
- **Желательные (после MVP):** Google Sheets (импорт данных контрольного рубежа — USM US-022), GitHub API (парсинг репозиториев студентов — USM SS-014).
- **Протоколы:** REST API (Telegram, LLM, GitHub). Файловый обмен для загрузки данных организатором (CSV/JSON).

### 7.4 Ограничения платформы Telegram

| Ограничение | Значение | Влияние |
|-------------|----------|---------|
| Сообщений в секунду (один бот) | 30 msg/sec | Очередь обязательна при массовых рассылках |
| Сообщений в чат в минуту | 20 msg/min | Не спамить одного пользователя |
| Размер сообщения | 4 096 символов | Длинные подборки и овервью разбивать на части |
| Callback data (inline-кнопка) | 64 байта | Короткие идентификаторы |
| Inline-кнопок в ряду | 8 | Учитывать при дизайне интерфейса |

## 8. Жизненный цикл данных (Data Lifecycle)

*Как долго хранить и когда удалять.*

### 8.1 Хранение

- **Срок хранения активных данных:** **1 DD-цикл** (~3 месяца: подготовка + DD + follow-up + анализ).
- **Архивация:** После завершения DD-цикла — архивация для аналитики следующего DD. Нужна для сравнения: "в прошлый раз было так, в этот — так".
- **Объём на один DD-цикл:** < **500 MB** (330 проектов × метаданные + пользователи + оценки + ОС + логи).

### 8.2 Удаление

- **Право на удаление:** Да. Пользователь может запросить удаление своих данных. 152-ФЗ.
- **Автоматическая очистка:** Через **6 месяцев** после DD. Архивные данные (без PII) сохраняются для аналитики.

### 8.3 Резервное копирование

- **Частота:** **Ежедневно** в период подготовки и проведения DD (за 2 недели до и 1 неделю после). Вне цикла — еженедельно.
- **Глубина хранения бэкапов:** **30 дней**.
- *Перевод:* «Если всё умрёт, мы можем откатиться максимум на вчерашний бэкап. Потеряем данные за последние сутки — но только для некритичных данных. Для подтверждений и оценок RPO = 0 (раздел 2.2).»

## [Сводка аналитика]

### Главные вызовы

1. **Событийная нагрузка с резким пиком.** Система используется ~2 недели в год, но в день DD должна быть доступна 99.5%. Нет возможности "разогнаться" — пик с первых минут мероприятия. 200 одновременных пользователей нажимают кнопки, получают напоминалки.

2. **Двойная зависимость от внешних API.** Telegram (единственный канал, нет fallback) + LLM API (ядро AI-фич). Если Telegram упадёт — система полностью неработоспособна. Если LLM упадёт — AI-фичи недоступны, но CRUD (подтверждения, оценки, напоминалки) продолжают работать.

3. **Жёсткие ресурсные ограничения.** VM: 4 vCPU, 8 GB RAM, 65 GB SSD — не расширяется. Должно хватить на 435 пользователей + БД + LLM-клиент + очередь рассылки. При росте до 2000 пользователей — пересмотреть инфраструктуру.

4. **Модерация AI = бутылочное горлышко.** Все критические AI-решения требуют подтверждения организатора (кластеризация, ОС, подборки для бизнеса). Организаторов ~5 человек. При 330 проектах × 2 эксперта = 660 комментариев ОС на модерацию — нагрузка на оргов.

5. **Согласие на передачу контактов.** CustDev #3 (Олег): "Не всем ребятам будет интересно." Двусторонний consent замедляет цепочку бизнес→студент. Но без него — "негативный фактор".

### Серые зоны

1. **Точное кол-во бизнес-партнёров.** Олег говорит о "пятнадцати-двадцати" проектах, которые может показать. Но сколько партнёров реально придёт — неизвестно. Reach = 15 (грубая оценка).

2. **Формат ОС экспертов.** Свободный текст? Шкала + комментарий? Критерии оценки? CustDev #2: "оценка по критериям, не просто 1-5", но точные критерии не определены. Требует уточнения с организатором.

3. **Поведение при одновременном DD + Research.** День 2: Demo Day (4 зала) + AI Talent Conf Research (4 зала) параллельно. Один бот обслуживает оба трека? Или Research — отдельная система?

4. **Веб-дашборд для оргов.** Brief: "Веб-дашборд — если хватит ресурсов." Если дашборд = веб-страница, нужен HTTP-сервер на VM. Если дашборд = вывод в Telegram — ограничение 4096 символов. Формат не определён.

5. **Объём данных при парсинге GitHub.** USM SS-014 (Release 1.1): парсинг GitHub-репозиториев. 330 репозиториев × README + структура + активность — какой объём данных? Влияет на хранение и latency.

### Рекомендации

1. **Очередь сообщений — обязательна с первого дня.** Telegram rate limit 30 msg/sec жёсткий. При 435 пользователях рассылка без очереди будет дропать сообщения.

2. **Graceful degradation для LLM.** CRUD-ядро (подтверждения, напоминалки, оценки) не должно зависеть от LLM API. Это ~70% функций по RICE.

3. **Batch-модерация ОС.** 660 комментариев × модерация организатором — нужен UX для массового approve/reject, не поштучно. Иначе организатор потратит часы.

4. **Мониторинг в Telegram.** Дашборд мониторинга — в Telegram организатору (алерты, статус). Не требует отдельного веб-интерфейса для MVP.

5. **Архитектору:** предусмотреть рост до 1000 проектов / 2000 пользователей при выборе хранилища и структуры данных. Не блокирует MVP, но влияет на решения.
